Design 1 — DAG Graph (Node-Link View)

Tree of goals → iterations → attempts rendered as a directed graph. The structure of work is the dominant visual. Selecting a node populates an inspector with the agent roster (main / subagents / helpers) and live sandbox state. Best when the user thinks structurally: "which branch of the attempt tree am I on?"

Design 2 — Timeline Stream (Event Feed)

Linear chronological feed of every agent turn, tool call, and sandbox event. Color-coded rail makes agent role obvious. Best for debugging "what just happened" — closest to a Claude Code transcript, but multi-agent aware.

Design 3 — IDE Workspace (Multi-Pane)

Three panes: explorer tree of attempts/agents, message thread for the active attempt, live sandbox terminal on the right. Familiar to anyone who uses VS Code. Best for users who want to follow execution in real time alongside the actual shell output.

Design 4 — Card Hierarchy (Nested Cards)

Expandable cards that mirror the conceptual nesting: goal ⊃ iteration ⊃ attempt ⊃ agent activity. No graph, no tree, no terminal — just readable narrative blocks. Best for status reports, post-mortems, and stakeholders who don't want infra.

Side-by-side trade-offs

1 · DAG Graph 2 · Timeline Stream 3 · IDE Workspace 4 · Card Hierarchy
Mental model Structural — branching attempts Causal — what happened when Operational — driving execution Narrative — story of the run
Best for Replans, fan-out, parallel attempts Debugging, audit, multi-agent tracing Live work, watching sandbox Reports, stakeholders, post-mortems
Weakness Heavy layout cost; bad on tiny screens Hard to see structure; long scrolls Dense; needs 3-pane width No real-time signal; hides parallelism
Live updates Node pulses + edge animation Auto-scroll feed (kills focus) Terminal streams natively Card badge ticks; less obvious
Recommended primary Design 3 (IDE Workspace) as default — embeds Designs 1 & 4 as switchable tabs in the middle pane, and Design 2 as a "Feed" tab in the right pane. One layout, four lenses.
