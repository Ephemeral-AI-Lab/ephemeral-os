Task Center — UI Designs v3

Corrected model from db/models/{goal,iteration,attempt}.py and submit_execution_handoff: roles are planner / generator(s) / evaluator, and any generator may spawn a nested goal.
Goal (origin) ── owns ordered → Iteration[]
Iteration (vertical · continuation) ── owns ordered → Attempt[] (budget=N)
Attempt (horizontal · retry) ── planner → generator[] (parallel DAG) → evaluator
  generator may emit submit_execution_handoff(goal) ⇒ child Goal recurses

Iter N+1 fires when iter N's last attempt produced a continuation_goal. Attempt N+1 fires when attempt N failed within the iteration's attempt_budget.

A. Pipeline view · planner ▸ generators ▸ evaluator, stacked by iteration

The body is a vertical stack of iteration bands. Each iteration shows its attempt strip horizontally (retry chain). The active attempt expands into its three-phase pipeline: planner → generators (parallel fan-out) → evaluator. Generators that called submit_execution_handoff embed their nested goal as a sub-pipeline inline.

Inspired by GitHub Actions runs + Buildkite pipelines + LangSmith span detail.

Trade-offs

Aspect A · Pipeline (stacked iterations) B · Recursive Outline C · Mission Control
Strongest at Seeing the planner→gen→eval shape and the iteration continuation chain visually Inspecting the data model; reading recursion clearly Daily driver — structure + causality in one view
Nested goals Embedded as sub-pipeline inline under spawning generator Indented as a child Goal block in the tree Indented event-log subsection + sub-pipeline card
Iteration continuation Explicit "↓ continuation_goal" arrow between bands Just another iteration row Continuation arrow between iter groups in strip
Attempt retries Horizontal strip per iteration Sibling nodes under iteration Horizontal in strip, only one attempt expanded
Best for Designers/PMs; explaining the model Debugging; deep inspection; copy-paste-able state Operators watching live runs
Live updates Animated phase connectors; live chip on gen cards Subtle row-level dot pulse Pipeline cards pulse + event log auto-grows
Recommendation: ship C · Mission Control as the default; offer A as a "model view" tab for explaining the run to a stakeholder, and B as the "inspector" panel for debugging. All three share the same backing data: Goal → Iter → Attempt → (planner, generators, evaluator) → (optional nested Goal).
