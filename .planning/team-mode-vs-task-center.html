Multi-Agent Collaboration: Claude Code Team Mode vs EphemeralOS task_center

A side-by-side comparison anchored on a real session running the O(1)-overlay-mount migration plan, where team-mode friction surfaced the architectural delta between the two systems.

Claude Code Team Mode — generic agent orchestration
EphemeralOS task_center — typed AI-task DAG with retry semantics

1. The friction observed in this session

Running the O(1)-overlay-mount plan as /team o1-overlay-mount exposed two recurring failure modes:

Issue 1 Multiple tasks assigned per worker → silent blocking. Workers got two tasks (e.g. worker-2 owned T2 and T4; worker-3 owned T3 and T5). After completing the first, they went idle. When T4's blockers resolved later, no event woke worker-2 to claim it — they just sat. The lead had to manually SendMessage with the unblock notice. This happened twice in the run.
Issue 2 Whole team went idle; lead unaware. After T1, T2, T3 completed, every worker idled. The lead (me) had no automatic event saying "T4 is now unblocked, dispatch worker-2" — and the lead is itself a chat session, only re-invoked on inbound messages or user prompts. The user had to type "all workers went into idle" to wake me to re-dispatch.

Both issues trace to one architectural fact: team mode is generic agent orchestration with no domain semantics. Task dependencies are stored as blockedBy metadata, but no scheduler advances tasks when blockers resolve. The lead is a normal Claude session; it does not auto-tick on state changes.

2. Mental model side-by-side

Claude Code Team Mode

USER │ v TEAM-LEAD (Claude session) │ ├─ TaskCreate × N (flat task list) ├─ TaskUpdate owner (pre-assign) ├─ Agent spawn × N (background sessions) │ ▼ ── async ── workers: worker-1, worker-2, worker-3, worker-4 ↑ ↑ ↑ ↑ └── SendMessage ── (inbox-based DM) └── TaskUpdate(status) — workers self-mark └── idle notification (turn ended)

Hierarchy: flat (lead + N peers). No nested structure.

Tick model: event-driven on inbound messages OR user prompt. No periodic tick.

Schedule: manual — lead reads TaskList, sends messages.

EphemeralOS task_center

Goal (id, status, iteration_ids) │ ├─ Iteration (id, status, attempts[]) │ │ │ ├─ Attempt (planner → generator DAG → evaluator) │ │ │ │ │ ├─ Planner task │ │ │ └─ produces PlannedGeneratorTask[] (DAG) │ │ │ │ │ ├─ Generator tasks (DAG, deps[], typed roles) │ │ │ ├─ ordered_generator_tasks(topo sort) │ │ │ ├─ ready_pending_generator_ids │ │ │ └─ may spawn sub-Goal → WAITING_GOAL state │ │ │ │ │ └─ Evaluator task │ │ └─ EvaluatorSubmission → close attempt │ │ │ └─ Next attempt on failure (IterationManager.retry) │ └─ Closure: TerminalSuccess / SuccessDeferred / AttemptPlanFailed

Hierarchy: 4 levels — Goal → Iteration → Attempt → Generator-DAG. Each level has its own state machine and lifecycle owner.

Tick model: AttemptDispatcher drives launch/quiescence; deterministic state machine.

Schedule: automatic — ready_pending_generator_ids() at every submission event.

4. How each handles the two failure modes I hit

Issue 1 — Worker has two tasks, second one's dep resolves later

Team Mode

worker-2 owns T2 (no deps) and T4 (blocked by T1+T2+T3). After T2 completes, worker-2 idles. T3 completes on worker-3; T4 is now unblocked. But:

  • No event fires when blockers resolve.
  • worker-2's blockedBy on T4 is just a string list in tasks/4.json.
  • worker-2 won't re-check TaskList until someone SendMessages them.
  • Lead has no scheduler that polls completion deltas.

Result: silent stall. User must observe and prompt.

task_center

In the generator DAG, two tasks for "the same agent" would simply be two DAG nodes with deps=("T2",) on T4. On every generator submission:

def on_generator_submission(submission):
    persist(submission)
    for ready_id in ready_pending_generator_ids(attempt):
        dispatcher.launch(ready_id)

The dispatcher is invoked on every state change; ready_pending_generator_ids() computes which tasks are newly executable. There is no "worker idles until poked" state.

Issue 2 — Whole team idle; lead unaware

Team Mode

Lead is a Claude session — it only runs when:

  • An inbound SendMessage arrives
  • The user types a new prompt

If every worker is idle and no one is sending anything, the lead has no scheduled wakeup. There is no periodic tick. There is no "watchdog" agent.

The user observed exactly this: workers idled, lead was dormant, user had to prompt.

task_center

There is no "lead session" to be dormant. The dispatcher is an ordinary Python object running inside the daemon process; it doesn't sleep waiting for messages. Every persisted submission triggers a synchronous tick.

When all generators finish for one attempt, AttemptDispatcher immediately drives the evaluator launch, and on its submission immediately drives attempt closure. There is no human-in-the-loop required for advancement; the human is only the goal-author and the final viewer.

6. Nested goal planning

Team Mode

To delegate a sub-task in team mode, a worker would need to:

  • Spawn a sub-team via TeamCreate — but workers are told NEVER to do this (rule from the team-worker preamble: "NEVER run team orchestration commands").
  • Or send a message to the lead asking the lead to create more tasks — manual, ad hoc.

There is no first-class nested-goal primitive. Hierarchies must be flattened by the planner before spawning.

task_center

A generator may emit a SpawnReason indicating it wants a sub-Goal. The task transitions to WAITING_GOAL; the sub-goal opens its own iteration loop. On sub-goal closure, GoalClosureReport routes back to the parent generator, which resumes.

Files showing this seam:

  • task_center/goal/starter.py — sub-goal creation
  • task_center/goal/close_report_router.py — closure routing back to parent
  • task_center/task_state.py:WAITING_GOAL — the parked state

7. When to reach for which

Scenario Team Mode task_center
Ad-hoc PR-sized work; a handful of parallel agents with explicit task list Good fit. Low ceremony. Overkill. Requires database + schema + role definitions.
Multi-step plan with task dependencies (DAG) Workable but fragile. Lead must manually re-dispatch on dep resolution. Designed for it. DAG is the primitive.
Long-running iteration: retry on failure, multi-attempt No primitive. Lead writes retry loops by hand. First-class. IterationManager + AttemptStatus.FAILED → next attempt.
Worker decides mid-flight that it needs to spawn a sub-goal Disallowed. Workers cannot spawn teams. Native. WAITING_GOAL + GoalClosureReport.
Need typed roles (planner / generator / evaluator) with role-specific recipes Role is just an agent-type string at spawn; no semantic enforcement. TaskCenterTaskRole enum + role-keyed context recipes in context_engine/recipes/.
Need structured audit / observability of every state transition Tail JSON files; informal. TaskCenterAuditEmitter.
Cross-session continuity (resume a workflow across days) JSON files persist, but lead session is gone; pick-up requires manual reconstruction. Persistent Goal / Iteration / Attempt rows in the database; any process can resume by reading state.

8. The fix bridge — what would make team mode behave like task_center

Reading the friction symptoms backwards, the missing pieces in team mode are:

  1. An auto-dispatcher loop. Something that ticks on every TaskUpdate(status=completed) event, computes the newly-ready set, and sends wake messages to assigned owners. Could be implemented as a hook on the task-update tool.
  2. Periodic lead heartbeat. When all workers are idle and any pending task is unblocked, the lead session should be auto-poked. Today there's no scheduler that does this.
  3. One task per worker session. Multi-task ownership is the architectural sin behind Issue 1. Spawning a fresh worker per task (the task_center model) avoids it entirely — workers don't sit idle holding queued work because they don't persist past their one task.
  4. Typed task roles and closure reports. So that "complete" means something more than a boolean — the next step can be auto-routed by role.

Three of these four are exactly what task_center does. The fourth (one-task-per-worker) is the deepest structural difference: team-mode workers are persistent Claude sessions; task_center generators are launches.

Takeaway Team mode is a great tool for fan-out work that fits in a session. For anything with dependencies, retries, or nested goals, it's worth either (a) wrapping team mode with a scheduler hook that auto-pokes idle workers when blockers resolve, or (b) using a domain-specific harness like task_center that has those semantics baked into its state machine.
