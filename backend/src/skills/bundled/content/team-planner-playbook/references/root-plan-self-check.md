# Root Plan Self-Check
Use this reference after decomposition examples and immediately before `plan-json-contract` on a fresh root or any crowded parent layer.
If `plan-json-contract` already loaded earlier in the same turn, do not load more references; finish this self-check in reasoning and emit final JSON next.
## Stop signs
- Your recap dropped a launched exact-file scout, still calls a nameable exact owner "needs more scouting", flattened everything into direct developers plus one terminal validator, or rewrote a named benchmark file or node onto its production owner path.
- Your draft contains a benchmark path that does not appear verbatim in the prompt, validator packet, or inherited benchmark-surface ledger.
- A package path, directory path, or broad single file survived the scout wave but no child planner owns its internal split.
- An atomic leftovers lane owns several unrelated exact files or benchmark files even though one child planner could schedule them without another scout wave.
- The terminal validator widened to a repo-scale command mainly because the parent layer flattened everything.
## Repair moves
- Rebuild a literal scout ledger and a literal benchmark-surface ledger from prompt-backed paths before final JSON, restore any benchmark file or node that got rewritten onto an owner path, then keep at least one direct ready developer when evidence is already concrete.
- Promote package or directory slices to `team_planner` when their next decision is internal decomposition.
- Promote a broad single file to `team_planner` when families or verification surfaces inside it are still separable.
- Replace any atomic leftovers lane that owns unrelated exact files with either direct leaves, one residual child planner, or a return to scout launching for any still-nameable exact owner.
## Few-shot examples
- Example: root scouts map `pkg/io/hdf.py`, `pkg/io/parquet/`, `pkg/groupby.py`, and five small exact files.
  Emit `developer(hdf)` now, keep `parquet_child` and `groupby_child` expandable, and keep the five residual files as direct leaves or one residual child planner.
  Do not hide `cli.py`, `compat.py`, `config.py`, `json.py`, and `utils.py` inside one atomic leftovers developer; the first shared import blocker will drag that leaf toward benchmark-test rewrites instead of the minimal production owner.
- Example: the wave launched `pkg/compat.py`, but the recap lists only five scouts and still says compat needs more scouting while `pkg/config.py` remains unlaunched.
  Reconcile the scout ledger, count compat as mapped, then launch `pkg/config.py` before `plan-json-contract`.
- Example: the owner file is `pkg/io/json.py`, but the prompt only named `pkg/io/tests/test_json.py` and `pkg/tests/test_cli.py`.
  Keep those prompt benchmark surfaces verbatim in `owned_failures` and `verify`. Do not emit `pkg/io/tests/test_io_json.py`, `pkg/cli/tests/test_cli.py`, or any other repaired-by-intuition sibling.
