# Root Plan Self-Check

Use this reference after decomposition examples and immediately before `plan-json-contract` on a fresh root or any crowded parent layer.

## Stop signs

- Every mapped slice became a direct developer lane and the only non-developer item is one terminal validator.
- A package path, directory path, or broad single file survived the scout wave but no child planner owns its internal split.
- Several residual single-file slices were flattened into the parent layer even though one child planner could schedule them without another scout wave.
- The terminal validator widened to a repo-scale command mainly because the parent layer flattened everything.

## Repair moves

- Keep at least one direct ready developer when evidence is already concrete.
- Promote package or directory slices to `team_planner` when their next decision is internal decomposition.
- Promote a broad single file to `team_planner` when families or verification surfaces inside it are still separable.
- When the parent layer is crowded with residual exact-file slices, prefer one residual child planner over flattening the whole layer.
- Let child planners inherit scout briefs; do not relaunch scouts unless ownership is still missing.

## Few-shot examples

- Example: root scouts map `pkg/io/hdf.py`, `pkg/io/parquet/`, `pkg/groupby.py`, and five small exact files.
  Emit `developer(hdf)` now.
  Emit `team_planner(parquet_child)` and `team_planner(groupby_child)`.
  Emit either direct small-file developers or one residual child planner.
  Do not submit eight root developers plus one terminal validator just because every scout has `scope_coverage: 1.0`.
- Example: root scouts map one directory owner and one 3000-line file plus three tiny helpers.
  Keep the helper leaf lanes direct if they are ready.
  Keep the directory owner and broad file expandable so child planners can shape the deeper DAG.
