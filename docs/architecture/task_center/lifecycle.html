This is the most important TaskCenter page: the control plane is the persisted Workflow -> Iteration -> Attempt model. A Workflow owns the objective, an Iteration is the bounded planning slice for that objective, and an Attempt is one planner-authored plan — a DAG of generator and reducer tasks — run inside that Iteration. Every workflow is generator-spawned, including the root (via a synthetic run-level bootstrap generator task). Everything else in TaskCenter exists to feed, advance, retry, continue, or close this state chain.

Workflow -> Iteration -> Attempt Model #

  • Workflow: owns the objective (workflow_goal), ordered iteration ids, terminal status, and the backward link parent_task_id (bidirectional with Task.child_workflow_id). There is no origin kind and no stored final outcome — workflow.outcomes is derived as the last iteration's outcomes.
  • Iteration: owns one iteration-local goal string (iteration_goal), a retry budget, and the persisted outcomes projection. Deferred continuation creates another Iteration under the same Workflow.
  • Attempt: owns one planner-authored plan inside one Iteration: planner terminal submission, two task-id tuples (generator_task_ids + reducer_task_ids), and the attempt close.
  • Task rows: store the launched planner, generator, and reducer agents for an Attempt (plus a parent generator waiting on a child workflow). They are not a fourth control-plane level.

Workflow Start Workflow #

WorkflowStarter.start(*, prompt, parent_task_id) is the only prompt-to-Workflow start path; there is no WorkflowOrigin. It strips and validates the prompt, asserts the parent task is running with no open child workflow, creates the Workflow row linked by parent_task_id, creates Iteration 1 with a registered IterationAttemptCoordinator, creates the unstarted initial Attempt, marks the parent task WAITING_WORKFLOW (with its child_workflow_id), and then starts the Attempt orchestrator. It returns a StartedWorkflow with no origin field.

Retry And Deferral #

Retry and deferral are separate Workflow -> Iteration -> Attempt decisions. A failed Attempt can retry inside the same Iteration while budget remains. A passed Attempt can either terminally close the Workflow or carry deferred_goal_for_next_iteration, which closes the current Iteration as succeeded and starts the next Iteration under the same Workflow.

Delegated Child Workflows #

Delegation creates another complete Workflow -> Iteration -> Attempt chain. It starts from the generator submission context, not from a direct agent-to-agent channel. submit_workflow_handoff resolves the running generator task and calls GeneratorSubmissionContext.start_delegated_workflow(goal_handoff=...), which invokes WorkflowStarter.start(prompt=goal_handoff, parent_task_id=current_task_id). There is no closure report or router layer.

handoffThe parent generator calls submit_workflow_handoff while running inside the parent attempt's plan DAG.
child startWorkflowStarter validates the parent, inserts the child state chain, and calls the parent orchestrator's start_child_workflow(), which atomically flips the generator RUNNING to WAITING_WORKFLOW and sets child_workflow_id.
parent non-quiescentWAITING_WORKFLOW is not terminal, so the parent attempt cannot advance past that generator until the child-workflow outcome is delivered.
child-local policyChild retry stays inside the child iteration; child deferral creates another child iteration and still does not resolve the parent.
outcome deliveryA terminal child workflow close routes via WorkflowLifecycle._route_close() to the parent orchestrator's apply_child_workflow_outcome(), which marks the waiting generator DONE on success or FAILED on failure and copies the child workflow's outcomes directly onto the parent generator task.
Resolution rule On child close the parent generator becomes terminal via AttemptOrchestrator.apply_child_workflow_outcome(): DONE on success, FAILED on failure. It does not return to RUNNING. The only path back to RUNNING is cancel_child_workflow(), which restores the generator when a child-workflow start fails before the child runs.
single childWorkflowStarter lists open workflows for the parent task (list_for_parent_task) and rejects a second open child workflow before creating another one.
waiting invariantapply_child_workflow_outcome() is idempotent: it only resolves a parent currently in WAITING_WORKFLOW; an already DONE or FAILED parent (an earlier delivery or race) returns silently.
attempt progressA child success can let the parent attempt close PASSED once every plan task is DONE. A child failure renders dependents unreachable and eventually closes the parent attempt with TASK_FAILED.

Nested Planning Case #

The nested planning case is a delegated child workflow whose planner must plan inside a parent generator task. This is still the same model: the parent Attempt pauses one generator task, the child workflow gets its own Iteration and Attempt, and the child planner authors its own plan DAG of generators and reducers.

Depth rule The root Workflow is depth 1. A delegated child workflow is depth 2. Depth greater than 1 disables further planning deferral and generator handoff through submission prehooks, so nested planning stays bounded by outcome submissions.

Close Routing And Invariants #

Closing is a state transition, not a report. WorkflowLifecycle.handle_iteration_closed(iteration_id, succeeded, deferred_goal, final_attempt_id) takes primitive args (no IterationClosureReport): a succeeded iteration with a deferred goal creates the next iteration; otherwise it calls close_workflow(), then deregisters the closed coordinator. close_workflow() sets the workflow status and hands off to _route_close(). There is no WorkflowClosureReportRouter and no WorkflowOrigin.

close fork_route_close() reads the workflow's parent_task_id and forks on attempt_id_from_task_id: a real attempt-bound parent (a child workflow) delivers to that parent orchestrator's apply_child_workflow_outcome(); the synthetic root parent <run_id>:root (which encodes no attempt) delivers to the injected run_close_handler.
root run closeThe run-close handler is RunController.on_root_workflow_closed(): it writes the root workflow's outcomes onto the <run_id>:root bootstrap task (DONE/FAILED) and finishes the TaskCenter run as done or failed, treating an already finished run as idempotent.
process-local registryWhile a parent task is WAITING_WORKFLOW, its parent attempt cannot be quiescent and its AttemptOrchestrator must still be registered in this process. A missing orchestrator on a child-workflow close is a hard invariant violation.
idempotent closeapply_child_workflow_outcome() returns silently if the parent generator is no longer WAITING_WORKFLOW (an earlier delivery or race).
state validation_core/invariants.py centralizes checks for open workflows, contiguous iteration and attempt sequence numbers, stage correctness, ownership, fail-reason-present-on-failure, and valid attempt close states.
  • Workflow -> Iteration -> Attempt policy stays in WorkflowStarter, WorkflowLifecycle, IterationAttemptCoordinator, RunController (root close), and AttemptOrchestrator.
  • The context engine later reads persisted state. It does not choose retry, deferral, child-workflow resolution, or workflow close policy.
  • The per-attempt orchestrator is not a global agent orchestrator; it only advances one attempt's planner, generators, and reducers.
