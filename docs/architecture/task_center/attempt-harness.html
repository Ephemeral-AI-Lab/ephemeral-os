The attempt harness is the Attempt layer of Workflow -> Iteration -> Attempt: planner submission, a persisted plan DAG of generator and reducer tasks, then a single RUN stage that schedules that DAG to quiescence and closes the Attempt. It owns one Attempt, not the whole Workflow or a global agent orchestration layer. There is no separate evaluator stage; judging is a reducer task inside the plan.

Attempt State Machine #

  • AttemptStage collapses to PLAN to RUN to CLOSED. The old GENERATE and EVALUATE stages are gone; the single RUN stage schedules the whole plan DAG (generators and reducers) to quiescence.
  • AttemptStatus is running until the close path marks the attempt PASSED or FAILED.
  • AttemptFailReason has only two values: TASK_FAILED (any plan task or the planner failed or blocked) and STARTUP_FAILED (the agent could not launch). The old per-role planner/generator/evaluator reasons collapse into TASK_FAILED.
  • AttemptOrchestratorRegistry is process-local lookup for active attempts, used by launch/exhaustion and child-workflow outcome delivery (RegisteredAttemptOrchestrator exposes start_child_workflow, apply_child_workflow_outcome, and cancel_child_workflow).

Planner And Plan-DAG Persistence #

A planner submission is accepted only for the current attempt planner task while the attempt is in PLAN. The plan is two tuples: tasks (generators, each with an agent_name and task_spec) and reducers (each with a prompt, no agent name — reducers always run the reducer profile). Both use needs (not deps). Full (completes) plans cannot set deferred_goal_for_next_iteration; deferring (defers) plans must set it. There is no plan_spec and no evaluation_criteria. After validation, the orchestrator marks the planner task done, persists generator rows then reducer rows, records both id tuples on the attempt, moves the attempt to RUN, and advances ready work.

structural gateordered_plan_tasks(generators, reducers) enforces the multi-lane DAG rules at PLAN to RUN: at least one reducer ("Plan must contain at least one reducer"), generator needs can reference one or more generators but not reducers, reducer needs must contain one or more generators and cannot reference reducers, and every generator must feed at least one downstream generator or reducer ("Plan has generator(s) no downstream task needs: ..."). This guarantees every attempt has an exit and all work is judged.
DAG validationThe same helper rejects duplicate local ids, needs on unknown local ids, and dependency cycles across the combined generator + reducer graph before any runtime rows are used.
topological orderA stable Kahn sort orders both tuples together. Runtime task ids are derived from the attempt id plus planner local id (<attempt_id>:gen:<local>, <attempt_id>:red:<local>), and needs are normalized into the persisted ids.
stage moveattempt_store.set_stage(..., AttemptStage.RUN) happens only after the planner task, generator rows, and reducer rows have all been persisted and both id tuples recorded on the attempt.

Plan-DAG Waves And Quiescence #

AttemptStageAdvancer (in attempt/run_stage.py, class name unchanged) is store-driven and owns the single RUN stage. On every generator or reducer submission, or launch failure, it lists the attempt's persisted plan tasks — the union of generator_task_ids and reducer_task_ids — launches every pending task whose needs are done, and otherwise summarizes whether the DAG is still active, failed, blocked, or fully done. Generators and reducers are one DAG: a reducer simply has generators in its needs.

[ready_pending_plan_ids selects ready generators/reducers]

        v
[build AgentLaunch: for_generator / for_reducer]

        v
[run ephemeral agent]
        |
        v
[terminal submission captured]
        |
        +--> [more plan tasks ready] -> [advance next task]
        |
        +--> [plan DAG quiescent] -> [dag_status closes the attempt]
readyready_pending_plan_ids() selects pending generator and reducer tasks only after their entire needs set is DONE.
launchReady tasks move to RUNNING, emit launch audit events, receive composed launch context, and are handed to the launcher (for_reducer for reducer rows, for_generator otherwise).
settleFailed launches re-enter advancement so remaining runnable siblings, unreachable descendants, or terminal closure can settle from store state.
quiescedag_status() reports all_quiescent, all_done, and any_failed_or_blocked in one pass. Pending descendants of failed or blocked upstreams are unreachable but persisted; once runnable siblings finish and the DAG is quiescent, the attempt closes — PASSED if every plan task is DONE, otherwise FAILED with TASK_FAILED.
waiting workflowWAITING_WORKFLOW is not quiescent and not done. A parent generator waiting on a child workflow keeps the parent attempt from advancing until the child-workflow outcome is delivered.
unreachable descendantsPending descendants of FAILED or legacy BLOCKED tasks remain persisted as pending not-started work. They are counted as unreachable for quiescence, not launched.
failure semanticsGeneratorSubmission(status="failed") and ReducerSubmission(status="failed") mark the task FAILED; the attempt waits for runnable siblings, then fails with TASK_FAILED. The stored task outcome status is failed, not blocker or failure.

RUN-Stage Closure #

There is no evaluator stage and no separately spawned judging task. Judging is one or more reducer tasks inside the plan DAG, scheduled by the same RUN stage as generators. Closure is purely a function of plan-task quiescence: when no plan task is still ready or running, dag_status() decides the verdict, and _close_attempt() applies it.

passWhen every plan task (generators and reducers) is DONE, dag_status().all_done is true and the stage advancer closes the attempt PASSED with no fail reason.
failWhen the DAG is quiescent and any_failed_or_blocked is true, the attempt closes FAILED with TASK_FAILED. The planner-failure and launch-exhaustion paths reach the same TASK_FAILED close.
close ownerAttemptOrchestrator._close_attempt() is the only attempt-closing path. It asserts a valid close, deregisters the orchestrator, then notifies the iteration coordinator via on_attempt_closed.
outcome projectionAttempt close persists generator/reducer execution outcomes. Iteration close projects successful reducer outcomes from all attempts plus failed generator/reducer outcomes from the final failed attempt. There is no separate evaluator summary.
retry handoffThe orchestrator never creates retry attempts. Retry is decided after closure by IterationAttemptCoordinator.handle_attempt_closed().
Boundary The attempt harness closes only one Attempt. Iteration retry, deferred continuation, nested child workflows, and Workflow closure live one level up in the Workflow Iteration Attempt page.

Launcher Failures And Exhaustion #

EphemeralAttemptAgentLauncher schedules role-specific agent runs. Terminal submission tools are expected to move the task out of RUNNING during the run. If the runner returns, crashes, returns None, or exits without a terminal submission while the task remains running, the launcher reports role-specific exhaustion back into the same orchestrator API.

        v
[start ephemeral agent process]

        v
[engine loop runs until stop condition]

        +--> [terminal submission received]
        |       -> [persist attempt event]
        |
        +--> [agent exits / tool budget exhausted]
                |
                v
        [launcher records exhaustion/failure]
                |
                v
        [AttemptOrchestrator applies closure]
schedulelaunch() resolves the role definition, requires an active asyncio loop, creates an async task, and tracks it until completion.
metadata_run_launch() builds ExecutionMetadata and maps AgentLaunch rows into the runner prompt shape.
normal terminalIf a terminal submission moved the TaskCenter task out of RUNNING, launcher completion does not synthesize another state change.
exhaustionIf the task remains RUNNING, _report_exhaustion() calls the planner, generator, or reducer failure path on the active orchestrator (apply_planner_failure / apply_generator_submission / apply_reducer_submission) with fail_reason="run_exhausted".
unownedIf the orchestrator is missing, _fail_unowned_attempt() marks the task failed, closes the attempt directly as TASK_FAILED, and notifies a registered iteration coordinator when present.
no launcher retryThe launcher invokes the runner once per launch. Query-loop soft warnings and overshoot tolerance own in-run recovery.
startup failuresPlanner startup failure in AttemptOrchestrator.start() deregisters the orchestrator and marks the planner task failed; IterationAttemptCoordinator owns closing the inserted attempt as STARTUP_FAILED.
role-parametrized exhaustionCurrent source uses one _report_exhaustion() branch tree over the planner, generator, and reducer roles. There is no per-role reporter helper table.
plan-task launch failureA generator or reducer launch failure in AttemptStageAdvancer._mark_launch_failed() marks that task FAILED with agent_launch_failed and re-enters advancement; the quiescent DAG then closes the attempt with TASK_FAILED.

Audit Events #

TaskCenter attempt advancement emits a small set of audit events through TaskCenterAuditEmitter. The emitter is a write-only facade over the shared audit sink; it shapes node identity and payload fields so stage advancement does not repeat audit plumbing.

task_center.task.readyEmitted before a plan-task (generator or reducer) launch with satisfied dependency ids and the task identity payload.
task_center.task.launchedEmitted after a task row moves into running launch state, carrying run id, attempt id, task id, role, agent name, and needs.
task_center.task.failedEmitted when stage advancement marks a launched plan task failed because agent launch failed.
audit sinkAttemptDeps.audit_sink (in attempt/launch.py) defaults to NoopAuditSink, so production wiring can observe events without making attempt logic depend on a concrete recorder.

Evidence And Refresh #

Primary Code

attempt/orchestrator.py owns submission entry points (planner, generator, reducer, child-workflow) and attempt close. attempt/run_stage.py owns the single RUN-stage plan-DAG advancement. attempt/plan_dag.py owns the structural gate (≥1 reducer + generator-only DAG ending in reducer sinks), DAG validation, and quiescence (ordered_plan_tasks, ready_pending_plan_ids, dag_status).

Launch Code

attempt/launch.py owns agent scheduling, row mapping, exhaustion reporting, AttemptDeps, and AgentLaunchFactory (for_planner / for_generator / for_reducer). Result data lives in _core/outcomes.py as task-bound planner/execution outcome records.
