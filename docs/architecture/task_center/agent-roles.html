This page documents the live TaskCenter role contracts loaded from agent profile Markdown. Current verdict: roles are declarative profiles plus submission-time prehooks; TaskCenter coordination flows through persisted workflow, iteration, attempt, task, and terminal submission state, not through peer-to-peer agent messages or a global agent orchestrator.

Scope And Verdict #

  • Main agents are planner, executor (generator role), and reducer. Their role contracts live in backend/src/agents/profile/main/*.md. The evaluator role and the verifier profile are gone — the reducer (a plan task) is the attempt's exit gate.
  • Every main profile gets _main_role_contract.md prepended at load time, so the shared terminal discipline and XML-context boundaries are part of the actual system prompt.
  • Planner generator tasks may target only registered generator-role profiles (executor) — the gate is role == AgentRole.GENERATOR. Reducers always run the reducer profile and carry no agent_name.
  • Delegation is state-backed. submit_workflow_handoff starts a child workflow through WorkflowStarter.start(prompt, parent_task_id), the parent task moves to WAITING_WORKFLOW, and AttemptOrchestrator.apply_child_workflow_outcome() resumes the parent. There is no closure-report router.

reducer is the attempt exit gateA reducer digests <dependencies> and completes its <assigned_task>. It may apply trivial inline fixes via edit_file / write_file; scope is enforced by profile prose and the advisor's pre-terminal audit (which sees full edit-input transcripts).

Dispatch And Terminals #

Profiles declare the broad tool surface and TaskCenter now uses those terminal lists directly. The engine registers the union of allowed_tools and terminals; planner-submission validation and submission prehooks enforce role/depth policy at the actual decision point.

Runtime tool surface
engine.agent.factory._build_agent_tool_registry() registers sorted(set(agent_def.allowed_tools) | set(agent_def.terminals)) and records the role as agent_def.role.value in tool metadata.
Planner dispatch gate
_is_generator_capable_agent() requires role == AgentRole.GENERATOR (only the executor profile). Reducers carry no agent_name and always run the reducer profile.
Terminal descriptors
tools/_terminals/registry.py owns selection guidance and advisor-review focus for every terminal. Row 3, row 4, and advisor prompts render from the same catalog.
Nested planner policy
submit_planner_outcome remains visible everywhere, but DisallowNestedPlannerDeferral rejects nonblank deferred_goal_for_next_iteration inside delegated workflows.
Nested generator policy
submit_workflow_handoff remains visible everywhere, but DisallowNestedWorkflowHandoff rejects handoff from a nested workflow generator.

Workflow Boundaries #

Plannersubmits one generator + reducer DAG
Executoredits or delegates assigned task
Reducerdigests needs, gates the attempt
Terminal submissionstate mutation, not prose
AttemptOrchestratorstage and task transitions
WorkflowStarterchild workflow from handoff
apply_child_workflow_outcomeparent task resume
planner boundaryThe planner writes one attempt plan: generator tasks, task specs, and reducers (edges are needs). It does not execute tasks, close iterations, or decide final workflow state.
executor boundaryThe executor completes <assigned_task>, submits a concrete failure outcome, or starts a delegated workflow through WorkflowStarter when submit_workflow_handoff is exposed.
reducer boundaryA reducer runs after the plan tasks its <dependencies> depend on produce outcomes, then completes its <assigned_task>; it may apply trivial inline fixes via edit_file / write_file and then submit reduction success or failure. The attempt PASSES iff every plan task is DONE and FAILS (TASK_FAILED) if any task failed or is unreachable behind a failed dependency.
structural gateA plan must contain at least one reducer. Generator needs can reference one or more generators (or be empty for roots); reducer needs must contain one or more generators and cannot reference reducers; every generator must feed at least one downstream generator or reducer. ordered_plan_tasks enforces this multi-lane DAG shape at planner-submission time.
handoff boundaryA handoff marks the parent generator task WAITING_WORKFLOW. When the child workflow closes, the parent orchestrator runs apply_child_workflow_outcome() and advances ready tasks. There is no closure-report object.
coordination boundaryNo role sends peer-to-peer instructions to another role. Durable fields, task outcomes, rendered role context, and terminal payloads are the handoff surfaces.

Helper Profiles #

advisor

Blocking read-only helper. It audits a parent agent's pending terminal submission and must end with submit_advisor_feedback.

explorer

Read-only subagent for focused investigation. It uses read_file, glob, and submit_exploration_result.

Notification triggers

request_workflow_after_edit reminds executors about handoff timing after edits. The reducer declares no notification triggers; its inline-edit policy is enforced by profile prose plus the advisor pre-terminal gate.

Helper and subagent profiles may omit context_recipe. validate_agent_definitions_resolved() allows definitions with no recipe, and build_explorer_launch_prompt() documents that explorer subagents do not go through the TaskCenter composer path.
