TaskCenter is the persisted multi-agent control plane for complex coding work. It turns a user prompt or delegated task into a Goal, runs one open Iteration at a time, and executes each Attempt through a planner -> generator DAG -> evaluator harness. Current verdict: lifecycle authority is in TaskCenter state and terminal submissions, while engine, tools, and sandbox provide execution surfaces.

Scope And Verdict #

  • The durable state model is Goal -> Iteration -> Attempt. A Goal has ordered Iterations; an Iteration owns sequential retry Attempts; an Attempt owns one planner task, a persisted generator DAG, and one evaluator task.
  • TaskCenter does not add peer-to-peer agent communication. Delegation is persisted as a task-origin child Goal; the parent generator waits in WAITING_GOAL and later resumes through a closure report.
  • The AttemptOrchestrator is per-attempt lifecycle machinery, not a global orchestrator. It persists the planner contract, generator rows, generator outcomes, evaluator outcomes, and attempt close.
  • The Context Engine builds and renders role-specific packets from stores. It does not decide retry, continuation, goal closure, launch policy, or sandbox behavior.
  • Terminal tools are the agent-facing state boundary. Successful terminal tool calls are stamped does_terminate=true by the tool framework, and their payloads mutate TaskCenter through TaskCenter services.

  1. start_task_center_run() validates stores, creates a top-level request/run, prepares a sandbox binding, registers context recipes, and constructs AttemptDeps.
  2. GoalStarter.start() creates an entry-origin Goal, initial Iteration, and initial unstarted Attempt, then starts the attempt through the iteration coordinator.
  3. AttemptOrchestrator.start() registers itself, composes planner launch messages, upserts the planner task, stores the planner task id on the Attempt, and schedules the planner agent.
  4. The planner must terminate with submit_plan_closes_goal or submit_plan_defers_goal. The submission persists the attempt plan contract and generator DAG rows.
  5. AttemptStageAdvancer launches ready generator waves. Dependencies are task ids derived from the planner's local ids, and failed or blocked ancestors make downstream pending tasks unreachable but quiescent.
  6. When every generator is done, TaskCenter starts one evaluator. Evaluator success closes the Attempt as passed; evaluator failure closes it as failed.
  7. The iteration coordinator either emits terminal success, emits success with deferred continuation, or starts the next retry Attempt within the same Iteration budget.
  8. GoalLifecycle starts the next Iteration for deferred continuation, or closes the Goal and delivers a GoalClosureReport to either the top-level run or the parent generator task.

callsSandbox tools mutate or inspect workspace state. Their results become evidence that generator, verifier, or evaluator terminal submissions summarize.

Stale Claim Map #

  • The older standalone TaskCenter HTML still names GoalHandler, IterationManager, IterationClosureRouter, and IterationFactory. Current source names are GoalLifecycle and IterationAttemptCoordinator.
  • The older HTML says child Goal closure transitions the parent task back to RUNNING. Current code routes the closure report into AttemptOrchestrator.apply_goal_closure_report(), which marks the parent generator task DONE on success or FAILED on failure.
  • The older HTML says executor terminal selection is no longer depth-gated. Current TerminalToolRouter removes submit_execution_handoff when nested Goal depth is greater than one.
  • The older HTML mentions entry_executor and obsolete renderer compression behavior. Current built-in recipes are planner, generator, evaluator, and XmlPromptRenderer preserves required/high-priority blocks while truncating low then medium blocks.
