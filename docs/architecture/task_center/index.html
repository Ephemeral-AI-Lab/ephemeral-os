TaskCenter is the persisted multi-agent control plane for complex coding work. Its most important model is Workflow -> Iteration -> Attempt: a Workflow owns the objective, an Iteration owns one bounded planning slice, and an Attempt owns one planner-authored plan — a DAG of generator and reducer tasks (edges are needs). Engine, tools, and sandbox provide execution surfaces around that state model.

Scope And Verdict #

  • The durable state model is Workflow -> Iteration -> Attempt. A Workflow has ordered Iterations; an Iteration owns sequential retry Attempts; an Attempt owns one planner task plus a persisted plan: a DAG of generator and reducer tasks whose edges are needs. There is no separate "node" concept — everything is a task by role.
  • TaskCenter does not add peer-to-peer agent communication. Delegation is a single child-Workflow handoff; the parent generator waits in WAITING_WORKFLOW and later resumes when the child workflow's outcome is applied. There is no closure-report abstraction — "close" is a state transition.
  • The AttemptOrchestrator is per-Attempt machinery, not a global orchestrator. It persists the plan, generator rows, reducer rows, their outcomes, and Attempt close.
  • The Context Engine builds and renders role-specific packets from stores. It does not decide retry, continuation, workflow close, launch policy, or sandbox behavior.
  • Terminal tools are the agent-facing state boundary. Successful terminal tool calls are stamped is_terminal=true by the tool framework, and their payloads mutate TaskCenter through TaskCenter services.

bootstrapstart_task_center_run() validates stores, creates request/run rows, prepares the sandbox binding, validates agent definitions, and constructs AttemptDeps.
rootRunController.start_root_run() seeds a synthetic GENERATOR bootstrap task <run_id>:root (RUNNING), then delegates the root workflow to it — so every workflow, including the root, is generator-spawned.
startWorkflowStarter.start(prompt, parent_task_id) creates the Workflow, initial Iteration, and initial Attempt, then starts the attempt through the iteration coordinator.
planAttemptOrchestrator writes planner task state, launches the planner, and applies its plan submission (generator tasks + task specs + reducers, validated by ordered_plan_tasks). The PLAN stage advances to RUN.
runAttemptStageAdvancer._advance_run_stage() launches ready generator and reducer waves in one pass; failed or blocked ancestors make descendants unreachable but quiescent.
closeWhen the plan reaches quiescence the attempt closes PASSED if every task is DONE, else FAILED with TASK_FAILED. No separate evaluate stage exists.
finishThe iteration coordinator chooses retry, terminal success, or deferred continuation, and projects iteration outcomes (passing reducer outcomes, or failed-task outcomes); WorkflowLifecycle then starts the next Iteration or transitions the Workflow closed.

Ownership Map #

owns-stateRunController owns the synthetic <run_id>:root bootstrap generator: it seeds the task, starts the root workflow, and on close writes the root task's outcomes and finishes the run.
owns-stateWorkflowLifecycle owns Workflow status, the iteration chain, and the close transition (no closure-report object).
owns-stateIterationAttemptCoordinator owns one open Iteration's attempt creation, retry budget, and outcome projection (passing reducer outcomes, or failed-task outcomes) onto the iteration row.
owns-stateAttemptOrchestrator owns attempt stage mutation and task outcome persistence for planner, generators, reducers, and child-workflow parent completion (apply_child_workflow_outcome).
owns-stateAttemptStageAdvancer (in attempt/run_stage.py) owns plan-task wave readiness over the generator+reducer union and the PASSED/FAILED close decision at quiescence.
importsSubmission tools import TaskCenter submission contexts and delegate Workflow -> Iteration -> Attempt mutation to AttemptOrchestrator or WorkflowStarter; they are adapters, not state owners.
callsThe engine query loop calls tools and stops on terminal results, but it does not know Workflow, Iteration, or Attempt policy.
callsSandbox tools mutate or inspect workspace state. Their results become evidence that generator and reducer terminal submissions summarize.

Stale Claim Map #

  • Older docs name an evaluator role, a verifier profile, and an EVALUATE attempt stage. These are gone: the evaluator is replaced by the reducer (a plan task), the verifier profile is removed, and stages collapse to PLAN -> RUN -> CLOSED.
  • Older docs say child Workflow closure delivers a WorkflowClosureReport through a closure_report_router into apply_workflow_closure_report(). There is no closure abstraction now: the parent resumes via AttemptOrchestrator.apply_child_workflow_outcome() (with start_child_workflow / cancel_child_workflow), and Task.child_workflow_id <-> Workflow.parent_task_id link the two directions.
  • Older docs name submit_execution_handoff and WorkflowOrigin. The generator handoff terminal is now submit_workflow_handoff; WorkflowStarter.start(prompt, parent_task_id) has no origin object. DisallowNestedWorkflowHandoff blocks submit_workflow_handoff at nested Workflow depth.
  • Older docs name per-state modules (attempt/state.py, iteration/state.py, workflow/state.py), plan_spec, context packets, and recipe registries. State is consolidated in _core/state.py; plan_spec is removed; launch context is a role-scoped AgentContext.
