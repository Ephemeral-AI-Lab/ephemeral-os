A Scenario is a pure description of what a mocked planner / executor / reducer should submit at each decision point. The mock runner translates those descriptions into real submission tool calls against a real Workflow run, and an audit-bus-fed lifecycle accumulates the events into a rich RunReport for assertions.

REMOVED ⚠️ The Python backend/src/test_runner documented here has been removed. The Rust replacement is specified in integration-test/SPEC.md (see also docs/plans/test_runner_rust_SPEC.md) but is not yet implemented; the evidence paths below describe the removed Python design for historical reference.

Scope #

  • Scenarios live under backend/src/test_runner/scenarios/. Composite scenarios live at the top level; focused scenarios are grouped by concern under subpackages.
  • SCENARIO_REGISTRY is the catalog (~45 entries across pipeline, sandbox, capacity, planner-validation, and composite lanes).
  • Scenarios never own runner state. The mock runner drives real Workflow tool calls, the bus carries mock side-channel events, and ScenarioLifecycle accumulates those records for the report.
  • The decision protocol is intentionally narrow: planner / executor / reducer responses and optional delegated-workflow goal text.

Scenario Protocol #

The protocol is implemented either via ScenarioBase (with overridable defaults) or by satisfying Scenario structurally.

nameDotted-path identifier used by SCENARIO_REGISTRY and as the audit recorder's scenario label. Example: pipeline.initial_workflow, sandbox.background_shell_golden.
planner_response(ctx)Returns a ToolCallSpec(submit_planner_outcome, args). args is the plan: tasks (each {id, agent_name, needs}), task_specs (per-task-id spec text), reducers (each {id, needs, prompt}), and optional deferred_goal_for_next_iteration. The plan is a DAG of generator + reducer tasks joined by needs; there is no plan_spec or evaluation_criteria.
executor_actions(ctx)Returns a sequence of action tokens consumed by the runner's executor dispatcher: "preflight", "fail"/"fail:reason", "delegate_workflow:<pkg>", probe action names. Default is empty.
reducer_response(ctx)Returns a ToolCallSpec targeting submit_reducer_outcome. The reducer is the plan task that decides whether the attempt passes; its task prompt can carry a VERIFY checkpoint=... directive that the executor lane reads back.
delegated_workflow_goal(ctx)Optional goal text used when an executor action calls delegate_workflow; defaults to None (the runner synthesizes a placeholder).

ScenarioContext is the live state visible to the scenario at each decision: attempt, iteration, workflow, prompt, metadata, audit_recorder, task_id, agent_name, instruction, plus optional requirement_ledger, package_plan, and matrix_plan for composite scenarios.

ScenarioLifecycle Wiring #

One builder, one lifecycle. build_scenario_config is the single point of truth where the ScenarioLoopRunner factory and lifecycle are wired together.

build_scenario_config(scenario, ...)
   |
   +--> ScenarioLifecycle()
   +--> runner_factory = (ctx) -> ScenarioLoopRunner(repo_dir, ctx.bus,
   |                              scenario=scenario)
   |
   v
RunConfig (mock mode):
   sandbox = AttachExisting(sandbox_id)
   bootstrap = None
   lifecycle = ScenarioLifecycle (subscribed to bus on_event)
   run_label = "scenario_logs/<scenario.name>"


run_pipeline executes; per event:
   AuditEventBus.publish(event)
      |
      +--> ScenarioLifecycle.on_event(event):
              captured_events.append(event)
              if event.type in MOCK_*:
                  decode payload into LaunchRecord /
                  ToolCallRecord / PromptInspection /
                  SandboxCheck and append
captured_eventsEvery event the bus published in subscription order. Read by run_scenario to populate RunReport.events.
launches / tool_calls / prompt_inspections / sandbox_checksTyped records decoded from the four MOCK_* event payloads. Used by mock tests to assert on prompt content, sandbox correctness, and tool dispatch.

Scenario Catalog #

The registry has five lanes, each owning a concern. Use the lane to pick where a new test should live; do not add cross-lane scenarios.

Pipeline (state machine)

Drives planner / plan DAG (generator + reducer) / RUN-stage control flow with the lightest possible executor action. Failures here mean a regression in workflow/ proper.

Reference scenarios: InitialWorkflow, NestedWorkflow, NestedWorkflowFailure, IterativeDeferral, DependencyDagSerial, DependencyDagParallel, DependencyDagDiamond, DependencyDagMixed, DependencyBlockedDescendants, AttemptRetryPlannerFailure, AttemptRetryGeneratorFailure, AttemptRetryReducerFailure, AttemptBudgetExhausted, GeneratorFailureQuiescence, DeferredParentPlannerTerminalRouting, InitialMessagesCapture.

Sandbox

Exercise LayerStack / OCC / overlay / plugin / ephemeral-workspace / background-shell paths through real submission tools. Failures here mean a regression in sandbox/.

Subgroups: complex project build (ComplexProjectBuild*, ComplexProjectBuildGrepGlob*, ComplexProjectBuildShellEditLsp*), ephemeral workspace (EphemeralWorkspaceAllVerbs, EphemeralWorkspaceConcurrentWrites, EphemeralWorkspaceSamePathConflict, EphemeralWorkspacePolicy, EphemeralWorkspaceCancellation, EphemeralWorkspaceO1Disk), plugin (PluginReadOnlyLspRefresh, PluginWriteAllowedPublish, PluginIntentContract, PluginIwsPolicy, PluginSetupFailure, PluginServiceEvict), background command (BackgroundShellGolden, BackgroundShellStop, BackgroundShellInterleave, BackgroundShellExhaustion, BackgroundShellPartialWriteCancel, BackgroundShellStopDuringMaintenance, BackgroundShellLateCancelRace, BackgroundMixedFgBgSamePathConflict, BackgroundHeartbeatLossReapsOnlyStaleBg, BackgroundExitIwsDrainsAgentTasks, BackgroundEngineRestartNoLeaseLeak, BackgroundManySmallWritesDoNotStarveDispatcher), concurrency (HighConcurrencyLayerstackOverlayOcc, HeavyIoZonedConcurrent, OccConcurrentConflicts, AutoSquashCommitResume).

Capacity

FullSystemCapacityMatrix — composite scenario that walks a pack catalog (capacity/pack_catalog.py) to stress the full Workflow / Iteration / Attempt graph plus sandbox surface together. Skipped unless runner.live_e2e.capacity_enabled is true.

Planner Validation

Negative-path scenarios for planner output validation: PlannerCycleInDeps, PlannerDuplicateLocalId, PlannerEmptyTasks, PlannerBlankDeferredGoal, PlannerUnknownAgentName, PlannerUnknownDep. The mock planner returns an intentionally bad plan; the runner asserts Workflow rejects it through the expected error path.

Composite

Multi-stage scenarios that combine pipeline and sandbox concerns under one entry prompt: CorrectnessTesting, FullCaseUserInput (requirement-ledger / package-plan / recursive-workflow flow), FullStackAdversarial (occ conflict matrix, overlay edge matrix, layerstack squash lease, LSP refresh semantics, recursive oversized matrix, final reconciliation).

Graph-Backed Assertions #

Role lifecycle assertions are derived from persisted Workflow graph state instead of synthetic event hooks. The runner still exposes true audit events for tools and sandbox probes.

count_role_tasks(report, role, status=)Counts plan tasks whose agent_name matches role (planner / executor / reducer) across report.graph_summary. Use status="done" or status="failed" for outcome-specific checks.
count_deferred_attempts(report)Counts attempts that wrote deferred_goal_for_next_iteration, replacing old planner deferral event assertions.
recursive_workflows(graph_summary)Finds nested delegated workflows — those whose parent_task_id does not start with root-. Entry-origin delegated workflows are parented by the request root Task; nested delegated workflows are parented by the launching generator task.
  • Keep sandbox events such as SANDBOX_CONFLICT_DETECTED as event assertions because they are real probe/audit signals.
  • Use graph state for role scheduling and terminal outcomes because the real Workflow stores are the durable source of truth.

Example: InitialWorkflow #

The smallest pipeline scenario — the canonical template for "single-attempt success" tests.

class InitialWorkflow(ScenarioBase):
    name = "pipeline.initial_workflow"

    planner_response(ctx)  -> ToolCallSpec(submit_planner_outcome,
                                           preflight_full_plan())
                              # one generator task "preflight" + one reducer
                              # "reduce" that needs ["preflight"]
    executor_actions(ctx)  -> ("preflight",)            # runs shell pwd + git probe
    reducer_response(ctx)  -> ToolCallSpec(submit_reducer_outcome,
                                           {"summary": "..."})

Run flow under the mock runner (single RUN stage to quiescence):
  planner_response -> submit_planner_outcome
  executor_actions ("preflight") -> command tool probe
                                -> submit_generator_outcome(status="success")
  reducer_response -> submit_reducer_outcome
                  -> attempt PASSES (every plan task DONE) -> workflow closes

Assertions:
  report.workflow_status == "done"
  count_role_tasks(report, "executor", status="done") >= 1
  count_role_tasks(report, "reducer", status="done") >= 1

Branch on ctx.iteration.sequence_no and ctx.attempt.attempt_sequence_no to cover more configurations from the same scenario.
