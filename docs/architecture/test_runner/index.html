The test_runner package is the test-only harness that drives a real Workflow run end-to-end. One coroutine, run_pipeline, serves three modes: mocked-agent scenario tests, real-LLM freeform runs, and the SWE-EVO benchmark CLI. The five mode-specific seams (runner factory, bootstrap, lifecycle, sandbox provisioner, run label) are the only differences between modes.

Scope And Verdict #

  • The harness is dataset-agnostic. The generic scenario runner lives at core/runner.py + core/engine.py; SWE-EVO is one benchmark adapter under benchmarks/sweevo/.
  • One engine, three modes. core/engine.py::run_pipeline is the only orchestrator; RunConfig swaps in a ScenarioLoopRunner for mock scenarios, returns None for real-agent (delegating to start_workflow_run's default runner), or wraps the production runner via build_agent_delegate for the benchmark.
  • The harness is NOT Workflow. It binds stores, audit bus, sandbox lease, and lifecycle around a real Workflow run; it does not own Workflow, Iteration, Attempt, or terminal-tool policy.
  • Mocked agents call real tools. ScenarioLoopRunner injects a deterministic ScenarioEventSource into the real query loop, so scripted mock turns still dispatch through production tool execution; agent definitions come from backend/src/agents/profile/main/*.md, the same markdown the real launcher resolves.
  • Persistent sandbox per SWE-EVO instance. Naming is deterministic (sweevo-<instance_id>); back-to-back runs reuse the container, and the CLI does NOT destroy it on exit. Reclaim with docker container prune.

Ownership Map #

owns-statecore/engine.py::run_pipeline owns the per-run audit-bus subscription, sandbox lease lifecycle, recorder start/stop, timeout-driven cancellation, and final perf-report scheduling.
owns-statecore/stores.py::WorkflowStoreBundle owns the per-test database carving (PostgreSQL schema or SQLite file) and releases it via close().
owns-statescenarios/lifecycle.py::ScenarioLifecycle accumulates captured events, launches, tool calls, prompt inspections, and sandbox checks for the mock-rich RunReport view.
owns-statebenchmarks/sweevo/eval.py::SweevoLifecycle owns the after-run materialize+grade contract, writes sweevo_result.json, and appends one line to aggregate.jsonl.
registersagent/mock/definitions.py::registered_mock_agents swaps the agent registry to the production main-profile markdown definitions for the duration of one mock run, then restores the previous registry on exit.
importsThe mock ScenarioLoopRunner injects scripted turns into the real engine loop; probe bridges import real sandbox and submission tools while scenarios import the runner only indirectly through build_scenario_config.
callsBoth modes call workflow.start_workflow_run with the per-test store bundle and either the mock or delegated runner; the engine never knows which mode it is in.

Stale Claim Map #

  • The benchmark CLI lives at python -m test_runner.benchmarks.sweevo. Older docs may reference a top-level benchmark_sweevo module — that has been relocated under test_runner.benchmarks.sweevo per commit f915434df.
  • The legacy provisioner.py (verify-only) and agent_runner.py (triple-factory) were folded into benchmarks/sweevo/run.py during the migration; SweevoProvisioner and build_agent_delegate are the current names.
  • Daytona-specific lifecycle vocabulary (state) does not apply to the SWE-EVO sandbox model. The benchmark is docker-first; sandbox status uses Docker's status key, never Daytona's state.
  • RunReport carries mock-only fields (launches, tool_calls, prompt_inspections, sandbox_checks). Those are not on PipelineReport; they travel through MOCK_* audit events and are rebuilt by ScenarioLifecycle.
