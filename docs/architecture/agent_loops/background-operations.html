Launch and Alias

Background execution is a query-loop dispatch mode, not a persistent shell session. A tool must advertise background="always" or background="optional"; otherwise the background launcher rejects the call.

The model sees short aliases such as bg_1. These aliases are generated by BackgroundTaskSupervisor.next_alias(), not by the provider tool-use id, and they are the required ids for check, wait, and cancel operations.

Model tool callbackground=true or always-background tool
Stream defersfinal batch still validates terminal exclusivity
Clean inputstrip background, validate real schema
Assign aliasbg_N
Launch coroutineasyncio task plus progress callback metadata
Acknowledge[BACKGROUND LAUNCHED] result names check, wait, cancel
Optional shellThe sandbox shell tool declares background="optional". When launched in the background, its tool context carries background_task_id, and the sandbox ShellRequest sets background=True.
Always-background toolsTools such as run_subagent are routed to the same supervisor path because their metadata requires background execution.
Validation failureInvalid background input returns a normal tool-result error and no supervisor record. Unsupported tools produce a rejection completion event and no alias.

Status Progress and Delivery

BackgroundTaskSupervisor owns the in-memory records for the current agent run. Each record stores the alias, tool name, clean input, task type, agent id, sandbox binding, status, final result, progress lines, and an optional progress provider.

runningCreated at launch with the initial progress line and updated while the asyncio task is active.
completed failed cancelledChosen by the done callback or cancel path, with the latch preserving the highest-precedence terminal outcome during races.
deliveredSet only by collect_completed(), which keeps terminal background output from being surfaced twice.
Provider historyreduce_background_task_history() keeps provider input compact by retaining only the latest meaningful background snapshots, preferring terminal snapshots over stale running snapshots.
Race handlingIf a natural completion and cancellation race, the latch keeps the highest-precedence terminal outcome so a real completed result is not overwritten by a late cancel acknowledgement.

Check Wait Cancel Tools

The background management tools are registered only when the active agent has at least one background-capable tool and is not itself a subagent. They operate on the supervisor stored in tool metadata for the current run.

check_background_task_resultRequires an exact task_id such as bg_1. It returns JSON with id, normalized status, rendered tool_command, and either running progress or the full final result. For subagents, a completed task is finished only when metadata says a terminal tool was called.
wait_background_tasksWaits for all running tasks up to a schema-bounded timeout. It returns compact status snapshots and marks completed tasks delivered so the caller should use check for result bodies.
cancel_background_taskCancels one alias. task_id="auto" resolves only when exactly one task is running; task_id="all" is rejected because cancellation is a mutating operation that must name each task explicitly.

Sandbox Invocation Control

Sandbox control is attached only when the backgrounded tool requires SANDBOX_CONTEXT and a sandbox id exists. The launcher generates a fresh sandbox_invocation_id, stores it on the supervisor record, and passes it through ExecutionMetadata into the tool body.

Engine taskbg_N plus sandbox_invocation_id
Sandbox APIapi.v1.shell payload has background=True
Daemon registrytracks invocation id, agent id, op, last_seen
Heartbeatapi.v1.heartbeat refreshes live background ids
Cancelapi.v1.cancel targets the invocation before local asyncio cancel
TTL reaperstale background invocations are cancelled and marked reaped
Transport contractSandboxTransport.call() puts an invocation id on the daemon envelope. If the payload already has one, the same id correlates the engine background record with daemon in-flight state.
Daemon dispatcherdispatch_envelope_async() registers the current daemon asyncio task in InFlightInvocationRegistry and deregisters it in finally.
Workspace routeOnce inside the daemon, background state does not choose the workspace path. The workspace dispatcher still routes file fast paths through LayerStack/OCC, isolated workspace handles when open, or the overlay pipeline for shell/search/plugin-style operations.

Daemon Ring Audit Events

Background supervision emits background_tool.* events into the daemon audit ring. The schema is part of the daemon audit v1 event families, not a separate log file.

Normal lanebackground_tool.started, completed, failed, cancelled, and delivered are normal-lane events with a background_tool payload.
Sample lanebackground_tool.heartbeat is sample-lane audit traffic and includes current uptime for long-running sandbox-bound work.
PayloadBackgroundToolSection carries background_task_id, task kind, tool name, agent id, status, exit code, duration, error kind, cancel reason, and delivery latency when available.
Pull surfaceConsumers inspect the ring through api.audit.pull, api.audit.snapshot, and guarded api.audit.reset_floor operations.
