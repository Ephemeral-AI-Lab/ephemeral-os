Provider Turn Workflow

The query loop owns one provider-visible turn at a time. It prepares tool metadata, background supervision, terminal-tool names, notification state, provider-safe history, and prompt-report rows before the model-provider stream starts.

The durable transcript remains local to the agent. Provider requests receive a sanitized copy from build_provider_messages(), which compacts stale background snapshots and drops malformed tool-use/result pairs without mutating the run transcript.

Streaming and Deferral

StreamingToolExecutor can execute normal foreground tools as soon as tool-use deltas arrive. The executor itself does not know background or terminal policy; the query loop passes a should_defer predicate.

Terminal-capable runIf context.terminal_tools is non-empty, every streamed tool is deferred until the final assistant message is available. This lets the final batch validator reject a terminal tool batched with siblings before any sibling side effect runs.
Background-bound calldefer_background_dispatch() returns true for background="always", or for background="optional" when the tool input carries background=true. The final dispatcher later launches it through the background path.
Foreground callNon-deferred tools start mid-stream, emit lifecycle/progress events, and are gathered by executor.get_remaining() before final dispatch decides which final-message tool ids remain unresolved.
Failure modeIf the provider stream ends without an AssistantMessageCompleteEvent, the executor cancels in-flight streamed tools and the loop raises a provider-facing runtime error instead of dispatching a partial assistant message.

Budget and Terminal Refusal

tool_call_limit is a soft threshold. Tool execution only increments tool_calls_used; it does not reject calls and it no longer reserves the last call for a terminal tool. Recovery pressure is delivered through notification rules and the query loop's shared overshoot ceiling. The model is nudge then ceiling: the agent is warned at fractions of the limit, reminded once it exceeds the limit, and only forced to stop once overshoot_units exceeds max_tolerance_after_max_tool_call.

Pressure Ramp

One unified budget (overshoot_units = tool_overshoot + text_only_no_terminal_turns) gates both retry and exit. Extra tool calls and text-only turns burn the same currency, so an agent cannot evade the ceiling by alternating modes.

tool_calls_used            overshoot_units            What the loop does
---------------            ---------------            -----------------------------------------
< 50% of limit             0                          (silent)
>= 50% / 75% / 90%         0                          inject budget_warning (one-shot per threshold)
== limit                   0                          (last legal turn — silent)
> limit                    1, 2, 3, ...               inject budget_overflow_reminder (every N units)
text turn, no terminal     +1 per text-only turn      inject missing_terminal_reminder (next turn)
> tolerance, via tools     > tolerance                EXIT — RESOURCE_LIMIT
> tolerance, via text      > tolerance                EXIT — TERMINAL_REFUSED
any turn, terminal_result  (any value)                EXIT — TOOL_STOP (always wins)

Retry Workflow on Cap Reached without Terminal

When the agent burns its soft tool_call_limit without ever calling a terminal tool, the loop does not retry the agent with a fresh budget. Instead it injects increasing nudges through the existing transcript and only stops the run when the shared overshoot_units counter exceeds max_tolerance_after_max_tool_call. The two ways the agent can sit past the soft limit (keep calling tools, or fall back to text) route to two different exit reasons.

Loop Soundness Notes

Soft warning railsmake_budget_warning() fires at configured fractions of the limit. make_budget_overflow_reminder() fires after positive tool_overshoot and repeats on configured overshoot boundaries (default every 5 units). Both inject as user-message blocks at the top of the next iteration via dispatch_rules.
Shared overshootQueryContext.overshoot_units equals tool calls beyond the soft limit plus text-only turns that still omit a terminal call. The hard ceiling is a single comparison against max_tolerance_after_max_tool_call, evaluated on both the tool-dispatch tail and the text-only branch of _run_query_loop.
Resource limit (tool branch)After tool dispatch, if overshoot_units > tolerance the loop sets RESOURCE_LIMIT, cancels pending background tasks, appends the tool results that pair the assistant tool-use blocks (Anthropic Messages API requires tool_use blocks to be paired with tool_result blocks in the next user turn), and emits a single error ToolExecutionCompletedEvent describing overshoot, tolerance, and counters.
Terminal refused (text branch)A terminal-capable run that returns text only routes through the continue arm while tolerance remains. The next iteration's dispatch_rules injects missing_terminal_reminder as a user message before the provider request. If text-only overshoot eventually exceeds tolerance, the loop exits with TERMINAL_REFUSED; the transcript ends with the offending assistant text turn and no synthetic error block is appended.
Terminal wins orderingThe loop checks dispatch.terminal_result before the hard overshoot branch. A successful terminal call ends the run with TOOL_STOP even when the agent already exceeded the soft budget — finishing the work always beats burning the ceiling.
No-tolerance fall-throughIf tolerance is None, a text-only turn cannot route through the nudge-and-continue arm: there is no upper bound on retries, so the loop exits immediately with TEXT_RESPONSE instead. The "retry on max_tool_call reached" workflow above only applies when an agent definition has both terminal tools and a configured tolerance.
Notification rule wiringspawn_agent() appends budget_overflow_reminder and missing_terminal_reminder only when the agent definition carries a tool_call_limit AND at least one terminal tool, and only for rule names not already present on the agent definition (user-supplied rules win).

Run Lifecycle Boundary

run_ephemeral_agent() is the single lifecycle entry point for top-level agent runs and subagent launches. It spawns the agent, drives agent.run(..., auto_close=False), forwards stream events, persists the agent run, closes the provider client, and returns an EphemeralRunResult.

There is no retry-with-fresh-budget mechanism at this layer. If the query loop exits without a terminal result because of RESOURCE_LIMIT, TERMINAL_REFUSED, text response, or crash, run_ephemeral_agent() returns no terminal result; only unhandled exceptions set status="failed" directly through run_error.

Soft recovery remains in-band. The factory attaches default budget-overflow and missing-terminal notification rules when an agent definition has a tool-call limit and at least one terminal tool. Those reminders are user-message blocks in the same transcript, not a hidden outer retry.

            QueryExitReason                EphemeralRunResult                  Caller view
            ----------------                -------------------                 ------------
            TOOL_STOP            ---->     terminal_result = ToolResult        success
            TEXT_RESPONSE        ---->     terminal_result = None              soft fail (in-band)
            RESOURCE_LIMIT       ---->     terminal_result = None              soft fail (in-band)
            TERMINAL_REFUSED     ---->     terminal_result = None              soft fail (in-band)
            unhandled exception  ---->     status = "failed", run_error set    hard fail

            * No fresh-budget retry. Soft pressure (warnings, reminders) is
              the only retry mechanism, and it lives inside the same run.
Factory owns setupspawn_agent() builds the tool registry, registers background management tools only when needed and not for subagents, appends terminal guidance, and constructs the QueryContext.
Query loop owns stop reasonQueryExitReason distinguishes TEXT_RESPONSE, TOOL_STOP, RESOURCE_LIMIT, and TERMINAL_REFUSED.
Lifecycle owns persistenceThe run tracker receives the final transcript, terminal tool payload if present, token count, and crash error. It does not reinterpret background results or retry budget failure.
