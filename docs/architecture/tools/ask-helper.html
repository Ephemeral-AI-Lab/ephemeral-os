ask_advisor synchronously launches a helper agent and returns its terminal output to the caller.

Tool Shape #

backend/src/tools/ask_helper exposes ask_advisor. It is a blocking helper launch, not a background task: it is not listed in engine.background.policy, and it awaits run_ephemeral_agent() directly inside its tool body.

ToolIntentHelper terminalUse case
ask_advisorREAD_ONLYsubmit_advisor_feedbackAudit a pending parent terminal submission before the parent submits it.

Verifier and evaluator agents apply trivial inline fixes themselves via edit_file / write_file on their own tool surface; there is no separate edit-capable helper. Scope is enforced by profile prose plus the advisor's pre-terminal audit, which sees full edit-input transcripts.

Rust port divergence: in the Rust engine, ask_advisor execution is engine-driven — dispatch_assistant_tools intercepts ToolName::AskAdvisor and runs run_ephemeral_agent directly (no AdvisorPort seam), and the advisor approval pre-hook is a stateless transcript scan that additionally gates submit_root_outcome (Python leaves root purely advisory). See docs/reviews/rust_parity/advisor-remediation-PLAN.md.

The helper tool reconstructs the helper launch messages directly instead of using the Workflow context composer. build_helper_messages() resolves the helper agent definition, reads the parent conversation from tool context, extracts the parent's first two user messages, resolves the parent agent definition when possible, captures active terminals, and builds a filtered transcript starting after the parent launch messages.

assemble_user_msg_1() wraps the parent context, parent task, and parent transcript as evidence with a prompt-injection guard. The second helper message is the advisor's terminal catalog plus the pending submission.

If the runtime config, helper agent definition, or parent transcript shape is missing, the tool returns an explicit error ToolResult rather than launching a weakly grounded helper.

Advisor Flow #

  1. The parent supplies the terminal tool it intends to call and the exact payload it intends to submit.
  2. ask_advisor renders the parent's terminal catalog through render_terminal_catalog(..., focus="advisor_review_focus"). If parent terminals cannot be resolved, it uses a fallback note and still runs the helper.
  3. The pending submission is rendered as JSON so the helper can compare tool selection and payload support against the parent transcript.
  4. The helper is launched with role="advisor", agent_type="agent", persist_agent_run=False, and the current sandbox id when present.
  5. The helper must terminate with submit_advisor_feedback. The parent receives the terminal output and metadata, including the advisor verdict.

The advisor sees verifier/evaluator inline edit_file / write_file tool inputs verbatim in the parent transcript — Claude Code's literal Edit / Write / NotebookEdit names are stripped, but EphemeralOS's lowercase tool names are not. That visibility is how the advisor catches inline edits that exceed the typo / single-line scope.

Connections #

engineThe helper launches through engine.api.run_ephemeral_agent and blocks until the helper run returns.
terminal toolsThe helper terminates through its own submission terminal, then the ask-helper tool forwards that terminal output to the parent as the tool result.
WorkflowHelper advice is evidence for the parent. It does not directly close the parent Workflow attempt or bypass the parent's terminal submission contract.
sandboxThe advisor uses normal tool permissions and sandbox routing. Active sandbox id is forwarded, but background task supervision is not involved.
subagent contrastrun_subagent is engine-background and returns through typed subagent controls. The ask helper is a synchronous tool call whose child terminal output is returned immediately to the current tool result.

Verification #

Tests cover helper retry behavior through run_ephemeral_agent, missing terminal-result errors, transcript/message composition, terminal catalog rendering, and advisor submission metadata. The strip-set freeze test (test_advisor_strip_set_is_frozen_to_claude_code_literal_names) protects the verifier/evaluator inline-edit visibility invariant. Refresh this page when helper prompts, transcript filtering, helper terminal contracts, or direct-launch runtime metadata change.
