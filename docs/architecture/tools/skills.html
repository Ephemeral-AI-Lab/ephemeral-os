load_skill_reference is a caller-scoped read-only tool for loading named reference files from the agent's own skill.

Factory Scoping #

Agent definitiondeclares one skill folder
Factory contextpasses caller agent_name
Scoped toolbuilds allowed skill/reference map
Runtime callloads only named references in that scope
Main skill bodyprovided by launch prompt, not this tool
Unavailable namesreturn explicit errors plus available choices
No sandbox routereads skill registry, not workspace files

load_skill_reference is registered as a per-agent factory, not a shared stateless instance. tools._framework.factory registers make_load_skill_reference_from_context; that factory reads ToolFactoryContext.metadata["agent_name"], resolves the agent definition, and scopes the tool to the folder name of that agent's declared skill.

The loaded skill body itself is supplied at agent launch by Workflow skill-message construction. This tool only handles on-demand references under that skill's references/ collection. If no agent skill resolves, the factory intentionally creates a no-op scoped tool with an empty availability map so calls return explicit errors.

Runtime Behavior #

The runtime tool is read-only and accepts two strings: skill_name and reference_name. It does not accept arbitrary paths, directory traversal, or a request for the main skill body. The reference name must match a named reference already indexed for the scoped skill.

CaseOutputWhy it matters
Skill outside the scoped availability mapJSON error with available skill names.Prevents one agent from reading another skill's references.

Architecture Connections #

tool frameworkThe framework creates the tool with caller metadata, so scope is fixed before the model can call it.
engineThe tool is a normal foreground read-only tool. It has no background policy, no sandbox context requirement, and no isolated-workspace lifecycle.
WorkflowWorkflow/agent launch supplies the main skill message. load_skill_reference supplements that launch context with named reference files only.
terminal toolsSkill references can guide terminal-submission quality, but the tool does not submit or mutate Workflow state.
sandboxReference loading reads the process skill registry, not repository files inside a sandbox workspace. It does not publish through OCC and does not allocate an ephemeral or isolated workspace overlay.

Verification #

test_skills_toolkit.py covers successful named-reference resolution, rejection of unscoped skills, no-skill behavior, and context-derived agent-skill scoping. Agent tests cover planner exposure and the main skill-message path. Refresh this page when skill declaration shape, factory metadata, registry loading, or reference error payloads change.
