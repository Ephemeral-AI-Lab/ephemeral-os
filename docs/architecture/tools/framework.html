BaseTool Contract #

BaseTool is deliberately small: a tool is metadata plus one async execute(arguments, context) method. Its metadata is not decorative. The engine and sandbox consume it as control-plane data.

Decorator and Factory #

The @tool decorator converts a Python function into a concrete FunctionTool instance. It requires explicit Pydantic models and explicit intent; validates hook targets once at construction; preserves the original entrypoint for tests; and sets all control metadata on the instance.

When the decorated function accepts context or **kwargs, FunctionTool.execute() writes the tool intent into context["__intent"] before invoking the function. This is the handoff from LLM-facing tool metadata into sandbox/plugin execution policy.

tools._framework.factory owns built-in tool factory registration. It registers sandbox tools, submission tools, ask-helper tools, run_subagent, load_skill_reference, and optional plugin tools unless EOS_SKIP_PLUGIN_IMPORTS_FOR_TESTS is set. register_tool_factory() rejects accidental shadowing unless override=True, while create_tool() ensures the factory returns a tool with the requested name.

Execution Lifecycle #

  1. execute_tool_call_streaming() looks up the named tool in the active ToolRegistry, copies or creates ExecutionMetadata, stamps the tool id, workflow task id, and conversation messages, then builds ToolExecutionContextService.
  2. execute_tool_once() parses raw input through the tool input model after stripping runtime-only fields such as background.
  3. ToolHookExecutionPipeline.run_pre_hooks() can transform the parsed input or deny execution with a structured tool error.
  4. The framework optionally emits ToolExecutionStartedEvent, runs the tool body, catches exceptions as bounded ToolResult errors, and validates successful output against the declared output model.
  5. Post-hooks can validate or replace the ToolResult. Finalization adds effective input, hook trace, and system-notification metadata unless a hook failure already owns the result.
  6. If the tool is terminal and the final result is successful, execute_tool_once() returns a copy with is_terminal=True. Errors and non-terminal tools are not stamped.

Engine Integration Points #

The query loop treats terminal capability as a property of the registered tools. During runtime preparation it derives context.terminal_tools from tools whose is_terminal_tool flag is true. Once any terminal tool is available, the stream dispatcher defers all tool bodies until the complete assistant message is available, so terminal exclusivity can be validated before sibling calls mutate state.

engine.tool_call.dispatch rejects any assistant batch where a terminal tool appears with siblings. For valid batches, background-eligible calls are handed to the background dispatcher and foreground calls use execute_tool_call_streaming(). The first returned block with is_terminal=True becomes the terminal result. The query loop records that result, sets QueryExitReason.TOOL_STOP, and the loop's finally block cancels pending background work before exit.

  • BaseTool metadata is the shared contract: schema for providers, intent for sandbox/plugin policy, terminal flag for query-loop lifecycle, and hooks for tool-local gates.
  • The framework owns validation and result stamping; Workflow and sandbox tools should delegate domain mutation to their own subsystems instead of bypassing the execution lifecycle.
