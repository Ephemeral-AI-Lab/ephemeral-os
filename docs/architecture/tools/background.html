Engine-owned async task plumbing for long-running tools, plus the user-facing tools that check, wait for, and cancel those tasks.

Dispatch Policy #

Background execution is an engine dispatch mode, not a separate tool runtime. StreamingToolExecutor defers tools whose schema says background="always", or background="optional" when the model supplies background=true. The final assistant-message dispatcher then routes deferred calls through dispatch_background_tool_call() when a BackgroundTaskSupervisor exists.

Optional background
A tool with background="optional" runs normally unless its input contains background=true. The background dispatcher strips that flag before validating and executing the real tool input.
Always background
A tool with background="always" is always deferred from foreground streaming and launched through the supervisor. run_subagent is the main current example.
Forbidden background
A tool with no background support, or background="forbidden", is rejected by launch_background_tool() if it reaches the background path.
Executor boundary
The streaming executor only asks a predicate whether to defer. Tool policy remains in engine dispatch and tool metadata rather than in the provider stream consumer.

Record Lifecycle #

BackgroundTaskSupervisor owns the in-memory record map for the current query loop. It creates short aliases such as bg_1, stores the original tool name and clean input, records task_type, agent identity, sandbox binding, live progress, and the final ToolResult.

StatusMeaningDelivery rule
RUNNINGThe asyncio task is still active.check_background_task_result returns progress lines or a progress-provider snapshot.
COMPLETEDThe task produced a successful ToolResult.Undelivered until collect_completed() is called by check/wait or other engine delivery code.
FAILEDThe task raised or returned an error result through the done callback.Undelivered until collected.
CANCELLEDThe supervisor marked the task cancelled.Undelivered until collected; completion has higher terminal precedence than cancellation in races.
DELIVEREDThe terminal state has already been surfaced to the model.Not collected again.

The terminal-status latch orders terminal states as completed over failed over cancelled over running. That keeps a near-complete sandbox command from being overwritten by a late cancel acknowledgement. progress_provider is a pull callback for structured progress; run_subagent uses it to expose the child agent's latest rendered messages.

Management Tools #

backend/src/tools/background exposes three management tools. They operate on the supervisor stored in ToolExecutionContextService metadata for the current agent run.

check_background_task_resultReturns JSON with id, normalized status, rendered tool_command, and either the terminal result or a running progress snapshot. For subagents, a completed task is only reported as finished when subagent_terminal_called metadata is true.
wait_background_tasksBlocks until all running tasks settle or the bounded timeout expires. It returns compact status rows, not result bodies; callers use check_background_task_result for each result they need.
cancel_background_taskCancels one task by alias. task_id="auto" resolves only when exactly one task is running, and task_id="all" is rejected so cancellation remains explicit.

Current completion delivery is polling through these tools. The code still contains historical comments and prompt wording about duplicate [BACKGROUND COMPLETED] messages, but no current automatic BackgroundTaskCompleted event path was found in the inspected engine code.

Sandbox Dispatch #

The background launcher marks a task as sandbox-bound when its tool definition requires SANDBOX_CONTEXT. It then records uses_sandbox, sandbox_id, and a fresh sandbox_invocation_id on the task record, and passes the invocation id through ExecutionMetadata into the actual tool execution.

The supervisor uses that binding for two control paths. Cancellation calls sandbox.api.cancel(sandbox_id, sandbox_invocation_id) before cancelling the asyncio task. While sandbox-bound background tasks are running, a heartbeat loop periodically calls sandbox.api.heartbeat() with active invocation ids so the sandbox side can keep in-flight work visible and cancellable.

This is separate from the workspace execution mode itself. A backgrounded sandbox tool can still be routed by the sandbox daemon into direct LayerStack/OCC fast paths, the ephemeral sandbox workspace overlay pipeline, or the isolated workspace lifecycle depending on the tool and active workspace binding.

Engine Workflows #

  1. The agent factory enables background tasks only when the agent tool registry includes background management tools.
  2. The query loop creates one BackgroundTaskSupervisor for the run and stores it on context.tool_metadata.background_task_manager.
  3. Provider streaming defers always-background and requested optional-background calls so the final assistant batch can still enforce terminal-tool exclusivity.
  4. Deferred background calls are validated, given a bg_N alias, launched as asyncio tasks, and acknowledged with a [BACKGROUND LAUNCHED] tool result.
  5. The model continues the same run and may call check, wait, or cancel. Terminal tools remain TaskCenter-facing completion signals and must still be called through the normal terminal-tool path.
  6. On resource-limit exit or query-loop finalization, pending background tasks are cancelled so engine exits do not leave orphaned sandbox work.
