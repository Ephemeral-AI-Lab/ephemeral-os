Engine-owned async task plumbing for hard-coded background surfaces: subagent launches and PTY command sessions.

Dispatch Policy #

Background execution is an engine dispatch mode, not a separate tool runtime or BaseTool policy. engine.background.policy hard-codes the surfaces that need the background manager: run_subagent is deferred through dispatch_background_tool_call(), while PTY command sessions attach to the same supervisor through typed command-session plumbing.

Engine policyrun_subagent / PTY sessions
Streaming deferralforeground executor skips background-bound calls
Final dispatchervalidates clean input and assigns typed ids

Running taskprogress callback or provider snapshot
Typed deliverynotifies completion or renders running status

Subagent launch
run_subagent is always deferred by the engine policy and launched through the supervisor with subagent_session_id as both record id and model-facing id.
PTY sessions
exec_command and typed PTY controls keep the manager available so interactive commands can be checked, written to, cancelled, and auto-notified by pty_session_id.
Rejected tools
launch_background_tool() rejects any tool that is not allowed by engine.background.policy.is_engine_background_tool().
Executor boundary
The streaming executor only asks a predicate whether to defer. Background policy remains in engine dispatch, not in provider stream consumption or tool schema decoration.

Record Lifecycle #

BackgroundTaskSupervisor owns the in-memory record map for the current query loop. Generic internal background work may still use short aliases such as bg_1; subagent records use subagent_session_id, and PTY commands use pty_session_id. Each record stores the original tool name and clean input, records task_type, agent identity, sandbox binding, live progress, and the final ToolResult.

StatusMeaningDelivery rule
RUNNINGThe asyncio task is still active.Typed control tools return progress lines or a progress-provider snapshot.
COMPLETEDThe task produced a successful ToolResult.Undelivered until the matching typed delivery path marks it delivered.
FAILEDThe task raised or returned an error result through the done callback.Undelivered until the matching delivery path marks it delivered.
CANCELLEDThe supervisor marked the task cancelled.Undelivered until delivered; completion has higher terminal precedence than cancellation in races.
DELIVEREDThe terminal state has already been surfaced to the model.Not collected again.

The terminal-status latch orders terminal states as completed over failed over cancelled over running. That keeps a near-complete sandbox command from being overwritten by a late cancel acknowledgement. progress_provider is a pull callback for structured progress; run_subagent uses it to expose the child agent's latest rendered messages.

Management Tools #

The generic background management tools are retired from model-facing runtime and catalog exposure. The internal supervisor remains, while PTY command tools and subagent controls expose typed ids. Subagent controls live in backend/src/tools/subagent/control.py: check_subagent_progress and cancel_subagent.

Retired generic toolscheck_background_task_result, wait_background_tasks, and cancel_background_task are no longer synthesized for agents or catalog output.
PTY controlscheck_pty_command_progress, write_pty_command_stdin, and cancel_pty_command operate on pty_session_id.
Subagent controlscheck_subagent_progress and cancel_subagent operate on subagent_session_id.

Legacy generic records can still be collected through internal helper paths, but the model-facing surfaces are typed. Subagents and daemon-owned PTY commands have automatic completion bridges: the query loop asks BackgroundTaskSupervisor.collect_subagent_completion_notifications() and collect_pty_completion_notifications() before the next provider request, then injects one typed system notification through SystemNotificationService. If a typed control tool already reports a terminal result, the supervisor marks that record delivered so the automatic bridge does not duplicate it.

Sandbox Dispatch #

The background launcher marks a task as sandbox-bound when its tool definition requires SANDBOX_CONTEXT. It then records uses_sandbox, sandbox_id, and a fresh sandbox_invocation_id on the task record, and passes the invocation id through ExecutionMetadata into the actual tool execution.

The supervisor uses that binding for two control paths. Cancellation calls sandbox.api.cancel(sandbox_id, sandbox_invocation_id) before cancelling the asyncio task. While sandbox-bound background tasks are running, a heartbeat loop periodically calls sandbox.api.heartbeat() with active invocation ids so the sandbox side can keep in-flight work visible and cancellable.

This is separate from the workspace execution mode itself. A backgrounded sandbox tool can still be routed by the sandbox daemon into direct LayerStack/OCC fast paths, the ephemeral sandbox workspace overlay pipeline, or the isolated workspace lifecycle depending on the tool and active workspace binding.

Engine Workflows #

  1. The agent factory enables background tasks when the registry includes run_subagent or PTY session tools. It synthesizes typed subagent controls only for the run_subagent surface.
  2. The query loop creates one BackgroundTaskSupervisor for the run and stores it on context.tool_metadata.background_task_manager.
  3. Provider streaming defers engine-background policy calls so the final assistant batch can still enforce terminal-tool exclusivity.
  4. Deferred subagent calls are validated, given subagent_session_id, launched as asyncio tasks, and acknowledged with [SUBAGENT LAUNCHED]. PTY commands are tracked by pty_session_id. The old generic background management tools are no longer exposed to the model.
  5. The model continues the same run and may call typed subagent or PTY controls. Terminal tools remain TaskCenter-facing completion signals and must still be called through the normal terminal-tool path.
  6. On resource-limit exit or query-loop finalization, pending background tasks are cancelled so engine exits do not leave orphaned sandbox work.
