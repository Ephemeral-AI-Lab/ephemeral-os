Tool hooks are explicit per-tool gates in backend/src/tools/_framework. They are not a global orchestrator, not provider schema fields, and not TaskCenter lifecycle state. The current implementation has a small contract module for hook results and target validation, plus a runtime pipeline that runs hooks around one validated tool body call.

Registration Contract

A hook is attached by passing pre_hooks or post_hooks to the @tool decorator or by setting the same immutable metadata on a BaseTool instance. Hook entries must declare target_tool, and the decorator rejects hooks whose target does not match the tool name. That keeps hook policy explicit at the tool definition instead of allowing a broad, name-sniffing policy layer.

Input contract
ToolPreHook.run(tool_input, context) returns HookResult and may replace the parsed input.
Output contract
ToolPostHook.run(tool_input, result, context) returns HookResult and may replace the result.
Target guard
validate_hook_targets() fails construction when target_tool is missing or mismatched.
Display name
hook_name() uses an explicit name attribute when present, otherwise the hook class name.

Caller-scoped wrapper tools must preserve the same invariant. tools.subagent._factory re-validates hook targets when it wraps a base tool name, even though the current subagent tool has no hook list, so future wrappers keep the same explicit target rule.

Execution Pipeline

  1. execute_tool_once() parses raw input through the tool input model before any hook runs.
  2. run_pre_hooks() executes each pre-hook in tuple order. A pass can replace input; a fail returns a structured hook failure before the tool body starts.
  3. The framework emits ToolExecutionStartedEvent using the effective input and calls the tool body.
  4. The normal output validator checks the body result against the tool output model.
  5. run_post_hooks() executes each post-hook in tuple order. A pass can replace the result; a fail replaces it with a hook failure.
  6. finalize_result() appends hook_trace, effective_tool_input, and pending system notification metadata unless the result is already a hook failure.

Hook failures are tool errors with provider-readable permission metadata: hookEventName is PreToolUse or PostToolUse, permissionDecision is deny, and the metadata includes the hook name, phase, tool name, and reason. Exceptions and malformed hook returns are converted to the same failure shape, so the tool body is not needed to explain hook denial.

  • Pre-hooks can prevent the body from running, so they are the right place for local deny gates.
  • Post-hooks run after output validation and are re-validated if they replace a successful result.
  • Background dispatch still uses the same hook pipeline because background workers call execute_tool_call_streaming().

Sandbox Shell Pre-Hooks

The main concrete hook family is the sandbox shell policy. shell registers DestructiveGitShellPreHook() and DestructiveShellPreHook() as pre-hooks. They inspect the shell command before sandbox API dispatch and block destructive git mutations or high-risk filesystem commands that would bypass the intended write/audit path.

registerstools.sandbox.shell.shell passes both pre-hook instances to the @tool decorator for shell.
callsEach shell pre-hook reads the command field from the parsed input and returns HookResult.fail(...) with policy metadata when blocked.
tested-bytest_shell_prehooks.py checks blocked git commands, allowed read-only git commands, destructive shell commands, and the before-sandbox-requirement failure path.
Boundary Shell pre-hooks are a front-door policy gate for obvious command forms. The sandbox write/audit and OCC publish boundaries remain the authoritative isolation boundary for commands that evade a regular-expression pre-check.

Advisor Approval Pre-Hook

The second concrete pre-hook family gates main-agent terminal submissions on an advisor approval. AdvisorApprovalPreHook in backend/src/tools/submission/_advisor_approval_prehook.py is wired per-terminal: each of the nine main-role submission tools attaches its own instance via pre_hooks=(AdvisorApprovalPreHook("<own_name>"),) in the @tool decorator. Helper / subagent terminals (submit_advisor_feedback, submit_exploration_result) intentionally omit the hook.

relies-onThe "terminal must be called alone" rule in engine/tool_call/dispatch.py guarantees the advisor result is at least one turn old when the hook fires, so the history-only conversation_messages always contains the pair.

does-notCompare payloads. An approve for (tool_name=X, tool_payload=stub) still lets the agent submit X(real_payload); payload-equivalence is out of scope.
does-notBranch on agent role. Exemption is at the terminal-tool level (helper/subagent terminals omit the hook), so a misconfigured subagent calling a main-role terminal still trips the gate.
tested-bytest_advisor_approval_prehook.py covers missing, approve-for-this, approve-for-other, rejected (summary echoed), most-recent-wins ordering, errored advisor, structural verdict, unpaired-after-compaction, and a static guard that helper terminals omit the hook.
Boundary This gate enforces that the advisor was consulted for the exact terminal being submitted. It does not enforce payload quality — the advisor's verdict is authoritative for the named tool, but it does not bind the payload that the agent ultimately sends. Profile prompts (agents/profile/main/*.md) carry the contract that the agent should re-consult if the payload materially changes after approval.
