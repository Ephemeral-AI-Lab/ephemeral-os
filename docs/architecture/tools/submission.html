Submission tools are terminal adapters under backend/src/tools/submission. They validate model output, resolve TaskCenter context, and delegate lifecycle mutation to TaskCenter attempt, workflow, and run services. After the reducers redesign there are three roles that mutate attempt state — planner, generator (executor), and reducer — plus two terminals that only return metadata to their caller: the advisor (helper) and the explorer (subagent). The old verifier and evaluator submission tools are gone.

Registration Map #

make_submission_tools() returns the submission tool set registered by the framework factory: planner outcome submission, generator outcome and workflow handoff submission, reducer outcome submission, the advisor feedback tool, and exploration result submission. There are six submission tools total; ask_advisor is the separate helper request tool used before main-role terminal submissions.

FolderToolsPrimary state owner
plannersubmit_planner_outcomeAttemptOrchestrator.apply_plan_submission().
generatorsubmit_generator_outcome, submit_workflow_handoffAttemptOrchestrator.apply_generator_submission() or WorkflowStarter.start().

explorersubmit_exploration_resultTerminal metadata for exploration findings and references.

The reducer is a plan task that submits a verdict (success / failed) on the slice it gates, replacing both the per-task verifier and the per-attempt evaluator. The registration tests assert these tools exist, that removed names such as submit_plan_defers_goal, submit_generator_success, submit_reduction_failure, submit_execution_success, and submit_execution_blocker are absent, and that all submission tools are terminal except helper request tools such as ask_advisor.

Planner, Executor, Reducer #

submit_planner_outcome resolves an attempt-bound submission context, builds a validated PlannerSubmission from flat tasks (each PlanTaskInput carries id, agent_name, and generator-only needs), task_specs, and reducers (each ReducerInput carries id, one or more generator needs, and a required prompt), then calls AttemptOrchestrator.apply_plan_submission(). The optional deferred_goal_for_next_iteration field names current-iteration goal items intentionally left for the next iteration; omitted or null means the plan covers all current-iteration goal items and leaves no remaining items.

submit_generator_outcome(status, outcome) resolves GeneratorSubmissionContext, which wraps the generator task row, runtime dependencies, and attempt context. It calls apply_generator_submission() with status="success" or status="failed"; the terminal result still records generator_role="executor" because the current generator profile name is executor.

submit_reducer_outcome(status, outcome) resolves the strict resolve_attempt_submission_context() and calls AttemptOrchestrator.apply_reducer_submission() with a ReducerSubmission whose status is the literal "success" or "failed". The reducer is the attempt's exit gate: it completes <assigned_task> using <dependencies>. After either submission the orchestrator runs advance_ready_tasks() over the single RUN stage; the attempt passes once every plan task is DONE and fails (TASK_FAILED) if any task failed or is unreachable behind a failed dependency. There is no separate evaluate stage and no per-task verifier.

Workflow Handoff and Child Workflows #

  1. submit_workflow_handoff(goal_handoff=...) validates a nonblank goal handoff and resolves a generator submission context for the current attempt-bound generator task.
  2. GeneratorSubmissionContext.start_delegated_workflow() constructs WorkflowStarter(runtime=...) and calls WorkflowStarter.start(prompt=goal_handoff, parent_task_id=current_task_id). There is no WorkflowOrigin abstraction — the parent task id alone carries the back-link.
  3. WorkflowStarter validates that the parent generator task is RUNNING and has no open child workflow, then creates the child Workflow (with parent_task_id set), its initial iteration, and its initial attempt. As the attempt starts, it atomically flips the parent from RUNNING to WAITING_WORKFLOW and links child_workflow_id via AttemptOrchestrator.start_child_workflow().
  4. The handoff tool returns metadata containing submission_kind="workflow_start", parent task id, parent attempt id, child workflow id, initial iteration id, and initial attempt id. Because it is terminal and successful, the framework stamps is_terminal=True.
  5. When the child workflow later closes, there is no closure-report abstraction: "close" is a state transition. WorkflowLifecycle._route_close() reads Workflow.parent_task_id, derives the parent attempt from the encoded task id, and calls AttemptOrchestrator.apply_child_workflow_outcome(generator_task=..., child_workflow=..., final_attempt_id=...) on the registered orchestrator.
  6. That handler is idempotent against the WAITING_WORKFLOW guard: it resumes the waiting generator task as DONE on a succeeded child or FAILED otherwise, copies the child workflow's outcomes directly onto the parent generator task, then advances the RUN stage. A failed child-workflow start is rolled back with cancel_child_workflow(), which restores the parent to RUNNING.

This flow is the TaskCenter handoff boundary: agents do not communicate peer-to-peer. The child workflow is persisted state linked bidirectionally (Task.child_workflow_id ↔ Workflow.parent_task_id), and the parent resumes when the child's close transition routes its outcomes back into the orchestrator. The root workflow uses the same path with a synthetic run-level bootstrap generator task <run_id>:root; its close fires the lifecycle's run_close_handler instead of a parent attempt.

Helper and Explorer Results #

submit_advisor_feedback (the helper/advisor terminal) and submit_exploration_result (the subagent/explorer terminal) are terminal tools, but they do not resolve an attempt context or call TaskCenter mutators. They return bounded output text plus metadata that the caller can interpret: advisor verdict and role, or explorer findings and references.

This distinction is intentional. Helper agents terminate their own helper run and provide structured metadata to the parent workflow, while the parent TaskCenter attempt state remains owned by the planner, executor, and reducer submission paths and the orchestrator.

Workflow Connections #

frameworkEvery submit_* function is a decorated BaseTool with is_terminal_tool=True; successful execution is stamped by execute_tool_once().
engineTerminal exclusivity and TOOL_STOP are enforced in engine.tool_call.dispatch and engine.query.loop, not in individual submission tools.
TaskCenterAttempt state changes are centralized in AttemptOrchestrator (apply_plan_submission, apply_generator_submission, apply_reducer_submission). Handoff state changes flow through WorkflowStarter.start() and the three child-workflow handoff methods — start_child_workflow, apply_child_workflow_outcome, cancel_child_workflow — driven by WorkflowLifecycle; there is no closure-report router.
sandboxSubmission tools use Intent.READ_ONLY. They do not perform workspace file mutations directly; generator work that touched the sandbox should already have gone through sandbox tools before terminal submission.
promptingProfile terminal frontmatter and the shared terminal descriptor registry decide which terminal options appear in TaskCenter task guidance and skills; nested-workflow limits are enforced by submission prehooks and soft notifications.
  • Submission tools are adapters, not orchestrators.
  • Parent generator handoff uses persisted child workflows and a close state-transition that routes outcomes back, not direct agent-to-agent calls.
  • Helper terminal outputs return metadata to their caller without mutating TaskCenter attempt state.
