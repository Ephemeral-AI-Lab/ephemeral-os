  • Overlay users need a stable snapshot while other calls keep publishing new changes.
  • LayerStack stores workspace versions as immutable layer directories plus one active manifest.
  • OCC publishes new layers, overlay mounts layer paths, and plugins lease projections from the same manager.
  • LayerStack is the durable truth; overlays and plugin projections are temporary views over that truth.

2.1 Storage §

  • The runtime needs one place to find manifests, layers, staging, metadata, and workspace binding.
  • The storage root contains manifest.json, workspace.json, layers/, staging/, and digest sidecars.
  • Daemon bootstrap builds the base layer; workspace binding prevents path ambiguity between the real workspace and layer-stack root.
  • The repository is copied once into a base layer, then every accepted change appends a delta layer.

2.3 Projection §

  • Tools need the latest workspace view without copying the full repository for each call.
  • A snapshot returns a lease plus ordered layer_paths for the active manifest.
  • Overlay mounts those paths, and MergedView reads them directly for fast read APIs.
  • A normal projection is not a rendered tree by default; it is a pinned list of existing layers.

A snapshot is O(1): a lease plus a list of existing layer paths, not a rendered tree. LayerStack.prepare_workspace_snapshot() reads the manifest, acquires a lease, and returns layer_paths copied from the manifest. It does not project a transient lowerdir for normal operations backend/src/sandbox/layer_stack/stack.py:105-129. The result object contains lease_id, manifest_version, root_hash, manifest, timings, and optional layer_paths backend/src/sandbox/layer_stack/stack.py:43-62. This is what makes N concurrent agents cheap: every snapshot is constant-cost regardless of repository size.

MergedView still exists for direct logical reads and projection. Its project() method applies layers oldest-first into a fresh destination backend/src/sandbox/layer_stack/view.py:195-219. That rendered tree is used by commit-to-workspace, squash checkpoint builds, and special-path maintenance — not by normal per-tool overlay lowerdirs.

2.4 Publish §

  • Concurrent writers need an atomic way to append changes without losing another commit.
  • The publisher stages layer changes, moves the layer into layers/, then atomically writes a new manifest.
  • OCC validates changes before calling the publisher; leases prevent live layers from being removed.
  • Every successful mutation becomes one new immutable layer and one manifest version bump.

This is the atomic-pointer-swap commit pattern used by Git (HEAD updates), Apache Iceberg (metadata.json swap), Delta Lake, and Hudi: a single mutable pointer over a tree of immutable content-addressed objects, swapped via CAS. The active manifest.json is the single linearization point; everything else is append-only.

The publisher performs a CAS check before writing, writes changes into a staging layer, moves it into layers/, then re-checks the manifest before writing the new active manifest backend/src/sandbox/layer_stack/publisher.py:49-132. Delete and opaque directory semantics use overlay-style marker files backend/src/sandbox/layer_stack/changes.py:123-142. Leases refcount every layer in the captured manifest for GC retention (leased_layers()) and expose each lease's head layer as the set of positions squash must keep (lease_head_layers()) backend/src/sandbox/layer_stack/lease.py:38-86.

Why auto-squash exists Manifest depth is bounded for two concrete reasons: (1) util-linux mount(8) rejects overlay mounts beyond ~16 layers even though the kernel mount(2) syscall tolerates ~200; (2) MergedView reads walk newest-first across every layer until a hit, so per-path read cost grows with depth (LSM-tree read amplification). Auto-squash after publish compacts unpinned segments to keep both constraints satisfied.

  1. OCC enters a LayerStack commit transaction and passes the expected manifest to the publisher.
  2. The publisher compares the expected manifest with the current active manifest before writing a layer.
  3. It prepares layer changes and computes a digest, allowing an idempotent duplicate head layer to short-circuit.
  4. It writes a staging layer, fsyncs files/directories, and atomically moves staging into layers/.
  5. It writes a digest sidecar, re-reads the active manifest, and removes the layer if another writer won the race.
  6. If the manifest is still stable, it writes a new manifest with the new layer prepended.

2.5 Squash and Deferred GC §

  • OCC publishes append layers without bound; both overlay mount(8) and MergedView read-amp force a depth cap.
  • LayerStack.squash() folds runs between lease-head layers into checkpoint layers and rewrites the active manifest to use them.
  • Squash is non-destructive: original layer directories stay on disk and remain readable through any lease that still references them. Deletion is deferred to the lease that retains them releasing.
  • Leases hold their full frozen manifest for reads (leased_layers() GC retention); only each lease's head is preserved in active by squash (lease_head_layers()).
Two lease-derived sets leased_layers() is the union of every layer in every active lease's manifest — the on-disk retention set used by _unreferenced_layers at lease release. lease_head_layers() is the newest layer of each lease's manifest — the set squash must keep in place in the active manifest. They are different things and used by different code paths: GC vs. squash planning.
Depth math: heads plus runs A run is a contiguous sequence of active-manifest layers that is not a lease-head barrier. Squash walks the active manifest newest-first, flushes the current run whenever it reaches a lease head, keeps that head as a passthrough layer, and then starts the next run. The resulting depth is therefore distinct lease heads + emitted runs, not just the number of leases plus one. Four distinct lease heads yield depth 5 only when there is a single foldable non-head run; if the heads are spread through the stack with one run above, three runs between heads, and one run below, the post-squash depth is 4 + 5 = 9. The workspace-base live scenario s7_four_lease_heads_create_nine_entry_squash fixes this case: pre-squash depth 14, four distinct lease heads, five non-head runs, post-squash depth 9, and a later no-lease squash back to depth 1 backend/tests/live_e2e_test/sandbox/workspace_base/test_squash_deferred_gc_scenarios.py.

Algorithm. LayerStack.squash(max_depth=...) takes the storage-writer guard, reads the active manifest under the lock, asks the squasher for a plan using lease_head_layers=self._leases.lease_head_layers(), and refuses if the plan is None backend/src/sandbox/layer_stack/stack.py:231-293. Plan acceptance criteria backend/src/sandbox/layer_stack/squash.py:61-93:

  1. Active depth must exceed max_depth.
  2. Segmenting around the lease-head set must produce at least one run of ≥ 2 contiguous non-head layers (a CheckpointSegment).
  3. Number of plan entries must be strictly less than the current depth.
  4. Reduction must meet min_reduction (default 1; can_squash() uses 2).
  5. If post-segmentation entry count still exceeds max_depth and no single segment is larger than max_depth, the plan is rejected — squashing wouldn't reach the cap.

When accepted, squash acquires a synthetic lease against the active manifest (prevents GC racing the checkpoint build), then calls build_checkpoint per segment: allocate a B{v+1:06d}-... staging dir, MergedView.project(staging, Manifest(segment.layers)), and os.replace into layers/. Under the lock again, it verifies the active manifest's prefix is still compatible (manifest_prefix_before_plan), assembles new_layers = live_prefix + [checkpoint | lease-head-passthrough], bumps the version, and writes the manifest atomically. If any check fails, partial checkpoints are discarded via discard_checkpoint. On success, the synthetic squash lease is released, which triggers _unreferenced_layers and removes any layer no longer referenced by the active manifest or any other lease backend/src/sandbox/layer_stack/stack.py:138-150, backend/src/sandbox/layer_stack/stack.py:333-346.

Worked example Setup. Active manifest [L7, L6, L5, L4, L3, L2, L1]. Lease A holds [L6, L5, L4, L3, L2, L1] (head L6). Call squash(max_depth=2).

Plan. lease_head_layers = {L6}. Segmenting around the head produces entries [L7], L6, CheckpointSegment(L5, L4, L3, L2, L1). The third entry is foldable (length 5 ≥ 2).

Result. Active becomes [L7, L6, C]. On disk: L1..L7 and C all coexist — L5..L1 dirs survive because lease A still references them through its frozen manifest. Reads through lease A continue to walk [L6, L5, L4, L3, L2, L1]; new readers using active walk [L7, L6, C].

Eventual deletion. When lease A releases, _unreferenced_layers computes skip = active.layers ∪ leased_layers = {L7, L6, C}. Of lease A's layers {L6, L5, L4, L3, L2, L1}, L6 is in active and is kept; L5..L1 are no longer referenced anywhere and are deleted along with their digest sidecars and view-index cache entries.

Why leases hold the full manifest, not just the head. Reads through a snapshot lease walk its frozen manifest newest-first backend/src/sandbox/layer_stack/view.py:60-118. The bytes answering read_bytes("foo.py") can live in any layer of the chain; a single head ref does not contain a lower layer's contents. Snapshot-by-reference (cheap O(1) acquire + GC-retain the chain) is the design trade vs. snapshot-by-copy (expensive O(workspace) acquire, head ref only).

Concurrency invariants. Storage-writer exclusive guard held end-to-end; manifest-version compare between plan and publish via manifest_prefix_before_plan makes squash OCC-safe against concurrent LayerPublisher.publish_layer; synthetic squash lease prevents _unreferenced_layers from racing the checkpoint build window backend/src/sandbox/layer_stack/stack.py:242-293.

Autosquash audit. OCC post-publish maintenance runs through AutoSquashMaintenancePolicy, which serializes autosquash attempts with a policy-local lock so at most one squash worker enters LayerStack.squash() at a time. The daemon runtime passes emit_squash_event into that policy, so every actual autosquash attempt emits focused critical-lane audit events: layer_stack.squash_triggered with squash_trigger_reason=post_publish_depth and input depth, then layer_stack.squash_completed with input depth, result depth, and manifest root hash, or layer_stack.squash_failed for a raced/aborted plan backend/src/sandbox/occ/maintenance.py, backend/src/sandbox/daemon/occ_runtime_services.py. The public edge-case suite asserts these audit events together with layer_stack.auto_squash.* timings, post-squash depth, and orphan/missing layer counters under complex-project, background-shell-lease, and isolated-workspace-lease conditions backend/tests/live_e2e_test/sandbox/layer_stack_overlay_occ/test_auto_squash_edge_cases.py.
