7. Provider §

Contents

  1. 7.1 Provider Boundary
  2. 7.2 Bootstrap Workflow
  3. 7.3 Evidence and Refresh
  • The Rust sandbox host is Docker-only today; non-Docker provider names fail fast instead of selecting an adapter.
  • eos-sandbox-host owns Docker container CRUD, raw exec, host lifecycle, daemon bootstrap upload, /eos upload transport, and host error mapping.
  • eos-sandbox-api is the typed daemon protocol boundary; it does not select providers or own runtime state.
  • The resident sandbox runtime is the static eosd binary plus eos-daemon workspace, command, plugin, and isolated-workspace handlers.

7.1 Provider Boundary §

The Rust provider boundary is a narrow host-side Docker contract, not an active provider matrix. DockerProviderAdapter is documented as the only Rust production provider agent-core/crates/eos-sandbox-host/src/docker.rs:1-7, provider resolution accepts only docker from config or EOS_SANDBOX_PROVIDER and rejects other values agent-core/crates/eos-sandbox-host/src/registry.rs:18-43, and the host error contract surfaces non-Docker selections as UnknownProviderKind with expected docker agent-core/crates/eos-sandbox-host/src/error.rs:20-23.

implementsDockerProviderAdapter holds the async Docker client and reports ProviderKind::Docker; SandboxLifecycle is the host lifecycle orchestrator over the provider registry and daemon client agent-core/crates/eos-sandbox-host/src/docker.rs:47-74, agent-core/crates/eos-sandbox-host/src/lifecycle.rs:39-47.
selectsProvider selection is Docker-only; a future Rust provider requires an explicit plan instead of reviving the old Python branch by config name agent-core/crates/eos-sandbox-host/src/registry.rs:18-43.
callsSandboxLifecycle creates or starts the container, then runs the same post-lifecycle setup path for runtime upload, git bootstrap, workspace-base binding, and readiness agent-core/crates/eos-sandbox-host/src/lifecycle.rs:60-76, agent-core/crates/eos-sandbox-host/src/lifecycle.rs:150-170.
owns-stateDocker owns container lifecycle and optional daemon TCP exposure; eos-daemon, LayerStack, and OCC own workspace semantics after bootstrap.

7.2 Bootstrap Workflow §

[sandbox config / EOS_SANDBOX_PROVIDER]
        |
        v
[resolve provider: docker only]
        |
        +--> [non-docker] -> [UnknownProviderKind]
        |
        v
[DockerProviderAdapter]
        |
        v
[create or start container]
        |
        v
[upload pinned eosd + ensure workspace base]
        |
        v
[eosd daemon: AF_UNIX / optional TCP]
        |
        v
[eos-sandbox-api typed ops -> eos-daemon handlers]
  1. Runtime composition resolves provider config to Docker; any other value is rejected before lifecycle work begins.
  2. The Docker adapter creates or starts a long-running container with the runtime tmpfs, capability/security settings, daemon TCP labels, and raw exec/put-archive primitives agent-core/crates/eos-sandbox-host/src/docker.rs:145-181, agent-core/crates/eos-sandbox-host/src/docker.rs:478-529.
  3. Post-lifecycle setup overlaps a best-effort background eosd upload with git bootstrap, then performs the authoritative upload and gates on workspace-base readiness agent-core/crates/eos-sandbox-host/src/lifecycle.rs:150-170, agent-core/crates/eos-sandbox-host/src/lifecycle.rs:319-383.
  4. The uploaded eosd binary routes daemon, ns-runner, and ns-holder subcommands; the daemon listens on AF_UNIX and optional loopback TCP sandbox/crates/eosd/src/main.rs:1-15, sandbox/crates/eos-daemon/src/server.rs:1-24.
  5. eos-sandbox-api defines request/result DTOs, daemon op constants, timeout policy, and the transport trait; eos-daemon registers the concrete workspace, command, plugin, audit, and isolated-workspace handlers agent-core/crates/eos-sandbox-api/src/lib.rs:1-19, sandbox/crates/eos-daemon/src/dispatcher.rs:97-149.
BoundaryDocker host ownsSandbox runtime owns

7.3 Evidence and Refresh §

Source files

  • agent-core/crates/eos-sandbox-host/src/docker.rs Docker adapter, container CRUD, raw exec, daemon TCP endpoint, and tar upload.
  • agent-core/crates/eos-sandbox-host/src/lifecycle.rs create/start/delete lifecycle and post-lifecycle bootstrap.
  • agent-core/crates/eos-sandbox-host/src/bootstrap_artifact.rs pinned eosd bootstrap upload and verification.
  • agent-core/crates/eos-sandbox-host/src/sandbox_upload.rs typed /eos-only upload requests and tar payload construction.
  • agent-core/crates/eos-sandbox-host/src/error.rs host error contract, including Docker-only provider rejection.
  • agent-core/crates/eos-sandbox-host/src/registry.rs Docker-only provider selection and per-sandbox adapter bindings.
  • agent-core/crates/eos-sandbox-api/src/lib.rs typed daemon protocol boundary; no provider selection.
  • agent-core/crates/eos-sandbox-api/src/ops.rs daemon operation constants and wire names.
  • sandbox/crates/eosd/src/main.rs static binary dispatcher for daemon, namespace runner, and namespace holder.
  • sandbox/crates/eos-daemon/src/server.rs AF_UNIX / optional TCP server, auth, framing, and shutdown.
  • sandbox/crates/eos-daemon/src/dispatcher.rs daemon op routing for runtime, workspace, command, plugin, audit, and isolated-workspace calls.

Historical comparison

  • Legacy Python provider files may still be useful for parity archaeology, but they are not active Rust evidence.
