1. Overview §

Contents

  1. 1.1 Differentiation and Moat
  2. 1.2 Core Invariants
  3. 1.3 Structural Map
  4. 1.4 Request Shape
  5. 1.5 Evidence and Refresh
  • Many agents and plugins inspect or mutate one repository while sharing a single durable workspace history.
  • The runtime separates durable truth, temporary execution state, conflict control, and daemon dispatch.
  • LayerStack stores versions, overlay exposes normal filesystem views, OCC publishes accepted changes, and the daemon routes calls.
  • Each operation starts from a clear snapshot and can only publish through the controlled OCC path unless it is explicitly isolated scratch.
Design Philosophy The sandbox is the substrate for a shared-state, blackboard-style multi-agent runtime, not a per-agent sandbox vendor. Default ephemeral_workspace publishes every accepted write back into LayerStack, so co-worker agents read each other's output on their next snapshot. This replaces brittle A2A / P2P message protocols with implicit coordination through the artifact itself. isolated_workspace is the explicit opt-out for low-confidence experimentation, where writes are captured for audit but never published.
Shared-state collaboration

Agent A writes a file; on agent B's next read, B sees A's published version. No A2A messaging required; the filesystem is the coordination channel. This is the classical blackboard pattern (Hayes-Roth, HEARSAY-II) applied to LLM agents, with overlayfs + OCC providing the concurrent-write substrate.

O(1) space for N concurrent operations

The repository is copied once into B000001-base. N parallel agents share that base as read-only lowerdir; each operation pays only for its changed bytes in its private upperdir. Disk cost is O(repo) + O(N × changed_bytes), not O(N × repo). Contrast with git-worktree-per-agent designs that pay O(N × checkout) and still miss gitignored deps (node_modules, .venv, target/).

Two workspace modes by intent

ephemeral_workspace is transient per tool call so co-workers see writes immediately through OCC publish. isolated_workspace is persistent per enter/exit lifecycle for an agent's private experimentation; writes are upperdir-only and discarded on exit. The same OverlayFS + LayerStack primitives back both modes.

Fast path for file APIs

read_file / write_file / edit_file skip overlay mount, namespace, and upperdir capture entirely. They bind through workspace.json and submit typed changes straight to OCC. Disjoint-path writes from N agents merge into one batched publish — near-linear throughput with no CAS contention. Shell and search still take the full overlay path because their causal write set isn't known upfront.

1.1 Differentiation and Moat §

  • The defining capability is conflict-resolved merge-back of concurrent agent writes into one shared workspace, layered above the container at the filesystem level.
  • A snapshot is a lease on an immutable manifest, so acquiring it is O(1) and N agents share one base — cost is O(repo) + O(N × changed_bytes), not O(N × repo).
  • Merge policy is split by data nature: tracked source is first-writer-wins (GATED), gitignored deps are last-writer-wins (DIRECT).
  • VM-fork and container-commit sandboxes can isolate or clone, but cannot merge concurrent writes back — that gap is structural, not a tuning detail.
Moat EphemeralOS is the only design in its class that can reconcile what many agents write, not just isolate or fork it. Because layers carry file-level semantics (per-path writes, whiteouts, opaque markers), one agent's overlay capture can be replayed onto the shared manifest under OCC with per-path conflict statuses. Block-level CoW (reflink) sees only opaque extents and full-image commits are coarse and single-shot — neither can express a per-path three-way merge. The merge-back capability is what the rest of the stack (LayerStack leases, overlay capture, OCC routing) exists to deliver.

Three open sandboxes occupy three different layers of the stack, and the layer determines what each can do. The comparison below is external positioning (point-in-time, not code-anchored); only the EphemeralOS row is substantiated by this section's evidence paths.

The asymmetry is not closeable by tuning. Reduced to one line: CubeSandbox forks processes, OpenSandbox isolates tenants, EphemeralOS merges files. Each mechanism enables exactly one thing the others structurally cannot — only file-semantic layers can merge, only block-level memory CoW can fork live execution state. EphemeralOS deliberately forgoes the latter by running agents outside the sandbox, so there is no in-VM execution state to clone (see Provider).

No memory fork
Agents run outside the box and coordinate through artifacts, so there is no in-VM execution state to clone. Shared-state convergence replaces fork-and-retry.
LWW for gitignored deps
Concurrent pip install / npm install conflicts are accepted, not resolved: dependency trees are replaceable wholesale and have no correct three-way merge, and serializing installs would reintroduce the global lock OCC exists to remove. See Routes and Statuses.
Read / maintenance cost
Cheap O(1) snapshots push cost to read-time chain traversal and write-time copy-up, bounded by squash + deferred GC. See LayerStack and Wiki Maintenance.
Where the bet lives The moat guards a specific bet: collaborative multi-agent work (a team converging on one codebase, needing to see each other's in-progress artifacts), not competitive multi-agent work (N agents racing to produce the best independent output, where worktree isolation is correct). The residual risk is whether the collaborative pattern wins adoption — not whether the infra can deliver it. If it does, merge-back is not replicable by VM-fork or container-commit systems without rebuilding their storage layer.

1.4 Request Shape §

Most runtime behavior is easier to understand as three nested boundaries: public API builds a typed request, the host transport wraps it as a daemon envelope, and the daemon router decides the workspace execution path. A direct file write can skip overlay mechanics, but it still publishes through OCC. A shell command usually needs a mounted overlay, then publishes captured upperdir changes. An isolated call reuses the already-open private namespace and discards changes on exit.
