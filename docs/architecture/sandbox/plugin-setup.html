  • The daemon-managed plugin path is manifest driven: stage the payload under /eos, call api.plugin.ensure, then dispatch plugin.<plugin_id>.<op_name>.
  • The sandbox daemon owns workspace freshness, PPC routing, process lifecycle, OCC publish, cleanup, and isolated-workspace blocking.
  • The sandbox host lifecycle bootstraps eosd only; it does not upload plugin packages, runtime bridges, Node, Pyright, or catalog files.
  • The easiest new plugin is a workspace_snapshot_refresh read-only service that answers PPC requests from the current mounted workspace.
  • Write plugins choose either daemon-owned oneshot_overlay execution or a self-managed long-lived service that publishes through daemon callbacks.

Setup Entrypoints §

The Rust host lifecycle has one bootstrap responsibility: upload and verify the pinned eosd daemon artifact, then bind the workspace and wait for runtime readiness. It is not a plugin package setup entrypoint and must not create catalog, runtime bridge, Node, Pyright, or package dependency files. Plugin setup enters the daemon through api.plugin.ensure with a manifest and an already staged payload.

Practical default For a new sandbox service today, start with the daemon-managed service manifest. It is the path covered by the live Rust plugin E2E, and it keeps service mode, refresh strategy, operation intent, and timeout explicit without coupling plugin payloads to daemon bootstrap.

Manifest Contract §

PluginManifest is the daemon input contract: identity fields plus service declarations and public operation declarations sandbox/crates/eos-plugin/src/manifest.rs:11-21. Validation rejects malformed identifiers, duplicate services or operations, operations referencing unknown services, plugin lifecycle intents, read-only operations without a service, and zero timeouts sandbox/crates/eos-plugin/src/manifest.rs:31-57, sandbox/crates/eos-plugin/src/manifest.rs:117-147.

{
  "plugin_id": "my_plugin",
  "plugin_version": "0.1.0",
  "plugin_digest": "my-plugin-digest-v1",
  "services": [
    {
      "service_id": "main",
      "service_profile_digest": "main-profile-v1",
      "service_mode": "workspace_snapshot_refresh",
      "refresh_strategy": "remount_workspace_and_notify",
      "command": ["python3", "/eos/plugin/my_plugin/service.py"],
      "ppc_protocol_version": 1
    }
  ],
  "operations": [
    {
      "op_name": "query",
      "intent": "read_only",
      "service_id": "main",
      "timeout_ms": 5000
    }
  ]
}
plugin_id
Identifier used in public op names: plugin.my_plugin.query.
plugin_digest
Part of service identity. Change it when the deployed payload changes.
service_profile_digest
Part of the service key so two different service profiles do not accidentally share one process.
service_mode
workspace_snapshot_refresh for long-lived services, oneshot_overlay for per-operation write workers sandbox/crates/eos-plugin/src/service.rs:11-21.
refresh_strategy
remount_workspace_and_notify, remount_workspace, or restart_service when a long-lived service is stale sandbox/crates/eos-plugin/src/service.rs:23-36.
auto_workspace_overlay
Defaults true. For self-managed long-lived writes, set it false and publish through daemon callbacks instead of per-operation overlay capture.

Service Process Contract §

For workspace_snapshot_refresh services, the daemon builds a process spec, creates a stable AF_UNIX PPC socket under /eos/plugin/ppc, starts the declared command, and injects environment variables into the process sandbox/crates/eos-daemon/src/plugin/process.rs:1-6, sandbox/crates/eos-daemon/src/plugin/process.rs:28-38, sandbox/crates/eos-daemon/src/plugin/process.rs:90-114.

EOS_PLUGIN_PPC_SOCKET
AF_UNIX socket path the service must connect to.
EOS_PLUGIN_LAYER_STACK_ROOT
LayerStack root backing the service workspace.
EOS_PLUGIN_WORKSPACE_ROOT
Path the service should treat as the mounted workspace root.
EOS_PLUGIN_ID / EOS_PLUGIN_DIGEST
Manifest identity for the running payload.
EOS_PLUGIN_SERVICE_ID
Service id from the manifest.
EOS_PLUGIN_WORKSPACE_MOUNTED
Starts as 0; long-lived Linux services are launched in a private overlay namespace and then see a mounted workspace.

A service can implement the PPC protocol directly, or it can run the bundled Python PPC service bridge. The bridge connects to EOS_PLUGIN_PPC_SOCKET, reads newline-delimited JSON frames, imports plugins.catalog.<plugin>.runtime.server, finds the registered handler, and dispatches the request backend/src/sandbox/ephemeral_workspace/plugin/ppc_service.py:33-74, backend/src/sandbox/ephemeral_workspace/plugin/ppc_service.py:121-167.

Deployment Flow §

  1. Package the payload. Copy the service script/binary and helper files into sandbox-managed /eos locations before ensure. The live harness still stages files directly for daemon protocol coverage; daemon bootstrap is not part of that payload path backend/scripts/bench_rust_daemon_plugin.py:6341-6465.
  2. Build the manifest. Declare services, service modes, refresh strategies, commands, and operations. The live manifest is a broad reference example backend/scripts/bench_rust_daemon_plugin.py:5924-6338.
  3. Ensure the plugin. Call api.plugin.ensure with workspace_root, layer_stack_root, start_services: true, and the manifest backend/scripts/bench_rust_daemon_plugin.py:4208-4220.
  4. Probe status. Call api.plugin.status, optionally with service probes, to confirm connected PPC routes and service health. The daemon registers api.plugin.ensure and api.plugin.status in the main op table backend/src/sandbox/daemon/rpc/dispatcher.py:420-435.
  5. Dispatch operations. Call plugin.<plugin_id>.<op_name>. The daemon chooses connected read-only, daemon-owned oneshot overlay, connected self-managed write, or deferred fallback routing sandbox/crates/eos-daemon/src/plugin/mod.rs:1039-1064.
await call_daemon_api(

    "api.plugin.ensure",

        "workspace_root": "/testbed",
        "start_services": True,
        "manifest": manifest,
    },
    layer_stack_root="/eos/layer-stack",
)

Verification Checklist §

Manifest validationEvery operation references a known service where required; no duplicate service ids or op names; no zero timeout.
Payload installThe declared command exists, is executable when needed, and can import any runtime package files after upload.
Ensure/statusapi.plugin.ensure returns registered routes; api.plugin.status reports expected connected PPC routes and healthy services.
FreshnessAfter a peer publish, a read-only service sees the new LayerStack manifest and does not serve stale cached bytes.
Write safetyWrite operations publish only through daemon-owned OCC paths, either oneshot overlay capture or self-managed callbacks.
CleanupAfter daemon cleanup, connected routes and service processes are gone, active leases are released, and orphan/missing layer counts stay zero.
