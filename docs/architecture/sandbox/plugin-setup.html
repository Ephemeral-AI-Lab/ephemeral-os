  • The daemon-managed plugin path is manifest driven: package identity, setup, services, operations, refresh policy, and PPC routing are declared by a neutral PluginManifest.
  • Agent-core bridges catalog plugins through neutral package descriptors: eos-plugin-catalog owns package source metadata, eos-sandbox-api owns the DTO, eos-sandbox-host owns warm/cold setup, and runtime calls ensure_plugin_package(...) before dispatch.
  • The built-in LSP package is the catalog example: package files live under agent-core/crates/eos-plugin-catalog/lsp, setup writes digest-scoped Node/Pyright material under /eos/runtime/packages/lsp/<digest>, and the service starts through the same generic lifecycle as fixture plugins.
  • Package install is daemon-owned: warm ensure probes digest-local .package-sha256 and .setup-sha256 markers; cold ensure validates staged package content, publishes the digest directory, and runs setup.
  • The sandbox daemon owns workspace freshness, PPC routing, process lifecycle, OCC publish, cleanup, and isolated-workspace blocking.
  • The sandbox host lifecycle bootstraps eosd only; it does not upload plugin packages, runtime bridges, dependency runtimes, or catalog files.
  • The easiest new plugin is a workspace_snapshot_refresh read-only service that answers PPC requests from the current mounted workspace.
  • Write plugins choose either daemon-owned oneshot_overlay execution or a self-managed long-lived service that publishes through daemon callbacks.

Setup Entrypoints §

The Rust host lifecycle has one bootstrap responsibility: upload and verify the pinned eosd daemon artifact, then bind the workspace and wait for runtime readiness. It is not a plugin package setup entrypoint and must not create catalog, runtime bridge, dependency runtime, or package dependency files. Agent-core catalog code builds a neutral PluginPackageDescriptor from package source metadata and model-facing tool specs agent-core/crates/eos-plugin-catalog/src/package.rs. Runtime binds model tools and calls ensure_plugin_package(...) before plugin_dispatch; the host-backed transport performs warm ensure, private cold-path upload under /eos/scratch/uploads/plugins/<plugin-id>/<plugin-digest>/..., and cold ensure with staged_package_root agent-core/crates/eos-runtime/src/plugin_tools.rs, agent-core/crates/eos-sandbox-host/src/plugin_package.rs.

Daemon plugin setup still enters through api.plugin.ensure with the neutral manifest contract consumed by eos-plugin. Warm ensure checks installed package/setup markers; cold ensure validates the staged package root, publishes it to /eos/runtime/plugins/catalog, runs setup with package-version-scoped dependency and temp roots, and records setup failures in plugin status.

The LSP package now uses that same path. Its catalog tree contains plugin.md, sandbox-plugin.json, setup.sh, and runtime/ppc_service.py; setup creates node22, pyright, and npm-cache under the daemon-provided dependency root, and the service computes its Pyright argv from that root agent-core/crates/eos-plugin-catalog/lsp/setup.sh, agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py.

Practical default For a new sandbox service today, start with the daemon-managed service manifest. It is the path covered by the live Rust plugin E2E, and it keeps service mode, refresh strategy, operation intent, and timeout explicit without coupling plugin payloads to daemon bootstrap.

Manifest Contract §

PluginManifest is the daemon input contract: identity fields, package metadata, optional setup, service declarations, and public operation declarations. Validation rejects malformed identifiers, package or setup paths that are absolute or contain .., duplicate services or operations, operations referencing unknown services, plugin lifecycle intents, read-only operations without a service, and zero timeouts sandbox/crates/eos-plugin/src/manifest.rs.

{
  "plugin_id": "my_plugin",
  "plugin_version": "0.1.0",
  "plugin_digest": "my-plugin-digest-v1",
  "package": {
    "runtime_dir": "runtime",
    "dependency_scope": "package_digest"
  },
  "setup": {
    "command": ["./setup.sh"],
    "working_dir": "runtime",
    "setup_marker_digest": "setup-v1",
    "timeout_ms": 30000
  },
  "services": [
    {
      "service_id": "main",
      "service_profile_digest": "main-profile-v1",
      "service_mode": "workspace_snapshot_refresh",
      "refresh_strategy": "remount_workspace_and_notify",
      "command": ["runtime/service"],
      "working_dir": "runtime",
      "ppc_protocol_version": 1
    }
  ],
  "operations": [
    {
      "op_name": "query",
      "intent": "read_only",
      "service_id": "main",
      "timeout_ms": 5000
    }
  ]
}
plugin_id
Identifier used in public op names: plugin.my_plugin.query.
plugin_digest
Canonical package tree identity and the value recorded in .package-sha256.
setup_marker_digest
Setup idempotency identity and the value recorded in .setup-sha256 after setup succeeds.
runtime_dir / working_dir
Package-relative paths only. The daemon derives absolute /eos roots from plugin id and digest.
service_profile_digest
Part of the service key so two different service profiles do not accidentally share one process.
service_mode
workspace_snapshot_refresh for long-lived services, oneshot_overlay for per-operation write workers sandbox/crates/eos-plugin/src/service.rs:11-21.
refresh_strategy
remount_workspace_and_notify, remount_workspace, or restart_service when a long-lived service is stale sandbox/crates/eos-plugin/src/service.rs:23-36.
auto_workspace_overlay
Defaults true. For self-managed long-lived writes, set it false and publish through daemon callbacks instead of per-operation overlay capture.

Service Process Contract §

For workspace_snapshot_refresh services, the daemon builds a generic process spec from the neutral manifest, resolves package-relative executable paths under the declared service working directory, creates a stable AF_UNIX PPC socket under /eos/plugin/ppc, starts the service through the workspace overlay runner on Linux, and injects daemon-derived package, dependency, workspace, and PPC environment variables sandbox/crates/eos-daemon/src/plugin/ensure_args.rs, sandbox/crates/eos-daemon/src/plugin/process.rs, sandbox/crates/eos-daemon/src/plugin/service.rs.

EOS_PLUGIN_PPC_SOCKET
AF_UNIX socket path the service must connect to.
EOS_PLUGIN_LAYER_STACK_ROOT
LayerStack root backing the service workspace.
EOS_PLUGIN_WORKSPACE_ROOT
Path the service should treat as the mounted workspace root.
EOS_PLUGIN_PACKAGE_ROOT
Resolved package root derived from plugin id plus digest.
EOS_PLUGIN_DEPENDENCY_ROOT
Package-version-scoped dependency root under /eos/runtime/packages.
EOS_PLUGIN_ID / EOS_PLUGIN_DIGEST
Manifest identity for the running payload.
EOS_PLUGIN_SERVICE_ID
Service id from the manifest.
EOS_PLUGIN_WORKSPACE_MOUNTED
Starts as 0; long-lived Linux services are launched in a private overlay namespace and then see a mounted workspace.

A service implements the PPC protocol directly: connect to EOS_PLUGIN_PPC_SOCKET, read newline-delimited compact JSON frames, and reply with the same message id. During refresh, the daemon sends prepare_refresh, quiesce, swap_workspace, optional notify_refresh, resume, and health frames before allowing the plugin operation to proceed. The live generic package fixture proves that services should read workspace files through EOS_PLUGIN_WORKSPACE_ROOT after remount, not through an old relative cwd sandbox/crates/eos-e2e-test/tests/plugin/packages.rs.

Deployment Flow §

  1. Build the descriptor. Catalog code produces model-facing tool specs plus one neutral PluginPackageDescriptor containing the daemon manifest and package tree. The LSP descriptor is file-backed by agent-core/crates/eos-plugin-catalog/lsp; runtime keeps no inline service command or package path agent-core/crates/eos-plugin-catalog/src/package.rs, agent-core/crates/eos-runtime/src/plugin_tools.rs.
  2. Warm ensure. Call api.plugin.ensure with the manifest and no staged_package_root. The daemon returns ready only when installed .package-sha256 and .setup-sha256 markers match; otherwise it returns needs_upload without mutating package state sandbox/crates/eos-daemon/src/plugin/package.rs.
  3. Stage the package. ensure_plugin_package(...) keeps upload private to the cold path and stages canonical package content under /eos/scratch/uploads/plugins/<plugin-id>/<plugin-digest>/<upload-id>/package. The live protocol harness fabricates the same tree for daemon coverage, including the LSP regression package agent-core/crates/eos-sandbox-host/src/plugin_package.rs, sandbox/crates/eos-e2e-test/tests/plugin/packages.rs, sandbox/crates/eos-e2e-test/tests/plugin/lsp.rs.
  4. Cold ensure. Call api.plugin.ensure with staged_package_root. The daemon rejects staging outside the digest upload root, validates the package digest, publishes to /eos/runtime/plugins/catalog/<plugin-id>/<plugin-digest>, runs setup when the setup marker is stale, and records .setup-sha256 only after success sandbox/crates/eos-daemon/src/plugin/package.rs, sandbox/crates/eos-daemon/tests/plugin/mod.rs.
  5. Probe status and dispatch. Setup failures are visible through api.plugin.status and prevent service start. After package/setup success, api.plugin.ensure records routes and service state; callers then invoke plugin.<plugin_id>.<op_name> through daemon dispatch sandbox/crates/eos-daemon/src/plugin/mod.rs.

Verification Checklist §

Manifest validationEvery operation references a known service where required; no duplicate service ids or op names; no zero timeout.
Catalog bridgeRuntime obtains package descriptors from eos-plugin-catalog, calls ensure_plugin_package(...), and only then dispatches plugin operations.
Package publishCold ensure validates staged package roots, publishes by digest into /eos/runtime/plugins/catalog, uses copy-then-rename when scratch and runtime are different filesystems, and cleans the upload staging root.
Setup idempotencyMatching .package-sha256 and .setup-sha256 markers skip upload/setup; a changed package digest installs and sets up a separate dependency root.
Setup containmentSetup receives package-version-scoped dependency roots, HOME, and TMPDIR under /eos; package-local scripts that reference /root or /var fail before service start.
Ensure/statusapi.plugin.ensure returns needs_upload for missing packages, returns ready for current packages, and records setup failures in api.plugin.status.
LSP package exampleThe LSP package is installed through the same warm/cold ensure path, starts as a generic PPC service, and proves at least one LSP operation against a workspace file through tests/plugin/lsp.rs.
FreshnessAfter a peer publish, a read-only service sees the new LayerStack manifest and does not serve stale cached bytes.
Write safetyWrite operations publish only through daemon-owned OCC paths, either oneshot overlay capture or self-managed callbacks.
CleanupAfter daemon cleanup, connected routes and service processes are gone, active leases are released, and orphan/missing layer counts stay zero.
