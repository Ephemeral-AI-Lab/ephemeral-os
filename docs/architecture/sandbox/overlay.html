  • Shells and plugins need a normal filesystem, not a layer-by-layer API.
  • Overlay mounts layer paths as read-only lowerdirs plus a private writable upperdir/workdir.
  • LayerStack supplies lower layers; OCC consumes captured upperdir changes.
  • Overlay turns a versioned layer log into a temporary working tree for one operation or one isolated session.

3.1 Overlay Handle §

The overlay lifecycle primitive allocates a per-operation run dir, leases a snapshot, requires layer_paths, allocates upper and work, and returns an OverlayHandle sandbox/crates/eos-overlay/src/kernel_mount.rs. Upper/work directories live under /eos/mount; there is intentionally no fallback because overlayfs requires a suitable writable filesystem sandbox/crates/eos-overlay/src/writable_dirs.rs.

Lowerdirs
Existing immutable LayerStack directories, ordered newest-first and pinned by the snapshot lease.
Upperdir
Only new writes/deletes/opaque markers for this one operation or one isolated session.
Workdir
Kernel-required overlayfs scratch that pairs with the upperdir and is not published.
Handle release
Releases the LayerStack lease and removes the run dir; isolated handles defer this until explicit exit.

3.2 Mount Namespace §

The kernel mount helper calls Linux fsopen, fsconfig, fsmount, and move_mount. It adds each layer with lowerdir+ in manifest order, where the first path is the highest-priority lower sandbox/crates/eos-overlay/src/kernel_mount.rs. Inputs are validated with O_DIRECTORY | O_NOFOLLOW and passed as /proc/self/fd/... paths to pin the checked directories sandbox/crates/eos-overlay/src/kernel_mount.rs.

Does overlay replace the whole filesystem? No. A fresh ephemeral operation runs in a new mount namespace and overmounts only the bound workspace_root. Other inherited filesystem paths remain available subject to denylist policy. Isolated workspace does the same idea once inside a persistent namespace at /testbed. The code mount target is workspace_root, not / sandbox/crates/eos-overlay/src/kernel_mount.rs.

Fresh ephemeral calls write a JSON payload and launch unshare -Urm python -m sandbox.overlay.namespace_entrypoint sandbox/crates/eos-runner/src/fresh_ns.rs. The entrypoint validates/mounts the overlay, runs the tool from workspace_root, unmounts, and writes timings sandbox/crates/eos-runner/src/fresh_ns/child.rs. For isolated workspaces, the runner passes mount_overlay=false and executes inside the already-mounted namespace sandbox/crates/eos-runner/src/setns.rs.

Per-op fixed cost Every fresh ephemeral overlay pays a non-trivial fixed cost: unshare -Urm + Python interpreter startup + mount-syscall sequence + upperdir capture walk + cleanup. Empirically this is in the tens to low hundreds of milliseconds even before the tool's own work runs. isolated_workspace amortizes this by mounting once at enter and reusing the namespace across many tool calls. The file-API fast path avoids it entirely. Choose mode by call-shape: many small writes → fast path; multi-step exploration → isolated; one-shot shell with required write capture → fresh ephemeral.

  1. The operation finishes and the parent pipeline keeps the overlay handle alive.
  2. For write-intent calls, capture_changes(handle) walks only upperdir.
  3. Directory traversal is sorted per level and emits opaque-directory changes before children.
  4. Regular files become disk-backed write changes with content_path and hash.
  5. Whiteout entries become deletes; symlinks become symlink changes; opaque xattrs/markers become opaque-dir changes.
  6. The pipeline converts captured path changes to OCC changes and then destroys the overlay handle.

The upperdir walk uses os.walk() and avoids loading the entire tree into memory sandbox/crates/eos-overlay/src/path_change.rs. Whiteout and opaque handling are explicit helpers sandbox/crates/eos-overlay/src/path_change.rs. This is why changed-data cost is tied to the operation's writes rather than repository size.

Atomicity invariant Capture and publish form one atomic unit per operation. The overlay runs to completion, then capture walks the upperdir once, then OCC commits the entire captured changeset as a single new layer. Other agents never see a partial write set from a shell command in progress; they see the state before the capture started or the state after the publish completed. This is why shell operations cannot use the file-API fast path — their write set is only known after the command finishes, so they must serialize through one atomic publish per captured operation. See Fast-path Technical Workflows for the contrasting file-API path that submits typed changes without capture.
