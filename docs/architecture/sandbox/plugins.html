8. Plugin in ephemeral_workspace §

Contents

  1. 8.1 Host and Daemon Flow
  2. 8.2 Operation Classes
  3. 8.3 Workspace Refresh and Service Writes
  4. 8.4 Evidence and Refresh
  • Plugins need current workspace state without bypassing LayerStack/OCC ownership.
  • Agent-core bridges catalog plugins through a neutral package descriptor: catalog owns source metadata and model tool specs, sandbox-api owns the DTO, sandbox-host owns package setup, and runtime ensures the package before dispatch.
  • LSP is a normal catalog package example: the catalog package tree supplies metadata, setup, and service code; the daemon installs it under digest-scoped package/dependency roots and starts it through generic PPC service lifecycle.
  • The Rust plugin path is manifest driven: package metadata, optional setup, services, operations, refresh policy, and PPC routing are declared in eos-plugin.
  • The daemon parses api.plugin.ensure, probes and publishes package digests, records setup markers/failures, registers public plugin.<plugin_id>.<op_name> routes after setup success, manages service status, and dispatches through PPC or overlay/OCC paths.
  • Plugins are daemon-managed extensions that consume the same snapshot/overlay/OCC pipeline as tools.

8.1 Host and Daemon Flow §

Agent-core catalog code builds a neutral PluginPackageDescriptor from package source metadata and model-facing tool specs agent-core/crates/eos-plugin-catalog/src/package.rs. The descriptor uses the eos-sandbox-api DTO, so sandbox-host consumes package content without depending on catalog internals agent-core/crates/eos-sandbox-api/src/tool_api/plugin.rs. Runtime registers model tools, calls ensure_plugin_package(...), and dispatches plugin.<plugin_id>.<op_name> only after setup succeeds agent-core/crates/eos-runtime/src/plugin_tools.rs.

For the built-in LSP plugin, the catalog package tree is file-backed by agent-core/crates/eos-plugin-catalog/lsp. Its setup script writes node22, pyright, and npm-cache below /eos/runtime/packages/lsp/<digest>; the service computes the absolute Pyright argv from EOS_PLUGIN_DEPENDENCY_ROOT and connects to the daemon PPC socket like any other service agent-core/crates/eos-plugin-catalog/lsp/setup.sh, agent-core/crates/eos-plugin-catalog/lsp/runtime/ppc_service.py.

The host-side package setup API stages package content under /eos/scratch/uploads/plugins/<plugin-id>/<plugin-digest>/... only after a warm daemon probe returns needs_upload, then calls daemon api.plugin.ensure with staged_package_root agent-core/crates/eos-sandbox-host/src/plugin_package.rs. The daemon manifest contract is Rust-owned by eos-plugin: it includes package metadata, optional setup, service declarations, operation declarations, refresh strategy, and digest markers sandbox/crates/eos-plugin/src/manifest.rs.

Inside the daemon, package ensure first checks installed marker state. Warm ensure returns needs_upload when .package-sha256 or .setup-sha256 is missing or stale; cold ensure validates the staged package root, publishes into /eos/runtime/plugins/catalog/<plugin-id>/<plugin-digest>, runs setup with dependency output under /eos/runtime/packages/<plugin-id>/<plugin-digest>, and records setup failures in status sandbox/crates/eos-daemon/src/plugin/package.rs, sandbox/crates/eos-daemon/src/plugin/mod.rs.

After package/setup success, ParsedEnsure decodes the manifest, builds service keys and process specs, and registers public operation routes such as plugin.generic.hover sandbox/crates/eos-daemon/src/plugin/ensure_args.rs. The daemon state then owns loaded plugin digest, registered operation routes, service statuses, service processes, PPC routing, refresh, setup failure reporting, and isolated-workspace blocking sandbox/crates/eos-daemon/src/plugin/mod.rs.

  1. Agent-core catalog code resolves plugin identity, model-facing tool specs, the neutral package descriptor, and package tree.
  2. Runtime calls ensure_plugin_package(...) through the sandbox transport before calling plugin_dispatch.
  3. The host package setup API performs warm ensure, stages package content under /eos/scratch/uploads only when needed, and calls cold api.plugin.ensure.
  4. The daemon validates the staged package, publishes the digest directory, runs setup when the setup marker is stale, then builds service keys, registers public operation routes, and records service status.
  5. The host sends plugin.<plugin_id>.<op_name> through the same daemon transport as built-in tools.
  6. The daemon dispatches through connected PPC services, daemon-owned oneshot overlay, or self-managed callback routes depending on the manifest.

The Rust manifest contract makes operation classes generic. read_only operations must reference a service and route to a connected PPC client after the daemon proves the service is ready on the active manifest key. write_allowed operations either use an oneshot_overlay service with daemon-owned overlay capture, or set auto_workspace_overlay: false and publish through validated daemon callbacks. Lifecycle intent is rejected for plugins in manifest validation sandbox/crates/eos-plugin/src/manifest.rs, sandbox/crates/eos-daemon/src/plugin/mod.rs.

Read-only PPC dispatch is multiplexed by message id and bounded by the plugin response size limit. A failed PPC round trip tears down the connected route and marks the service stopped; a later dispatch can restart the declared service. Self-managed write callbacks are checked against the expected LayerStack root before OCC apply, and oneshot overlay workers run through the daemon overlay path sandbox/crates/eos-daemon/src/plugin/ppc_router.rs, sandbox/crates/eos-daemon/src/plugin/occ_callbacks.rs, sandbox/crates/eos-daemon/src/plugin/overlay.rs.

The live generic package fixture is intentionally non-LSP-shaped: it starts a small PPC service from package content, dispatches plugin.generic.query, verifies package/dependency roots in the service environment, and proves the same operation family is rejected while isolated workspace mode is active sandbox/crates/eos-e2e-test/tests/plugin/packages.rs.

The live LSP fixture uses the same operation class: it installs the LSP package through api.plugin.ensure, starts the pyright service through daemon process supervision, then dispatches plugin.lsp.query_symbols against a small workspace file sandbox/crates/eos-e2e-test/tests/plugin/lsp.rs.

  1. Service start acquires a LayerStack snapshot lease and records the manifest key in service status.
  2. Before dispatch, the daemon compares that service manifest key with the active LayerStack manifest key.
  3. If stale, the daemon singleflights refresh for that service, acquires a new snapshot, and sends PPC refresh frames.
  4. For remount strategies, the daemon remounts the service workspace view through the service process namespace; for restart strategy, it tears down and starts a fresh service.
  5. Only after refresh health succeeds does the daemon send the original plugin operation over PPC.

The live refresh test writes a file through the normal workspace write API, dispatches a generic read-only plugin op, and requires both a service refresh count and the updated file content through EOS_PLUGIN_WORKSPACE_ROOT. This proves the refresh path is generic service lifecycle, not an LSP-specific session shortcut sandbox/crates/eos-e2e-test/tests/plugin/packages.rs.

Daemon change for plugins Plugin ops are registered into the same daemon OP_TABLE, but the daemon blocks plugin access while the caller has an open isolated workspace sandbox/crates/eos-daemon/src/plugin/mod.rs. That avoids a plugin reading/publishing through the shared ephemeral projection while the agent is intentionally inside private isolated scratch.
