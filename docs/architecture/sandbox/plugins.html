  • Plugins need current workspace state without bypassing LayerStack/OCC ownership.
  • The Rust plugin path is manifest driven: package metadata, optional setup, services, operations, refresh policy, and PPC routing are declared in eos-plugin.
  • The daemon parses api.plugin.ensure, registers public plugin.<plugin_id>.<op_name> routes, manages service status, and dispatches through PPC or overlay/OCC paths.
  • Plugins are daemon-managed extensions that consume the same snapshot/overlay/OCC pipeline as tools.

8.1 Host and Daemon Flow §

The host-side package setup API is expected to stage package content under /eos and call daemon api.plugin.ensure with a neutral PluginManifest. In Phase 3 the manifest contract is Rust-owned by eos-plugin: it includes package metadata, optional setup, service declarations, operation declarations, refresh strategy, and digest markers sandbox/crates/eos-plugin/src/manifest.rs.

Inside the daemon, ParsedEnsure decodes that manifest, validates it, builds service keys and process specs, and registers public operation routes such as plugin.generic.hover sandbox/crates/eos-daemon/src/plugin/ensure_args.rs. The daemon state then owns loaded plugin digest, registered operation routes, service statuses, service processes, PPC routing, refresh, and isolated-workspace blocking sandbox/crates/eos-daemon/src/plugin/mod.rs.

  1. Host code resolves sandbox id, layer-stack root, workspace root, caller identity, plugin identity, and package digest.
  2. The host package setup API stages package content under /eos and calls api.plugin.ensure.
  3. The daemon validates the manifest, builds service keys, registers public operation routes, and records service status.
  4. The host sends plugin.<plugin_id>.<op_name> through the same daemon transport as built-in tools.
  5. The daemon dispatches through connected PPC services, daemon-owned oneshot overlay, or self-managed callback routes depending on the manifest.

The op registry makes this generic. READ_ONLY handlers run in-process and should query a long-lived service. WRITE_ALLOWED handlers default to the overlay+OCC publish path. LIFECYCLE is rejected for plugins backend/src/sandbox/ephemeral_workspace/plugin/op_registry.py:15-24. The write wrapper acquires an operation overlay, runs the plugin child, publishes the upperdir, and destroys/releases the handle backend/src/sandbox/ephemeral_workspace/plugin/overlay_dispatch.py:29-90.

Pyright is the concrete stateful plugin example. LSP ops register read-only operations and write operations; write operations such as rename/format/apply opt out of the generic wrapper because the LSP runtime manages its own overlay/OCC path backend/src/plugins/catalog/lsp/runtime/server.py:47-138. session_manager.get_session() reconciles a cached Pyright session to the active manifest key, acquires an overlay handle through ctx.overlay.acquire_operation_overlay or projection fallback, subscribes to workspace change events, and refreshes/remounts when the manifest changes backend/src/plugins/catalog/lsp/runtime/session_manager.py:34-98, backend/src/plugins/catalog/lsp/runtime/session_manager.py:176-221, backend/src/plugins/catalog/lsp/runtime/session_manager.py:294-360.

It starts Pyright inside a private overlay namespace when layer paths are available backend/src/plugins/catalog/lsp/runtime/pyright_session.py:526-567. On refresh, it can remount the running Pyright namespace with fresh layer paths via nsenter and namespace_remount backend/src/plugins/catalog/lsp/runtime/pyright_session.py:583-650.   1. The daemon runtime caches WorkspaceProjection by layer-stack root using the daemon-owned LayerStack manager.
  2. A read-only LSP op asks session_manager.get_session() for a session keyed by active manifest identity.
  3. If the key changed, the session manager acquires a fresh overlay/projection handle or remounts an existing Pyright namespace.
  4. The ephemeral pipeline emits workspace-change events after publishes, and the LSP session pump forwards them to Pyright.
  5. Pyright receives file-watch notifications and refreshes open-document text so diagnostics/hover/rename see the latest projection.

The workspace-change event bus is generic, not Pyright-specific. The ephemeral pipeline exposes subscribe/unsubscribe methods and emits a bounded event queue; if a subscriber queue overflows, it sends a full-resync marker instead of losing correctness silently backend/src/sandbox/ephemeral_workspace/pipeline.py:284-315, backend/src/sandbox/ephemeral_workspace/events.py:51-101. LSP uses that generic stream to drive Pyright refresh, but another plugin could consume the same projection and event model.

Daemon change for plugins Plugin ops are registered into the same daemon OP_TABLE, but the dispatcher blocks plugin access while the caller has an open isolated workspace backend/src/sandbox/daemon/rpc/dispatcher.py:216-235. That avoids a plugin reading/publishing through the shared ephemeral projection while the agent is intentionally inside private isolated scratch.
