The request-level control plane is now Task-first. A top-level request mints a root Task with role=root and workflow_id=NULL, runs the root agent directly, and finishes through submit_root_outcome. A Workflow -> Iteration -> Attempt chain exists only when a running agent calls delegate_workflow. A Workflow owns the delegated objective, an Iteration is the bounded planning slice, and an Attempt is one planner-authored DAG of generator and reducer Task rows.

Task + Workflow Model #

  • Request: owns the user prompt, root task pointer, and final status. It replaces the old run container.
  • Task rows: store root, planner, generator, and reducer agent assignments. The root Task is identified by workflow_id IS NULL; workflow tasks carry workflow_id, iteration_id, and attempt_id.
  • Workflow: owns a delegated objective (workflow_goal), ordered iteration ids, terminal status, and the launching parent_task_id. It stores final projected outcomes on close.
  • Iteration: owns one iteration-local goal string (iteration_goal), a retry budget, and the persisted outcomes projection. Deferred continuation creates another Iteration under the same Workflow.
  • Attempt: owns one planner-authored plan inside one Iteration: planner terminal submission, two task-id tuples (generator_task_ids + reducer_task_ids), and the attempt close.

Workflow Start #

WorkflowStarter.start(*, prompt, parent_task_id) is the prompt-to-Workflow start path for delegated work. It strips and validates the prompt, asserts the parent task is running with no open delegated workflow for that same parent, creates the Workflow row linked by parent_task_id, creates Iteration 1 with a registered IterationAttemptCoordinator, creates the initial Attempt, and starts the Attempt orchestrator. It returns a StartedWorkflow containing workflow, iteration, and attempt ids.

Retry And Deferral #

Retry and deferral are separate Workflow -> Iteration -> Attempt decisions. A failed Attempt can retry inside the same Iteration while budget remains. A passed Attempt can either terminally close the Workflow or carry deferred_goal_for_next_iteration, which closes the current Iteration as succeeded and starts the next Iteration under the same Workflow.

Delegated Workflows #

Delegation creates another complete Workflow -> Iteration -> Attempt chain. It starts from the workflow tool context, not from a direct agent-to-agent channel. delegate_workflow resolves the running task and invokes WorkflowStarter.start(prompt=goal, parent_task_id=current_task_id). There is no close-time mutation of the launching task.

Nested Planning Case #

The nested planning case is a delegated workflow launched from a generator task that itself belongs to another workflow. This is still the same model: the parent generator keeps running with an outstanding handle, the delegated workflow gets its own Iteration and Attempt, and the delegated planner authors its own plan DAG of generators and reducers.

Depth rule A workflow launched by the root task is the first workflow depth. A workflow launched by a workflow generator is nested. Nested planning disables deferred planner submissions; generator delegation remains available but is terminal-gated through workflow handles.

Close And Invariants #

Closing is a state transition, not a report. WorkflowLifecycle.handle_iteration_closed(iteration_id, succeeded, deferred_goal, final_attempt_id) takes primitive args: a succeeded iteration with a deferred goal creates the next iteration; otherwise it calls close_workflow(), then deregisters the closed coordinator. close_workflow() sets workflow status and persists projected outcomes. Root request close is separate and owned by submit_root_outcome.

workflow closeWorkflow close no longer forks into root-vs-parent routes. It updates the workflow row and leaves parent-task synthesis to the launching agent.
root closesubmit_root_outcome marks the root Task DONE or FAILED and finishes the request as done or failed.
process-local registryAttempt orchestrators remain process-local owners for live attempts. Missing orchestration during an active attempt is still a hard invariant violation.
handle idempotencyWorkflow status delivery and cancellation are idempotent through the background workflow-handle record.
state validation_core/invariants.py centralizes checks for open workflows, contiguous iteration and attempt sequence numbers, stage correctness, ownership, fail-reason-present-on-failure, and valid attempt close states.
  • Workflow -> Iteration -> Attempt policy stays in WorkflowStarter, WorkflowLifecycle, IterationAttemptCoordinator, and AttemptOrchestrator.
  • The context engine later reads persisted state. It does not choose retry, deferral, handle delivery, or workflow close policy.
  • The per-attempt orchestrator is not a global agent orchestrator; it only advances one attempt's planner, generators, and reducers.
