The current control plane is Task-first. A top-level request mints a root Task and runs the root agent directly; delegated decomposition uses Workflow -> Iteration -> Attempt only when an agent calls delegate_workflow. A Workflow owns the delegated objective, an Iteration owns one bounded planning slice, and an Attempt owns one planner-authored plan — a DAG of generator and reducer Task rows (edges are needs). Engine, tools, and sandbox provide execution surfaces around that state model.

Scope And Verdict #

  • Current implementation is Rust. eos-runtime owns request bootstrap/root-agent entry; eos-state and eos-db own persisted Request/Task/Workflow/Iteration/Attempt state; eos-workflow owns delegated lifecycle, context, and attempts.
  • The durable request model starts with a root Task. Delegated work then uses Workflow -> Iteration -> Attempt: a Workflow has ordered Iterations; an Iteration owns sequential retry Attempts; an Attempt owns one planner task plus a persisted plan: a DAG of generator and reducer tasks whose edges are needs.
  • The control plane does not add peer-to-peer agent communication. Delegation is a background workflow handle; the parent task stays running, observes or cancels the handle, and later submits its own terminal outcome. Workflow close is a state transition.
  • The AttemptOrchestrator is per-Attempt machinery, not a global orchestrator. It persists the plan, generator rows, reducer rows, their outcomes, and Attempt close.
  • The Context Engine builds and renders role-specific packets from stores. It does not decide retry, continuation, workflow close, launch policy, or sandbox behavior.
  • Terminal tools are the agent-facing state boundary. Successful terminal tool calls are stamped is_terminal=true by the tool framework, and their payloads mutate request/task/workflow state through stores and orchestrators.

bootstrapeos-runtime/src/entry.rs provisions the sandbox binding, creates the request row and root Task, validates agent definitions, wires per-request workflow dependencies, and spawns the root agent.
rootThe root agent runs directly against the root Task. submit_root_outcome marks the root task and request done or failed.
startWorkflowStarter.start(prompt, parent_task_id) creates delegated Workflow, initial Iteration, and initial Attempt, then starts the attempt through the iteration coordinator.
planAttemptOrchestrator writes planner task state, launches the planner, and applies its plan submission (generator tasks + task specs + reducers, validated by ordered_plan_tasks). The PLAN stage advances to RUN.
runAttemptStageAdvancer::advance_run_stage() launches ready generator and reducer waves in one pass; failed or blocked ancestors make descendants unreachable but quiescent.
closeWhen the plan reaches quiescence the attempt closes PASSED if every task is DONE, else FAILED with TASK_FAILED. No separate evaluate stage exists.
finishThe iteration coordinator chooses retry, terminal success, or deferred continuation, and projects iteration outcomes (passing reducer outcomes, or failed-task outcomes); WorkflowLifecycle then starts the next Iteration or transitions the Workflow closed.

Stale Claim Map #

  • Older docs name an evaluator role, a verifier profile, and an EVALUATE attempt stage. These are gone: the evaluator is replaced by the reducer (a plan task), the verifier profile is removed, and stages collapse to PLAN -> RUN -> CLOSED.
  • Older docs say delegated Workflow closure automatically mutates the parent generator. Current delegation returns a workflow handle; the launching agent inspects or cancels it and submits its own terminal outcome.
  • Older docs name terminal delegation tools and workflow-wrapped request bootstrap. Current delegation uses non-terminal delegate_workflow; the root request is a direct root Task completed by submit_root_outcome.
  • Older docs name per-state modules (attempt/state.py, iteration/state.py, workflow/state.py), plan_spec, context packets, and recipe registries. State is consolidated in _core/state.py; plan_spec is removed; launch context is a role-scoped AgentContext.
