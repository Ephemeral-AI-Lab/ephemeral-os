Workflow terminal tools are agent-facing submission endpoints. They are normal registered tools until a successful call crosses the tool framework, where the result is stamped as terminating and the engine loop stops the agent run.

The current shape is intentionally layered: descriptors tell agents when to choose each terminal, submission tools validate and project role payloads, the framework validates output and stamps is_terminal, and the engine owns terminal exclusivity. The advisor helper uses the same terminal contract for its own child run, but its result returns to the caller instead of mutating the parent attempt directly.

  1. make_submission_tools() registers root, planner, generator, reducer, advisor, and explorer submission tools as one framework tool set.
  2. Each terminal tool is decorated with is_terminal_tool=True and a concrete pydantic input model. Most use TextToolOutput, so non-error output must validate as text before the result can terminate the run.
  3. engine.query.loop derives context.terminal_tools from the active tool registry. If a run has any terminals, streamed tool calls are deferred until the final assistant message is available.
  4. engine.tool_call.dispatch rejects any final batch that mixes a terminal tool with sibling tool calls. The rejection returns one error tool result per tool-use id and executes none of the batch.
  5. execute_tool_once() validates input, runs prehooks, emits start, executes the body, validates output, runs posthooks, and stamps non-error terminal results with is_terminal=True.
  6. The dispatcher projects the first terminating result into terminal_result. The query loop sets QueryExitReason.TOOL_STOP and returns without another model turn.

Descriptor Registry #

TERMINAL_DESCRIPTORS is the shared terminal prose registry. Each TerminalToolDescriptor has a parent-facing selection_guidance field and an advisor-facing advisor_review_focus field; render_terminal_catalog() renders either view for a selected terminal list.

Role Submission Terminals #

The role submission terminals are adapters over Task and Workflow state owners. They do not coordinate peer agents directly; they resolve the current task, build typed submissions, and call either the root-task request finisher or the Workflow -> Iteration -> Attempt owner for the current attempt.

Root

submit_root_outcome(status, outcome) is valid only for the root task (workflow_id IS NULL). It marks the root task done or failed, stores the root outcome, and finishes the request. It does not use ContextEngine or an AttemptOrchestrator.

Planner

submit_planner_outcome validates generator tasks, task_specs, and reducers (known needs, >=1 reducer, generator-only dependency DAG, reducers directly gating one or more generators - via ordered_plan_tasks) plus optional deferred_goal_for_next_iteration. There is no plan_spec or evaluation_criteria. Accepted payloads call AttemptOrchestrator.apply_plan_submission().

Executor (generator)

submit_generator_outcome(status, outcome) resolves GeneratorSubmissionContext and calls apply_generator_submission(). Workflow delegation is separate: delegate_workflow is a non-terminal tool that starts background workflow work and returns a handle.

Reducer

submit_reducer_outcome(status, outcome) calls AttemptOrchestrator.apply_reducer_submission(). A reducer is a plan task: its terminal closes that task, and the stage advancer (not the tool body) closes the attempt PASSED once every plan task is DONE, or FAILED if any failed or blocked.

  1. The root agent or executor calls non-terminal delegate_workflow(goal=...).
  2. The workflow tool resolves the current running parent Task, checks that no delegated workflow is already outstanding for that task, and calls WorkflowStarter.start(prompt=goal, parent_task_id=...).
  3. WorkflowStarter creates the delegated workflow, initial iteration, and initial attempt. The parent Task stays running; no waiting status is written.
  4. The caller uses check_workflow_status or cancel_workflow, then folds the delegated result into submit_root_outcome or submit_generator_outcome.

Validation And Stop Path #

Terminal tools terminate only after the normal tool validation pipeline succeeds. A pydantic input failure, a prehook failure, a body exception, an invalid output shape, or a posthook failure returns an error ToolResult; error results are deliberately not stamped as terminal.

Input validation
parse_tool_input() strips engine runtime-control fields such as background, validates the pydantic input model, and returns a retryable invalid-input message on validation errors.
Body execution
execute_tool_body() catches tool exceptions, logs the traceback, and returns bounded error text with the exception type.
Output validation
validate_tool_output() validates non-error output against the tool's output model. Invalid JSON or pydantic mismatch becomes an error result with output_validation_error metadata.
Terminal stamping
execute_tool_once() uses dataclasses.replace() to set is_terminal=True only when tool.is_terminal_tool and the final result is not an error.
Runtime metadata
ExecutionMetadata carries Workflow ids, attempt_runtime, composer, conversation messages, tool registry, background manager, sandbox invocation ids, and extras such as active_terminals.
Loop stop
dispatch_assistant_tools() returns the first terminating result. engine.query.loop stores it on context.terminal_result, sets QueryExitReason.TOOL_STOP, and stops the run.
Failure mode A terminal tool batched with siblings is rejected before any sibling foreground, background, sandbox, or Workflow side effect runs. A terminal tool that returns an invalid output shape is also non-terminal because output validation converts it to an error.

Advisor Before Terminal #

ask_advisor is the read-only validation pattern for terminal outputs before the parent makes its own terminal submission. The parent sends the intended terminal name and payload; the helper receives the parent's original context, original task, filtered transcript, terminal catalog rendered with advisor review focus, and the pending submission JSON.

Rust port divergenceThe Rust port gates submit_root_outcome with the advisor pre-hook too — a deliberate divergence from Python, which gates only planner / generator / reducer and leaves the root terminal purely advisory. In Rust the gate is a stateless transcript scan and ask_advisor is engine-driven (the AdvisorPort seam is deleted; the engine intercepts the call and runs run_ephemeral_agent). Helper Results And Notifications #

Helper terminal outputs are terminal inside the helper run, then ordinary tool results for the parent. This lets helpers use the same engine stop contract while keeping parent Workflow state under the parent submission tools and orchestrator.

Explorer

submit_exploration_result terminates explorer-style work and returns subagent_role="explorer", findings, and references. It does not resolve attempt context or call Workflow mutators.

Nested planner reminder

make_nested_planner_deferral_disabled_reminder() fires for planners launched inside delegated workflows and reminds them to omit deferred_goal_for_next_iteration. Generator workflow delegation uses normal non-terminal tools, not notification-triggered terminal delegation.

  • Advisor direct launches call engine.api.run_ephemeral_agent with persist_agent_run=False and inherit the active sandbox id when one exists.
  • Reducer inline edits via edit_file / write_file use ordinary tool permissions and sandbox routing; there is no separate edit-capable helper.
  • Helper results are evidence for the parent. The parent still must call its own terminal to mutate root-task or Workflow-attempt state.

Tests And Maintenance #

Registration and routing

test_submission_tool_registration.py pins the registered submission names and asserts helper request tools are non-terminal while submission tools are terminal. Main-role terminal tests pin the root, planner, generator, and reducer profile terminal sets plus nested-policy prehooks.

Validation and stop behavior

test_tool_execution.py covers terminal result capture, terminal batch rejection, framework stamping, and non-terminal continuation behavior.

Planner and role submissions

Planner submission tests cover duplicate ids, unknown needs, dependency cycles, empty reducers, task spec matching, close and defer variants, and Workflow -> Iteration -> Attempt advancement through the attempt orchestrator.

Helpers and reminders

Helper tests cover advisor direct launch shape, missing terminal-result errors, helper terminal metadata, explorer output metadata, and nested planner reminder wiring.
