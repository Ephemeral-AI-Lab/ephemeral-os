This page maps how Workflow crosses into the engine loop, tool framework, sandbox API, and live runner artifacts. Current verdict: Workflow owns persisted Workflow -> Iteration -> Attempt state plus task rows; engine owns provider turns and tool execution; sandbox owns workspace execution; the runner records the live artifact tree around those seams.

This boundary map is deliberately not a global orchestrator. Workflow starts a run, prepares one sandbox binding, composes launch messages, schedules attempt-scoped agents, and receives terminal submissions through persisted state. The engine loop remains the enforcement layer for terminal tools, and sandbox operations still pass through tools/sandbox, sandbox.api, and daemon workspace routing.

Entry And Sandbox Provisioning #

start_request() is the entry seam for top-level request work. It receives stores, runtime config, prompt, optional sandbox id, event callback, and optional sandbox provisioner. It creates the request row, mints a root Task with workflow_id=NULL, builds the workflow runtime for future delegated work, and schedules the root agent directly through engine.run_ephemeral_agent. WorkflowStarter.start() is used only after an agent calls delegate_workflow.

[user / external task request]
        |
        v
[runtime/entry.py]
        |
        v
[create request + root Task]
        |  root Task has workflow_id=NULL
        v
[root agent run]
        |  optional delegate_workflow
        v
[WorkflowStarter.start(prompt, parent_task_id)]
        |  creates delegated workflow, first iteration, first attempt
        |
        v
[AttemptOrchestrator + AgentLaunch through engine]
        |
        v
[terminal submissions update request or Workflow state]
preflightruntime/entry.py verifies task, workflow, iteration, and attempt stores before mutating request state.
request shellThe entry path generates ids, prepares the sandbox binding, creates the request row, then creates the root Task row.
runtime_create_runtime() builds AttemptDeps around stores, EphemeralAttemptAgentLauncher, registry, coordinators, config, and AgentEntryComposer.
state startThe root agent runs directly against the root Task. WorkflowStarter.start(prompt=, parent_task_id=) is used only later by delegate_workflow to create delegated Workflow -> Iteration -> Attempt chains.
handleThe entry handle exposes request id, sandbox binding, root task id, and the launcher used by wait_for_idle().
Explicit sandbox id
The runtime sandbox provisioner trims the id, calls start_sandbox, and returns a binding to the existing sandbox.
No sandbox id
The provisioner calls create_sandbox with a generated task-scoped name and labels that include the request id.
Missing created id
If the create response has no id, the provisioner raises RuntimeError("create_sandbox returned no id").
Bootstrap failure
If root task launch fails after request creation, the entry path marks the request failed before re-raising where possible.

Launch To Engine Path #

The attempt launcher is the Workflow-to-engine adapter. It receives an AgentLaunch record from AgentLaunchFactory, attaches Workflow runtime metadata, chooses the launch message shape, and awaits the runner. If no runner is injected, it imports engine.api.run_ephemeral_agent lazily and uses the real engine path.

AgentLaunchFactoryscope and composer output
AgentEntryComposercontext engine, XML renderer, task guidance, skill
ExecutionMetadatarun, task, attempt, workflow, runtime, composer, active terminals
run_ephemeral_agentengine lifecycle and query loop
composer pathAgentEntryComposer.compose() builds an AgentContext from the profile definition, renders <context>, builds task guidance, and optionally builds a skill row.
metadata pathThe launcher creates ExecutionMetadata with request id, task id, attempt id, workflow id, attempt_runtime, composer, and active_terminals. Some metadata field names still carry legacy prefixes for compatibility.
message shapeIf both task guidance and skill exist, the skill is the spawn prompt and the context plus task guidance are seeded through initial_messages. If only task guidance exists, guidance is the prompt and context is the one seeded initial message. Without task guidance, context is the prompt and no initial messages are seeded.
runner contracttest_runner_kwargs_contract.py freezes the runner keyword set: agent_def, sandbox_id, persist_agent_run, task_id, on_event, extra_tool_metadata, and initial_messages.
exhaustion pathIf the runner returns None, crashes, fails, or completes while the Workflow task is still RUNNING, the launcher reports a planner, generator, or reducer failure through the attempt orchestrator (apply_planner_failure / apply_generator_submission / apply_reducer_submission with fail_reason="run_exhausted").

Tool And Sandbox Routing #

The engine does not special-case Workflow sandbox work. Workflow metadata enters the normal ExecutionMetadata object, the tool framework exposes it through ToolExecutionContextService, and sandbox tools build typed sandbox.api requests with caller identity.

[agent sandbox tool call]
        |
        v
[engine tool-call dispatch]
        |
        v
[tools sandbox registry]
        |
        v
[sandbox daemon client]
        |
        v
[workspace_tool_dispatch]
        |
        +--> [direct LayerStack/OCC fast path]
        +--> [ephemeral overlay pipeline]
        +--> [isolated workspace pipeline]
metadataThe spawned agent receives Workflow launch metadata; the query loop preserves identifiers and conversation messages in each tool execution context.
registrySandbox tools carry SANDBOX_CONTEXT, letting runtime preparation attach provider context and background dispatch identify sandbox-bound work.
callersandbox_caller_from_tool_context() builds typed caller identity that includes Workflow ids and the per-call tool id.
daemon opssandbox.api.tool sends typed file, shell, grep, and glob operations through call_sandbox_daemon().
routingdispatch_workspace_tool_call() tries active isolated handles, direct LayerStack/OCC file paths, then overlay execution for shell/search and fallback paths.
Workspace boundary Workflow launches agents outside the sandbox. Workspace reads, writes, shell commands, search, isolated mode, overlay publish, and OCC conflict handling still live behind provider-backed sandbox APIs and daemon workspace routing.

Runner And Live Artifacts #

test_runner.core.engine.run_pipeline() is the live/mock/benchmark harness entrypoint around Workflow. It provisions the sandbox, opens an audit bus, starts AuditRecorder, injects a runner if the mode has one, starts Workflow, waits for the launcher to go idle, releases the sandbox, and writes a performance report task.

Run directory
The default path is audit_dir / run_label / utc_self_id. Reports store the run dir on PipelineReport.run_dir.
Stream audit adapter
stream_bridge() converts ToolExecutionStartedEvent and ToolExecutionCompletedEvent into audit events, preserving tool id, metadata, error state, and is_terminal.
State snapshots
AuditRecorder listens to Workflow, Iteration, Attempt, task, and agent-run ORM events, then writes workflow.json, iteration.json, attempt.json, task.json, run.json, and metrics.json.
Message capture
Primary role task directories get message.jsonl. Initial rows are recorded once from system prompt, seeded initial_messages, and the spawn prompt.
Sandbox events
Sandbox audit events are mirrored to sandbox_events.jsonl, either from daemon event pulling or stream-derived sandbox event projection.
Initial-message regression
test_initial_messages_capture.py asserts captured message.jsonl rows include <context>, <Task Guidance>, skill rows, and terminal-selection catalog metadata for planner/executor/reducer launches.
  • Real engine paths write initial messages via engine.query.request._record_initial_messages_once() and the global agent-run recorder registry.
  • Mock runner paths call AgentMessageJsonlRecorder.record_initial_messages() directly with launch metadata so live scenario assertions inspect the same row shape.
  • The artifact tree is diagnostic evidence, not state authority. Workflow -> Iteration -> Attempt state remains in the stores and is mirrored to files by listeners.

Boundary Rules And Failure Modes #

No global orchestratorAttemptOrchestrator is per Attempt. Entry bootstrap wires registries and stores, but mutation still flows through Workflow, Iteration, Attempt, task, and terminal-submission state.
No peer-to-peer delegationdelegate_workflow starts a persisted workflow handle through WorkflowStarter.start(). The parent task stays running; status delivery or cancellation clears the workflow handle gate, and the parent agent submits its own terminal outcome.
Missing attempt runtimeSubmission context resolvers require ExecutionMetadata.attempt_runtime, a task id, a persisted task row, matching attempt metadata, and a registered orchestrator. Missing pieces return terminal tool errors instead of guessing state.
Runner exhaustionWhen an agent run ends without moving its Workflow task off RUNNING, the launcher synthesizes a planner, generator, or reducer failure submission so the Attempt remains owned by Workflow state code.
Terminal refusalThe engine may return no terminal result for text-only refusal, resource limit, or crash. Workflow treats that as runner exhaustion unless a terminal submission already changed the task state.
Workspace routing surpriseSandbox calls can route isolated, direct, or overlay depending on active isolated handles, file verb, workspace binding, path shape, and daemon availability. Workflow should not assume a file mutation bypasses OCC.
