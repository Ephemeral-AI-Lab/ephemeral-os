Proposed test project — taskflow-mini (a from-scratch Python data-pipeline library)

What it is

Language: Python  ·  Stack: stdlib + pytest + a stdlib http.server demo (no third-party runtime deps)

A fresh Python package built into the emptied /testbed: a small typed data-pipeline/scheduler library 'taskflow' with src/taskflow/{__init__.py,core.py,scheduler.py,operators.py,graph.py,registry.py,serve.py}, a tests/ tree (tests/test_core.py, test_scheduler.py, test_graph.py), pyproject.toml (PEP 621, [project] table editable-installable), a committed .gitignore (ignoring .venv/, __pycache__/, *.pyc), and serve.py exposing a stdlib http.server status endpoint bound to a fixed port 3000. The build proceeds skeleton -> module authoring -> typed refactor (LSP-exercised) -> pytest gate -> a risky data-format migration step (advisor-gated) -> a delegated sub-package built via a handoff child workflow -> commit-back to /testbed git.

Why it fits the dask image & covers every area

The dask image (dask__dask_2023.3.2_2023.4.0) is a Python data-library image with python3, pip, pytest and Pyright available but no node/Next.js (an explorer subagent turn confirms this in iter1, satisfying the 'prefer Python unless node confirmed' constraint and exercising AGEN5/AGEN6 explorer paths). A multi-module typed package naturally needs: MANY files written sequentially (drives manifest depth past AUTO_SQUASH_MAX_DEPTH=100 for squash); parallel generators authoring DISJOINT files (tests/ vs src/, clean OCC) AND a deliberate same-path race on src/taskflow/core.py (OCC ABORTED_VERSION + recoverable retry); typed .py modules that Pyright can hover/diagnose/edit (READ_ONLY remount + WRITE_ALLOWED apply OCC path + stale-base conflict); serve.py runs a real http.server on the fixed port 3000 (two/three IWS bind the SAME port with per-netns isolation, own-loopback curl succeeds, cross-agent peer-IP curl blocked); a long pytest/collection run as background shell work holding a snapshot lease across a squash (deferred GC); a risky in-place data-format migration of registry.py is the advisor-gated action (approve/reject/wrong-tool/missing/advisor_failed); a delegated 'operators' sub-package is built by a depth-2 handoff child workflow; and the whole layerstack is committed back to /testbed git and verified at the end. Stdlib-only keeps the build deterministic and avoids network installs while still exercising venv/editable-install shell steps.

  3. Scaffold src/taskflow/ package tree + tests/ tree via one write-capable shell turn (multi-path overlay_capture).
  4. Author core.py/scheduler.py/graph.py/registry.py/operators.py as typed modules (write_file/edit_file/multi_edit fast path).

  6. Grow manifest depth past 100 with a sequential write_file depth-chain (auto-squash), with an independent disjoint generator alongside.
  7. Run pytest as a background shell holding a snapshot lease across the squash, then release.
  8. Author serve.py (stdlib http.server on port 3000); run the IWS same-port isolation lane inside the provider container.
  9. Risky registry data-format migration as the advisor-gated step (approve path on the closing lane; reject/wrong-tool/missing/advisor_failed on contained lanes).
  10. Delegate the operators sub-package to a depth-2 handoff child workflow; a consumer generator fans in its outcomes.
  11. Reducer (exit gate): read-back + grep/glob verify the built tree, then commit_to_workspace -> apply_layerstack_to_repo materializes /testbed/.git; attempt PASSES.
  12. Verify host-side git lists every materialized path and /testbed/.git survived.
