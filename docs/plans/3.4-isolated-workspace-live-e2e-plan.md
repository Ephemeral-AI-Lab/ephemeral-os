# 3.4 Live E2E Plan - Isolated Workspace

## Goal

Verify that isolated workspace is an explicit, discard-on-exit, pinned lowerdir workspace mode. It must not publish through OCC, must preserve network/filesystem/cgroup isolation, must support same-session parallel tool calls, and must surface enough audit data to debug failures from live artifacts.

## Current Workspace Routing Contract

This plan intentionally differs from default ephemeral mode. In default mode, in-workspace `read_file`, `write_file`, and `edit_file` use the direct layer-stack/OCC fast path; only shell and outside-workspace file paths use the ephemeral per-call overlay pipeline. With an open isolated handle, the same file verbs must route through `IsolatedPipeline.run_tool_call()`, execute inside the persistent isolated overlay/session, capture `changed_paths` for audit, and discard writes on exit without OCC publish.

## Performance Evaluation Contract

Direct isolated-workspace pytest tiers use the daemon audit JSONL, not normal scenario logs, unless a case is intentionally routed through the scenario harness.

Required assertions:

- Lowerdir disk stays O(1) across N enters and N tool calls: concurrent handles share lowerdir layer paths and no materialized workspace tree is created for readers/writers, even though isolated file verbs run through the persistent isolated pipeline rather than the default direct file fast path.
- Per-session disk scales with mutations only: `upperdir_bytes_discarded` approximates the bytes written in that iws session and is zero or near-zero for read-only sessions.
- Latency stays bounded: same-session parallel `sleep 1` calls complete below 1.5x a one-call baseline; `install_veth` max stays within 5x median in Tier 8; every event satisfies SUBSET-COVER for `phases_ms`.
- No OCC publish events appear during enter/tool/exit cycles; default-mode peer publishes must not change an already-pinned lowerdir view.

## Existing Surface

The folder already contains pre-flight, happy-path, isolation, concurrency, network, failure-mode, resource-control, GC/persistence, performance, and stress tiers. This Phase 3.2 plan adds the remaining unified-tool and cross-mode cases needed after workspace unification.

## 3.4.x Complex Cases To Add

### 3.4.1 `behavior_upgrade/test_iws_all_typed_verbs_same_session.py`

Scenario:

1. Enter iws for one agent.
2. Run `write_file`, `read_file`, `edit_file`, `grep`, `glob`, and `shell` against the same mini project.
3. Exit iws.
4. Read the same paths through default mode.

Assertions:

- Each typed verb returns the same result shape as ephemeral mode.
- `edit_file` performs real search/replace, not whole-body overwrite.
- `grep` honors mode/options, not just shell `grep -r -n`.
- `shell` reports `changed_paths`.
- After exit, none of the iws writes exist in default mode.

### 3.4.2 `concurrency/test_iws_parallel_conflicting_upperdir_writes.py`

Scenario:

1. Enter iws.
2. Launch two same-session writes to the same file and two writes to disjoint files.
3. Immediately read all files inside iws.
4. Exit iws and read from default mode.

Assertions:

- Tool calls overlap in wall time; no per-call session lock serializes them.
- Final same-file content is deterministic or explicitly last-writer-wins inside upperdir.
- Disjoint writes are both visible inside iws.
- No OCC publish occurs.
- Exit discards every upperdir write.

### 3.4.3 `isolation/test_iws_peer_publish_pin_and_refresh_boundary.py`

Scenario:

1. Publish `/testbed/base.txt` in default mode.
2. Enter iws.
3. Publish a newer `/testbed/base.txt` from a peer default-mode agent.
4. Read and edit `/testbed/base.txt` inside iws.
5. Exit iws, then read from default mode.

Assertions:

- Iws sees the pinned lowerdir version from enter time.
- Iws edit affects only private upperdir.
- Default mode sees the peer publish, not the iws edit.
- Audit enter event includes manifest version/root hash that matches the pinned lowerdir.

### 3.4.4 `policy/test_plugin_and_lsp_blocked_or_routed_in_iws.py`

Scenario:

1. Enter iws.
2. Invoke a generic plugin tool.
3. Invoke an LSP READ_ONLY operation.
4. Exit iws and repeat both operations in default mode.

Assertions:

- Generic plugin tools are forbidden in iws unless explicitly supported.
- LSP either rejects with a typed `forbidden_in_isolated_workspace` result or routes through the documented iws-safe path.
- Default mode plugin/LSP behavior remains unchanged.
- Fail-open behavior only applies when the iws pipeline is not bootstrapped, not when an iws handle is open.

### 3.4.5 `gc_and_persistence/test_iws_daemon_restart_mid_parallel_calls.py`

Scenario:

1. Enter iws.
2. Start multiple long-running tool calls.
3. Kill the daemon mid-flight.
4. Trigger daemon respawn.
5. Re-enter with the same agent.

Assertions:

- Startup GC reaps scratch dirs, veths, netns, cgroups, and leases from the dead session.
- The new enter gets a fresh handle and fresh IP allocation.
- Audit JSONL records orphan cleanup events with `kind`, `identifier`, `total_ms`, and `phases_ms`.
- No old upperdir content leaks into the new session.

### 3.4.6 `performance/test_iws_parallelism_and_phase_budget.py`

Scenario:

1. Establish a baseline from one `sleep 1` tool call.
2. Run two concurrent `sleep 1; touch ...` calls in one iws.
3. Run five agents concurrently with one iws each.

Assertions:

- Same-session two-call wall time is below 1.5x the one-call baseline.
- Five-agent enter contention stays bounded by the existing Tier 8 rule.
- `sum(phases_ms) <= total_ms + max(2ms, 5 percent)` for every event.
- Lowerdir layer paths are shared across handles and do not consume per-handle workspace-tree disk.
