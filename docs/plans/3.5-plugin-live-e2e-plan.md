# 3.5 Live E2E Plan - Plugin

## Goal

Create the missing TaskCenter runner live E2E tier for plugin tools after workspace unification. The tier must prove READ_ONLY plugin tools use long-lived services without per-call overlay publish, WRITE_ALLOWED plugin tools use plugin overlay + OCC semantics, and plugin behavior is explicit around isolated workspace boundaries.

## Current Workspace Routing Contract

Default in-workspace `read_file`, `write_file`, and `edit_file` use the direct layer-stack/OCC fast path. Do not use normal `write_file` / `edit_file` overlay timings as the baseline for plugin WRITE_ALLOWED behavior. Plugin WRITE_ALLOWED operations are still overlay-backed plugin operations when they need namespace/plugin runtime execution; normal `read_file` after a plugin write should be direct and should prove the committed layer-stack tip is visible.

## Performance Evaluation Contract

Plugin live tests must separate plugin service latency from overlay/OCC latency.

Required assertions:

- READ_ONLY plugin service calls do not allocate per-call operation overlays and therefore do not create lowerdir workspace-tree disk. LSP refresh should remount the long-lived namespace, not restart Pyright for every normal write.
- WRITE_ALLOWED plugin tools follow the same O(1) lowerdir contract as overlay-backed ephemeral operations: `workspace_tree_bytes == 0`; `upperdir_tree_bytes` scales with changed files; run dirs are removed after publish. Normal in-workspace file verbs remain direct fast-path calls and should be measured separately.
- LSP service latency remains low: warm hover/diagnostics p95 below 500ms for small projects; refresh/remount count may increase after writes, but start count should not increase in the normal path.
- Plugin setup failures must be classified separately from runtime latency, so network/setup problems do not look like overlay/OCC regressions.

## 3.5.x Complex Cases To Add

### 3.5.1 `test_plugin_read_only_lsp_refresh_without_publish.py`

Scenario:

1. Write a Python module in default ephemeral mode.
2. Run LSP hover/definitions/diagnostics through plugin tools.
3. Edit the module in default mode.
4. Run LSP diagnostics again.

Assertions:

- READ_ONLY LSP calls do not allocate operation overlays or publish OCC changes.
- Pyright session observes the edit through `subscribe_workspace_changes`.
- Normal refresh remounts the private namespace without restarting Pyright.
- Diagnostics reflect the post-edit content.

### 3.5.2 `test_plugin_write_allowed_apply_workspace_edit_publishes.py`

Scenario:

1. Seed a Python file.
2. Invoke a WRITE_ALLOWED plugin operation such as rename/format/apply workspace edit.
3. Read the file through normal sandbox API.

Assertions:

- Plugin write operation allocates overlay, runs in namespace child, captures changed paths, publishes OCC, and releases.
- Result contains plugin overlay timing plus published manifest version.
- Normal `read_file` sees the plugin change.
- Audit shape matches normal WRITE_ALLOWED tools.

### 3.5.3 `test_plugin_intent_mislabel_fails_fast.py`

Scenario:

1. Register a fake plugin tool without intent.
2. Register a fake plugin tool with `Intent.LIFECYCLE`.
3. Register READ_ONLY and WRITE_ALLOWED controls.

Assertions:

- Missing intent fails at import/registration.
- `Intent.LIFECYCLE` is rejected for plugin tools.
- READ_ONLY control dispatches in-process through service path.
- WRITE_ALLOWED control dispatches through overlay path.

### 3.5.4 `test_plugin_blocked_in_open_isolated_workspace.py`

Scenario:

1. Enter iws for an agent.
2. Invoke generic plugin ensure and a dynamic plugin op with that agent.
3. Exit iws.
4. Invoke the same plugin operations in default mode.

Assertions:

- Open iws blocks generic plugin ops with typed `forbidden_in_isolated_workspace`.
- Default mode permits the same plugin operations.
- Fail-open behavior is limited to pipeline-not-bootstrapped cases and emits audit.
- No plugin op mutates the iws upperdir through an undocumented path.

### 3.5.5 `test_plugin_setup_network_failure_is_actionable.py`

Scenario:

1. Force plugin setup to use a network-denied sandbox path.
2. Attempt plugin ensure.
3. Restore normal egress and retry.

Assertions:

- Failure is surfaced as setup/network failure, not a generic plugin dispatch error.
- Error payload includes plugin name, setup step, command, and stderr excerpt.
- Retry after egress restoration succeeds without stale install state.

### 3.5.6 `test_plugin_service_survives_peer_publish_and_evict.py`

Scenario:

1. Start Pyright service through a READ_ONLY LSP call.
2. Publish several default-mode edits from another agent.
3. Verify LSP refreshes against new layer-stack tip.
4. Evict the service and call LSP again.

Assertions:

- Service refresh uses remount path while alive.
- Eviction unsubscribes from workspace-change events.
- New service starts cleanly after eviction.
- No duplicate event pump task remains.
- Warm service p95 latency remains within the read-only plugin budget after peer publishes.
