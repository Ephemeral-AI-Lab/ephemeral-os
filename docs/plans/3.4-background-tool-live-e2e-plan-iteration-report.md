## 2026-05-25 08:37 +08:00 Follow-Up Root-Cause Finding

Finding:

- The `~2.7s` shell case is the original 16-way `many_small_writes` scenario, not the `1/2/5` diagnostic.
- Evidence from `.sweevo_runs/scenario_logs/sandbox.background_many_small_writes_do_not_starve_dispatcher/20260524T234934Z_4421c4a22556/performance_report.json`:
  - `16` concurrent background shell samples.

  - `occ.apply.commit_queue_wait_s` p95 approximately `43 ms`.

Interpretation: this is primarily the current execution algorithm/shape under high fanout, paid through OS process and namespace scheduling costs. Each tiny shell call creates a fresh `unshare -Urm` process, starts a Python namespace entrypoint, imports/dispatches the sandbox tool payload, then starts the actual shell command. At 16-way fanout, those fresh namespace/Python/shell launches dominate. The OS is where the time is spent, but the repo-side algorithm asks the OS to do expensive per-call setup concurrently.

What the evidence does not support:

- Not sleep: the slow conclusion holds after subtracting the explicit `sleep 0.2`.
- Not mount as the primary cause: mount p95 is `~168ms`, far below the `~2616ms` sleep-excluded shell p95.
- Not OCC serialization as the primary cause: OCC commit queue p95 is `~43ms`.
- Not background metadata by itself: actual background-wrapper probes at concurrency `1`, `2`, and `5` stay below `1s` sleep-excluded p95.

Design constraint: external replacement runtimes such as Rust-based helpers have been considered, but they require installation/build/runtime dependency management that is not acceptable for this workstream. Do not treat "rewrite the hot path in Rust" as the next step for this plan.

Next measurement step, without adding a new runtime dependency: split `command_exec.run_command_s` into stable subphase timings inside the existing Python path, at least:

- namespace process spawn/wait;
- Python namespace entrypoint bootstrap/import;
- tool payload dispatch inside the entrypoint;
- actual shell subprocess runtime;
- result write/capture.

This will identify whether the largest remaining cost is Python entrypoint startup/import or kernel/process scheduling under 16-way fresh namespace fanout.
