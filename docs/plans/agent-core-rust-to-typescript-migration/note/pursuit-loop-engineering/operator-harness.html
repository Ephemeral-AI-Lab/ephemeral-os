§07

The operator — the outer loop

The main agent supervises at the boundary; it never drives an inner iteration. Once launched, the pursuit loops autonomously — durable, DB-guarded, background. The operator's touchpoints are exactly an operator's:

01 · Launch
delegate_pursuit(pursuit_goal) — or createPursuit(...) for non-agent callers. One open pursuit per supervisor.
02 · Await
settle() resolves at terminal Success / Failed / Cancelled. No polling of internals.
03 · Judge
Read outcome.md — the ordered leg ledger — against intent, not against the goal string.
04 · Decide
Met → done. Not met → re-launch with an amended pursuit_goal. Runaway → cancel_background_session.

L-1 is the while-loop; L0 is the agent inside it. The main agent never self-loops: L-1 — the harness — re-invokes it each turn, executes its tool calls, and gates exit on submit_main_outcome. Step 3 shows the gate working: a submission that fails the condition returns false, L-1 re-enters, the failed pass is cleared and a fresh iteration begins — nothing is lost, because everything the failure taught lives in pursuit_1’s persisted context store, which iteration 2 reads before re-delegating. The retry-until-met behavior is semantics the agent enacts inside its turns; the control flow belongs entirely to L-1. At this altitude delegate_pursuit is just another tool call beside bash and edit, and the inner loops (FIG.2–7) run entirely behind it.

This makes the operator the system's actual verification layer: attempt success ultimately rests on worker- and verifier-claimed settlements, so the independent judgment that closes the loop objectively lives with whoever holds the intent. Pursuit supplies that judge the two hard ingredients — a structured, inspectable outcome and a clean terminal handle — and deliberately supplies no outer-loop machinery.

Continuity: the context store

A re-launched pursuit starts as a fresh mission, but the system is not amnesiac: every pursuit's full context tree — pursuit → leg → attempt → plan → work items, one fact per .md file — persists on disk under .eos-agents/pursuit/context/pursuit_<id>/ and outlives settlement. The operator folds its judgment into the next pursuit_goal, and context-exploration tools (planned) will let a successor pursuit's agents read predecessor trees directly rather than relying on goal prose alone. Within a leg, memory is already first-class: a retry planner shares context across attempts by declaring depends_on against work items of prior attempts that remain is_consistent_with_leg_goal.

§09

Inside L-1 — the native loop · Phase 04.10

FIG.8 drew the while-loop conceptually. This is the real one — agent-loop.ts, eleven steps — with the landed Phase 04.10 branch: a profile that omits terminal_tool terminates on a bare-text assistant turn, under exactly the gate the terminal submission tools already answer to. The “no terminal tool” rung is not an absence of mechanism; it is the submission guard, relocated into the loop.

Watch step 6 across the two passes: in turn A it is skipped (the assistant called tools), and the loop runs the full circuit — observer, batch, terminal check, loop. In turn B the four gate checks light in sequence and the run finishes before step 7: the finishing text turn announces no turnCompleted (a publish into a finished run informs nobody) and steps 7–11 never execute. The exit commits synchronously — no await between decision and finish() — so an abort that loses the race to a completed final stream yields completed, in parity with “the submission outranks late redirection.”

Two reads on one supervisor, deliberately different: backgroundSessionCount() (running only) gates the park — a settled-but-undelivered session must not park, its notification is already published. openBackgroundSessionCount() (running + undelivered) gates the exit — finishing with an undelivered settlement would drop a notification the model was owed.

Decisions worth knowing

  • Absence is the switch. terminal_tool present → today’s regime byte-for-byte; absent → text mode. No new flag, no kind-keyed default — and the engine stays profile-ignorant: terminationMode is threaded in, defaulting to terminal_tool.
  • The strictness gradient is enforced at load time. planner and worker profiles without a terminal tool fail loading by name — pursuit transitions consume schema-validated submissions, and a free-text planner outcome would synthesize failures. main, advisor, and subagent may omit.
  • The notification layer branches on terminal_tool: null. The reminder rule goes silent (text runs exit instead of spinning — there is nothing to nudge), and the budget rule swaps its tail to “finish by replying with your final answer as plain text.”
  • submission is the text projection. The final text rides the same outcome path parents already read: run_subagent passes a string child’s text verbatim into the settlement summary, and ask_advisor works unchanged. The first intended user is the research subagent that should not need a submit_subagent_outcome round-trip to say what it found.
