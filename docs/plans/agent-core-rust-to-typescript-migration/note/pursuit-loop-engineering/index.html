§00

Why loops, not prompts

A single model call cannot carry a multi-step objective: it cannot observe its own results, recover from failure, or hand work forward. Loop engineering is the discipline of wrapping agents in iteration cycles with objective verifiers, narrow feedback, explicit budgets, and verifier-gated exits.

Pursuit is EphemeralOS's loop runtime: a durable, caller-agnostic orchestrator that hard-codes two well-engineered loops — retry inside a leg and relay across legs — and exposes the interfaces an outer operator loop needs. Loop control is fixed and race-hardened in the service; loop policy (what to verify, when to refocus, what to defer) belongs to planners, prompts, and user-authored context scripts.

Sophisticated control, simple policy — never the other way around.
§01

The loop stack

Three loops, three owners, three verifiers. Each loop lives at the level that owns the goal representation its exit condition is judged against.

FIG.1 — Nested loop orbitstokens orbit at loop cadence

Reading the figure: inner loops spin faster. The retry token completes many circuits per relay circuit; the operator token moves slowest of all — it only acts at pursuit settlement. The work-item DAG at center is not a loop: it executes once per attempt, hard-gated by depends_on. The labels are doors: each orbit label links to its chapter.

Each loop lives where its goal representation lives — that is the only place its exit can be judged.

From inside, a pursuit is a one-shot mission: nothing in the service ever re-launches a pursuit. The outer iteration exists only in the operator's context — deliberately, because a pursuit can succeed at its stated goal while missing what the operator actually wanted, and only the intent-holder can detect that. One layer further out sits L-1, the harness while-loop itself — numbered negative because it is pure mechanism: it owns no goal and makes no judgment, it simply re-invokes the agent until submit_main_outcome satisfies the loop condition (FIG.8). Every layer above it judges; L-1 only gates.

§02

Anatomy & vocabulary

Five entities. Goal text is immutable at the top, effective-with-provenance in the middle, and disposable at the bottom. The vocabulary is deliberate: legs advance, attempts retry in place — the old workflow/iteration words obscured exactly that distinction.

// the durable entity stack — every transition idempotent, DB-guarded
Pursuit      goal: pursuit_goal (immutable — the caller's ask, the only content)

Width, lanes, budgets, memory — where each one lives

  • Exploration is horizontal, inside the attempt. A planner can fan out many exploration work items and gate a collection/synthesis item on them with depends_on — "run N approaches, synthesize the winner" is one plan. Workers carry their own subagents for depth. Parallel exploration lives in the work graph, not in extra leg lanes; planners are expected to be powerful and plan wide.
  • One baton per leg — intentional lane discipline. Attempt-level fan-out is already dynamic multi-baton, so legs stay single-lane: the pursuit never crosses too many lanes at once. How many legs exist is the agent's decision — in dynamic mode the newest leg always carries the goal that would close the pursuit, and its planner decides whether that is too much for one leg and defers the remainder.
  • Bounded horizontally, unbounded vertically. max_attempts (currently 6) bounds retries within a leg — deliberately lenient: a leg that cannot close in six attempts is signaling an unworkable task, not an underfunded one. The leg chain itself has no cap.
  • Memory is the context store. The pursuit → leg → attempt → plan → work-item tree persists to disk as per-field .md files; cross-attempt sharing rides depends_on against is_consistent_with_leg_goal attempts; context-exploration tools are planned for read access across the store (§07).
Fixed loop semantics are what make the invariants provable and the test matrix finite.
