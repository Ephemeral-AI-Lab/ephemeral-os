§03

The relay — legs & the baton

A pursuit advances in legs. The baton is next_leg_goal: a goal string declared by the planner, passed forward only when the current leg closes Success. No successor → the pursuit itself closes Success.

The axes: attempts extend horizontally inside a leg — the retry axis, bounded at max_attempts = 6 (deliberately lenient: a leg that cannot close in six attempts is signaling an unworkable task). Legs extend vertically — the relay axis, unbounded. The spawn rule: leg_n+1 is created only at the instant leg_n closes Success with a standing next_leg_goal; before that moment no leg_n+1 row, directory, or goal exists anywhere. In dynamic mode the newest leg always carries the goal that would close the pursuit — each planner decides whether that is too much for its own leg and defers the remainder downward.

Why "relay" is the right metaphor (both modes): each leg gets fresh runners (new agent contexts — no context-window exhaustion across a long pursuit), the baton is small (a prose goal, not accumulated history), and the handoff is gated (success means the full effective leg goal, never "goal minus deferral"). The modes differ only in who owns the route: dynamic legs are discovered by planners; predefined legs are caller checkpoints known upfront — which is why all three exist from creation in FIG.3.

The success invariant

Deferral happens at planning time, never at close time. A planner that cannot achieve the full leg_goal must narrow it (refocus) and peel the remainder into next_leg_goal — before the work runs. By the time the closing attempt succeeds, success is unambiguous:

Success means the full effective leg_goal was achieved.
Success never means "leg_goal minus next_leg_goal".

Contract subtlety: in dynamic mode there is no payload shape for clearing a standing next_leg_goal alone. Clearing successor scope is a refocus act — declare the replacement leg_goal and omit next_leg_goal in the same payload.

§04

Adaptive retry — attempts inside a leg

The leg is the loop frame: it owns the goal, the budget, and the exits. Attempts are its iterations. A retry is not a re-run — the retry planner receives structured failure_reasons, sees only focus-consistent attempts, and may depends_on the previous attempt's still-successful work.

FIG.4 — One leg, three attemptsloop · 16s
attempts 1/32/33/3· max_attempts:3
attempt_1 planning running failed
planner declares work graph · workers execute DAG
failure_reasons: [ work_item_tests [Failed]: assertion drift in fixtures ]
attempt_2 replanning running failed
retry planner reads failure_reasons — re-plans around the break
depends_on: attempt_1.work_item_schema [Success] · surviving work is reused, not redone
attempt_3 replanning running success
all work items terminal Success — verifier items included
leg closes success → baton check (next_leg_goal?)

The adaptive part: cross-attempt depends_on (same leg, same non-superseded leg_goal_version) means a retry can be "fix one item and re-verify" instead of "redo everything". That is the difference between retry and adaptive retry.

Replan has two grades — and the second one is priced

Re-planning work under the same goal is free; re-planning the goal itself (refocus) supersedes prior attempts and increments leg_goal_version — but the attempt budget does not reset. A pivoting planner cannot disguise flailing as progress; the honest path for genuinely new scope is next_leg_goal.

Derived, not mutated: "superseded" is a pure view — an attempt is superseded iff a later plan in its leg declared leg_goal. Nothing archives at mutation time, so cancel races and idempotent replays cannot corrupt the archive.

§05

The scheduler — inside one attempt

Work items form a DAG over depends_on. One invariant is physics, not policy: a work item launches only when every direct dependency is terminal Success — enforced in the claim query, the post-commit launch guard, and the stale-claim recheck.

FIG.6 — Failure propagationloop · 16s
reconcileAttemptStatus: no Running, no NotStarted → close attempt → FAILED failure_reasons: work_item_D [Blocked]: blocked by work_item_C failure_reasons: work_item_C [Failed]: <worker summary first line>

Watch item B: C's failure does not touch it. Unrelated running work continues; unrelated pending work still launches when its own dependencies succeed. The attempt closes Failed only after block propagation leaves nothing Running or NotStarted — so retries begin at true exhaustion, never early.

Scheduler invariants (physics)

  • Hard launch gate. Claimable ⇔ NotStarted ∧ attempt Running ∧ every direct dependency Success. No optimistic starts.
  • Running never becomes Blocked. If a dependency fails, the dependent was provably never launched. A Running item with a non-Success dependency is state corruption to surface, not repair.
  • Propagation runs to fixpoint. A fails → B Blocked → C (depends on B) Blocked. Transitive chains close in one settlement pass.
  • Failure is a list. failure_reasons[] records every failed and blocked item — the retry planner sees the whole break surface, not the first symptom.
§06

Verification is planned, not hardcoded

The framework does not impose a verification gate. The planner owns it: assign verifier work items that depends_on the work they check. The verify-fix loop then rides the existing retry machinery for free — bounded by the same budget.

The one non-negotiable

A verifier's verdict rides its STATUS, not its prose.
// Findings written into a Success outcome are a verification PASS —
// the attempt closes Success and the defects flow forward as achievement.
If verification fails → submit a Failed outcome. Always.

Where this policy lives: the planner directive in .eos-agents/pursuit/scripts/planner.cjs ("end the DAG with a validator") and the verifier profile's worker directive. Scripts are user-authored per deployment — each repo sets its own verification floor without touching @eos/pursuit. The floor is worker self-assessment; the ceiling is whatever verification the planner designs.
