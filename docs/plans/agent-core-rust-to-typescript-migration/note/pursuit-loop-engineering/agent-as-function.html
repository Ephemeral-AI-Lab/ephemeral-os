Theory & lab.

An agent is a function — and the L-1 lab is its signature editor.
§10

Agent as a function

Everything this document has built compresses into one sentence: an agent is a function. The caller writes the signature — submit_outcome_fn, a customized schema; the annotation riding it is the advisor instruction; the model improvises the body; and the harness guarantees the call returns. The body is the only nondeterministic part — the boundary is typed in both directions.

The totality row is the one that makes the whole metaphor load-bearing: ordinary functions can hang or throw past you, but an eos agent always produces a value of its declared type — a structured submission, a synthesized failure on death, or its final text. That guarantee is what lets every outer layer be ordinary code: once agents are total typed functions, a pursuit is a library of typed recursion, planner and worker are interface implementations, and L-1 is a while loop over return values.

Once agents are functions, orchestration is just programming.
§11

Program the gate — the L-1 lab

If an agent is a function (§10), the lab below is its signature editor. Only planner and worker carry a SubmissionBinding — their payloads are transitions of the strict pursuit state machine, so their schemas are fixed and their submission mutates durable state transactionally. The main agent’s terminal tool has no entity row to mutate; it only gates L-1. That asymmetry is deliberate, and it frees the outer contract to loosen.

The automata framing

A custom outcome schema turns L-1 into a programmable automaton: the gate field’s values are the input alphabet Σ, the exit-mapped values are the acceptance set, and the δ-table is the transition function — evaluated by code, never by a model, so L-1 stays judgment-free no matter how rich the schema gets. The good-faith layer rides the tool itself: docstring says what the act is, schema says what shape, advisor instruction says what good looks like — bound to the tool surface, so a context script that mangles initial messages can never strip the guidance.

The knowability rule: every required field must be derivable from the agent’s context and tools. An unknowable field doesn’t go unfilled — it gets filled with plausible fiction that validates. Schema checks shape, never truth.

δ — transition function · L-1 owns this: pure lookup, no judgment

run · you play the agent — the harness plays L-1

configure the gate on the left, then ▶ run loop. When the turn reaches submission, you choose what the agent submits — and watch δ decide.

What each rung buys: in no-terminal-tool mode (Phase 04.10 text termination, §09) the run finishes cleanly through the relocated gate — but the verdict is prose, so mechanical retry has nothing to look up. In boolean mode the loop can re-enter mechanically (FIG.8). With a custom schema, your δ-table is a state-machine harness — re-enter, re-enter-with-reminder, exit-success, exit-failure — and building it touched nothing in the strict pursuit interior. The dividing line stands: pursuit is the managed machine (durable, budgeted, superseding); custom L-1 is the programmable harness. Keep the interior strict and small; let the edges stay soft.
