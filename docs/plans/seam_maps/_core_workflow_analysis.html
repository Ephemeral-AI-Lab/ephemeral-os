_core_workflow.js — Analysis

Multi-agent orchestration driver for the reducers + unified outcomes redesign of EphemeralOS task_center · 7 dependency-ordered steps · implement→verify · halt-on-failure

What it is

A Workflow orchestration script — a self-contained JS program in the deterministic multi-agent orchestrator format. It executes the "reducers + unified outcomes redesign" as 7 strictly-ordered steps, each running an implement→verify agent pair with halt-on-first-failure semantics.

The redesign's core vocabulary change: the Evaluator/Verifier roles collapse into a single Reducer role, the scattered per-task "summary" notion unifies into one Outcome type, and three separate state modules consolidate into one _core/state.py.

Key design properties

  • Coherence-bound / sequential, not parallel. Despite the harness supporting parallel()/pipeline(), this uses a plain for loop with await — because each step depends on the prior step's final symbol names.
  • Context relay. Each implementer sees prior steps' emergent_decisions + deviations + notes_for_next_step (lines 184–186). This is the only inter-step channel.
  • Two-tier gating. HARDruff_ok and import_ok halt on failure. ADVISORYtargeted_summary (pytest) is expected red mid-refactor and does not gate.
  • Adversarial verify split. opus implements; an independent sonnet verifier re-runs the same checks and is told "Do NOT edit any files" — guarding against an implementer falsely reporting green.
  • The deep-import gate (_deep_import.py, must end DEEP_IMPORT ok=N bad=0) is load-bearing: it imports every submodule because a plain import task_center is a lazy facade that hides broken lifecycle imports.

The shim lifecycle — the script's central correctness trick

The hardest constraint is "the package MUST remain importable after every step" while canonical fields are renamed/dropped. The script solves this with temporary re-export shims and TEMP DTOs introduced early and torn down at the end.

Notable observations & risks

Authoritative inputs are external. Every step reads _PLAN.md + _DECISIONS.md + per-cluster seam manifests. The script is a driver; the real spec lives in _DECISIONS §1–§4. The prose scope blocks are dense pointers, not standalone instructions.
A live conflict is encoded in Step 4 (line 120). §1.2/§1.3 keep Attempt.plan_spec but §1.7 drops the attempts.plan_spec column. The script pre-resolves it (keep the column, remove it from _DROPPED_COLUMNS) and tells the agent to record the reconciliation in deviations. The plan docs themselves are internally inconsistent here.
WF-B boundary. Steps repeatedly forbid touching "WF-B groups G1–G10" — those vocab/test files are deferred to a separate parallel pass. This core workflow owns only CORE-coupled tests + shared mock seams.
Coarse halt granularity. A single ruff error anywhere in backend/src (not just the step's files) halts the chain. Since parallel agents may dirty the worktree, an unrelated agent's lint error could false-halt this workflow.
No resume/budget logic. It's a fixed 7-step linear pipeline — no budget guard, no loop-until-dry. Resume relies on the harness's resumeFromRunId cache, but editing an early STEPS entry invalidates everything after it.
The verifier can't catch logic regressions — only ruff + import + advisory tests. Behavioral correctness rests on the implementer + the (red-tolerant) targeted tests, with full validation deferred to the later WF-B pass and the full suite.
