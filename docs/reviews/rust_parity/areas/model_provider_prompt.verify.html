New findings

  • NF1 (investigator_missed, MEDIUM) — provider-history sanitization absent in Rust. Python build_provider_messages (provider_history.py:20-175) deep-copies the durable history and drops orphaned tool_use/tool_result blocks (_drop_unmatched_tool_blocks_in_place) plus reduces stale background-task snapshots (reduce_background_task_history) before the provider request. Anthropic rejects a tool_use without a matching tool_result and vice versa, so this is a correctness invariant, not cosmetics. The Rust loop has zero equivalent: run_query (loop_.rs:119) hands the raw append-only messages to build_query_run_request, which copies them verbatim into LlmRequest (request.rs:25-30). Severity MEDIUM with caveat: impact depends on whether the Rust loop can yet produce orphaned pairs (background-task history reduction may be out of the current Rust phase scope), but the orphan-block sanitization gap is a real assembly-parity hole the investigation did not flag.
  • NF2 (fixture-fidelity, LOW) — golden seq is not faithful recorder output. parity/prompt_report/session_golden.jsonl seq 1,2,3 are hand-authored constants (capture.py:126-148), not produced by PromptReportRecorder semantics. The real recorder shares one seq across a turn's three rows (request.py:31), so a real turn-1 transcript is 1,1,1 (Python) / 0,0,0 (Rust), never 1,2,3. README §22 calls this the "faithful recorder output," which is inaccurate for the seq field. Recommend regenerating the golden through the actual recorder or annotating the seq as illustrative.
  • NF3 (confirmation, none) — OpenAI message encoder richer than Codex, correctly so. serialize_openai_message (openai.rs:256-303) handles role-based input_text/output_text, tool_use→function_call, tool_result→function_call_output, system_notification — vs Codex's text-only build_body (codex.py:244-255). Since OpenaiClient has no Python source this is a forward-looking generic encoder, not a divergence; needs real-Responses-API validation.

Overall verdict

The investigation correctly identified D1 (HIGH), D5, D6, D7, D8, D9 and the matched constants — its source reading is accurate. Its primary defect is a false-alarm cluster (D2/D3/D4): it audited a net-new generic OpenAI Responses client (OpenaiClient, api.openai.com) as if it were the Codex client (codex.py, chatgpt.com), producing three "confirmed disparities" that have no Python ground truth. Decisive anchor: app_state.rs:504 vs codex.py:56. It also overstated D7 severity (golden seq is hand-authored, not faithful) and missed NF1 (provider-history orphan-block sanitization has no Rust port, a real context-assembly gap).

No FALSE MATCH found that breaks a confirmed-match invariant: the Anthropic encode/decode, error/retry/backoff constants, tool_result/system_notification wire shapes, and MessageRole system-rejection are all genuinely faithful (re-derived bilaterally, with tests). The most actionable genuine gaps are D1 (HIGH — guardrail text lost) and NF1 (MEDIUM — orphaned tool-block sanitization absent).
