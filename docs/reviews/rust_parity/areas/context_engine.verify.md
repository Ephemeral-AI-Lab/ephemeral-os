## Disparity adjudication

- **D1 — terminal-tool-selection block format diverges (CONFIRMED, MEDIUM).** Re-derived against the *running* Python code, not the docstring. PY `render_terminal_catalog` (registry.py:113-121) emits each row as `` - `{name}` — {text} `` (backtick name, em-dash ` — ` separator) joined with `"\n\n"`, and the wrapper adds NO header (skill_message.py:30-33). RS `terminal_selection_block` (composer.rs:172-184) emits `- {name}: {guidance}` (no backtick, colon separator), joins with `"\n"`, and prepends `Pick exactly one based on outcome:`. **Decisive:** `grep "Pick exactly one" backend/src` returns exactly ONE hit — the *stale docstring example* at `skill_message.py:69`, never any executing code. The Rust block was evidently ported from that stale docstring. PY byte-contract test `test_initial_messages_capture.py:203-216` asserts `render_terminal_catalog(..., focus="selection_guidance")` is a substring of the row-3/row-4 `<terminal_tool_selection>` body — the Rust block would FAIL that substring check. **This resolves investigation Open-Question 2: it is unintended drift, not an intended new wire shape; Python's running `render_terminal_catalog` is the target.** Per-terminal `selection_guidance` *text* does match byte-for-byte (registry.py:39-94 vs terminal.rs:90-106) — only the bullet/header framing diverges. Severity MEDIUM upheld: live prompt content on every workflow agent launch.
- **D2 — unknown/unparseable terminal: Python emits fallback bullet, Rust silently drops (CONFIRMED, LOW).** PY `render_terminal_catalog` appends `` - `{terminal}` — (no descriptor registered for this terminal) `` on unknown name (registry.py:116-117). RS `continue`s on `parse::<ToolName>()` / `from_tool_name` miss (composer.rs:165-169). **"Dead path?" question (advisor gap #2) resolved:** RS `AgentDefinition.terminals` is `Vec<String>` validated only to be non-empty-after-strip (model.rs:222-228) — there is NO load-time check that each name resolves to a `ToolName`/`TerminalTool` (test fixtures even use `terminals: [submit_x]`/`[t]`). So an unresolved terminal is structurally reachable; the divergence is *live*, not dead. LOW retained because shipped profiles use known terminal names.
- **D3 — `parse_outcomes_record` strict in Rust, lenient in Python (CONFIRMED, LOW).** PY (outcomes.py:156-177): non-`list` → `()`, non-`dict` records skipped, `_normalize_status` coerces unknown → `"failed"` — never raises; planner build skips a malformed iteration's outcomes and proceeds. RS `parse_outcomes_record` (engine.rs:294-298) does `serde_json::from_str::<Vec<ExecutionTaskOutcome>>` and propagates via `?` (engine.rs:189), hard-failing the whole planner build on malformed/legacy `iteration.outcomes`. Both short-circuit empty input (PY `if not value: return ()` / RS `raw.trim().is_empty()`). Behavior change is real but only fires on corrupt/legacy non-array JSON; in the typed/migrated column it is unreachable.
- **D4 — cross-boundary missing-status normalization (CONFIRMED-as-DEFERRED, MEDIUM, not a context-engine bug).** In the normal persisted case the record already carries `status`+`role`, so PY `setdefault` is a no-op and output equals RS reading pre-normalized `task.outcomes` (engine.rs:254). The eos-state docstring (outcomes.rs:53-58) and `present_status` (outcomes.rs:60-66) correctly encode PY `present_status` (done→success) as distinct from the eos-db `_normalize_status` (done→failed). The single discriminating question — *does the eos-db missing-status fill on a `done` row use `present_status`(→success) or `_normalize_status`(→failed)?* — cannot be settled from `context/`; it lives at the eos-db parse boundary. Correctly deferred to the eos-db/eos-state audit. **unproven within this area.**

## New findings

- **N1 (positive, advisor gap #1 closed).** The investigation's E1 ("`format!("{:?}").to_lowercase()` is correct only because variants are single-token") is the one inherited claim; I opened both `AttemptStatus` enums and confirmed it. PY `StrEnum` values are `running`/`passed`/`failed` (state.py:127-129); RS variants `Running`/`Passed`/`Failed` (attempt.rs:31-35). Lowercased Debug == PY `.value` for all three. No multi-word variant exists, so **invariant 4 has no FALSE MATCH** on the `<attempt status>` attribute. E1 stays correctly latent (future multi-word variant would diverge; none today). `ExecutionRole`/`TaskOutcomeStatus` likewise single-token (outcomes.rs:19-37), so `render_task_outcome` role/status attrs (xml.rs:18-25) are also safe today.
- **N2 (doc rot, root cause of D1).** The `build_skill_message` docstring (skill_message.py:67-73) shows an *illustrative* example output containing both `Pick exactly one based on outcome:` and the colon row form `- <tool_name>: <selection_guidance>` — neither of which the real `render_terminal_catalog` produces. This stale docstring is almost certainly what the Rust port was written against. Worth a one-line note to fix the Python docstring so it can't mislead future ports; does not affect runtime.
- **N3 (positive).** AC#15 byte-equality (row-3 `<Task Guidance>` block == row-4 skill block) is preserved *internally* in Rust: both `wrap_task_guidance` (composer.rs:119) and `build_skill_message` (composer.rs:145-147) call the same `terminal_selection_block`. So Rust is self-consistent; only the cross-language shape (D1) differs.
- **N4 (positive).** `prior_iteration_sections` short-circuit on empty `iteration.outcomes` matches: RS `iteration.outcomes.as_deref()` → `None`/empty `continue` (engine.rs:186-191) mirrors PY `if not outcomes: continue` (engine.py:212). Recompute-from-task-rows path is faithfully threaded (RS `attempt_execution_outcomes(&attempt, Some(task_store))` engine.rs:219-220 → eos-state outcomes.rs:126-134, persisted-win-else-project, == PY outcomes.py:93-99).
