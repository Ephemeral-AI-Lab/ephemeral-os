## Disparity adjudication

- **D1 (host `call_plugin` + install.py unported, high) — CONFIRMED.** Re-derived independently: the only non-test `api.plugin.ensure` reference is the daemon `register_builtin` (dispatcher.rs:129); all other hits are tests (mod.rs:1906+). No `.installed-` marker, no `EOS_PLUGIN_TRUSTED_SETUP_ROOTS`, no `setup.sh` execution, no Node/pyright provisioning anywhere outside docstrings. Decisive new corroboration the investigator did not cite: `eos-sandbox-host/src/lifecycle.rs:96` explicitly states *"The Python `forget_plugin_dispatch_state`/`forget_plugin_install_state` calls are dropped ... the Rust host ports no plugin internals (GC-03)."* `plugin.install` survives only as a wire op-name in the empty-response fail-closed policy list (daemon_client.rs:1220, inside test `empty_response_gating_matches_python_set`) — there is no implementation behind it in the audited tree. The setup-trust gate (a real sandbox-escape mitigation) and marker idempotency are genuinely absent. Severity high stands.

- **D2 (daemon ensure manifest-driven, not importlib, medium) — CONFIRMED, intentional.** `ParsedEnsure::from_args`/`from_manifest` (mod.rs:357-413) reads a manifest from `args`, validates, builds routes; no import, `runtime_warmed: false` hardcoded (mod.rs:193), no per-plugin async lock (serialized by global `state_cell()` Mutex, mod.rs:109-118). Python BL-01 warm-then-record + OP_TABLE rollback (runtime_api.py:130-146) have no analogue; a service that fails to start surfaces at dispatch time (`ensure_tracked_service_process_running`, mod.rs:1538-1560), not ensure time. Coherent redesign; documented in lib.rs.

- **D3 (`_validate_plugin_caller` frame-walk not ported, low / N/A) — CONFIRMED, structurally inapplicable.** `OpRegistry::register` (registry.rs:120-131) has no caller check; Python op_registry.py:263-284 walked 16 frames. In the manifest-over-RPC model there is no in-daemon `import`+decorate, so the frame-walk cannot apply. Manifest sender is trusted to scope op names. N/A stands.

- **D4 (OCC callback accepts `snapshot_version`+`base_hashes` the Python harness never sends, low) — CONFIRMED, additive.** `ApplyChangesetRequest` (occ_callbacks.rs:99-114) reads both as `#[serde(default)]` optional; Python body is `{layer_stack_root, changes[, parent_message_id]}` only (ppc_service.py:312-317). Forward-compatible widening; daemon test sends `snapshot_version: null` → head-publish, matching Python. Not a bug.

- **D5 (refresh redesign, no materialized Python counterpart, divergent/unverifiable, medium) — CONFIRMED as unverifiable.** Adjusted: the investigation labeled invariant 6 "divergent"; the correct verifier verdict is **unproven** because the authoritative Python refresh code is absent (see invariant 6 row). The `manifest_key` format divergence is real and re-confirmed below.

- **D6 (manifest identifier validator diverges from `_PLUGIN_NAME_RE`, medium) — CONFIRMED + sharpened.** Three Rust validators, and the disparity is **manifest-path-specific**:
  - Faithful one (`registry.rs:88 is_valid_plugin_name`, `^[A-Za-z_][A-Za-z0-9_]*$`) is NOT called by `PluginManifest::validate`.
  - Faithful one on the **no-manifest** ensure path: `validate_public_identifier` (mod.rs:587-609) exactly matches `_PLUGIN_NAME_RE`. (Investigator omitted this; it is the path used when `args` carries `plugin`/`digest` instead of a `manifest`.)
  - The **manifest path** uses `service::validate_identifier` (service.rs:127-148) for `plugin_id` AND `op_name`: permits `-`, `.`, `_` after first char → **accepts `my-plugin`/`my.plugin` that Python rejects**; and **requires identifier shape on `op_name`** → **rejects op_names Python accepts** (Python op_name only requires non-empty, op_registry.py:109). Host catalog `names.rs:117` is `^[a-z][a-z0-9_]*$` — stricter than Python (rejects uppercase + leading `_`). So the manifest boundary silently accepts/rejects a different name set than ground truth. Confirmed bug; severity medium stands.

- **D7 (8 MiB response cap + caller-field NUL/256-char caps dropped, low) — CONFIRMED both halves.** `response_payload_from_reply` (mod.rs:1516-1527) has no size cap; the only cap is the 16 MiB frame limit (`MAX_PPC_FRAME_BYTES = MAX_REQUEST_BYTES = 16 MiB`, ppc_router.rs:27 + version.rs:29 `16_777_216`). Python `_MAX_RESPONSE_BYTES = 8 MiB` (host_dispatch.py:60, 394-400). No NUL/256-char caller-field validation exists anywhere in `eos-daemon/src` or `eos-audit/src` (the only `256` hits are `OCC_SERVICE_CACHE_MAX` dispatcher.rs:1957 and sha256; the only `\0` is a hash separator). Python `_audit_field` (runtime_api.py:352-360) enforced NUL-reject + 256-char cap. Both dropped; severity low stands (16 MiB frame bounds worst case).

## New findings

- **NF-1 (HIGH-adjacent — primary failure-mode catch): invariant-5's auto-overlay WRITE trigger has no producer; the Python automatic intent-driven trigger is lost.** The investigation filed this as an "info" extra finding ("OneshotOverlay is the only WRITE projection path"). It is more serious as a *dynamic* gap and belongs tied to D1:
  - Python wired **any** `WRITE_ALLOWED + auto_workspace_overlay=True` op through the overlay+OCC path **automatically by intent at flush time** (op_registry.py:230-235) — no service declaration needed; any plugin author got the write path for free.
  - The Rust generic write path (`dispatch_oneshot_overlay_route`, mod.rs:1408-1475) executes **only** when `route.service_mode == Some(OneshotOverlay)` (mod.rs:1413); otherwise it returns `Ok(None)` → `dispatch_deferred_route` (mod.rs:1093) → `{"success": false, "status": "deferred"}` (a silent no-op for the write).
  - The Python host manifest builder `_daemon_manifest_for` (host_dispatch.py:287-316) **never emits `oneshot_overlay` and never assigns a `service_id` to a WRITE_ALLOWED+auto_overlay op** (it only emits `workspace_snapshot_refresh`). The only `oneshot_overlay` producer in the entire Rust tree is a **test** (`oneshot_overlay_manifest`, mod.rs:1799). There is no Rust host manifest builder at all (D1).
  - Net: a generic auto-overlay write op arrives `WriteAllowed, auto=true, service_id=None` → guaranteed deferred no-op. The mechanism is faithfully ported + tested; its **trigger is unwired**. In the materialized Python, the only WRITE_ALLOWED op is a benchmark fixture (`sandbox/api/plugin_support.py:107`, "helpers for runner and benchmark harnesses") and no `plugins/catalog` dir was materialized, so no *production* plugin exercises this on either side today — hence "dormant" rather than a live regression. But the parity loss is real: Python's contract was "intent selects the write path automatically"; Rust's is "an explicit oneshot_overlay service must be wired, by a host builder that does not exist." Recommend: when the host plugin facade lands (D1 fix), ensure WRITE_ALLOWED+auto_overlay ops are emitted as `oneshot_overlay` services, or add a daemon-side default-overlay fallback so the deferred branch cannot silently swallow a write.

- **NF-2 (low, confirms extra finding): `manifest_key` format divergence is real and within-Rust-only.** RUST `manifest_key(version, root_hash) = "{version}:{root_hash}"` (mod.rs:812-814); PY `build_manifest_key(root_hash, version) = "{root_hash}@{version}"` (projection.py:21-23). Re-confirmed harmless: the key is produced by `active_manifest_key` (mod.rs:804-810) and consumed by `RefreshAck::require_manifest` (refresh.rs:77-92) entirely inside the Rust daemon+harness; it never crosses to a Python peer. Matches the investigator's "extra finding"; recorded so no one "fixes" one side into incompatibility.

- **NF-3 (info, confirms extra finding): OCC callback adds `OpaqueDir` change kind** (occ_callbacks.rs:133-135, 156-158) that Python `_change_for_path` never emits (write/delete/symlink only, ppc_service.py:401-421). Additive, not a regression.

## Overall verdict

The investigation is accurate and well-anchored; every disparity (D1–D7) is CONFIRMED, with two refinements: D5 is properly **unproven** (not bilaterally verified — Python refresh code absent), and D6 is **sharpened** (the no-manifest ensure path `validate_public_identifier` IS faithful; only the manifest path diverges). The investigator's primary D1 finding (host orchestration + setup-trust gate unported) is the highest-value catch and is corroborated by the explicit `lifecycle.rs:96` GC-03 comment the investigation did not cite.

The one place the investigation under-weighted a dynamic was invariant 5: it called the mechanism a "match" (correct) and relegated the trigger gap to "info." On independent re-derivation the trigger gap (NF-1) is a genuine parity loss — Python auto-selected the overlay write path by intent; Rust requires an `oneshot_overlay` service no producer wires — so I elevate it to a HIGH-adjacent new finding tied to D1, while affirming the *mechanism* as confirmed_match. This is the closest thing to an investigator_missed in this area: not a false "match" on the mechanism, but a false-comfort "info" on the trigger.

No false alarms found (every flagged disparity is real). No FALSE MATCH on the OCC/PPC/projection mechanisms — those are solidly ported and, in the PPC case, more robust than the materialized Python.
