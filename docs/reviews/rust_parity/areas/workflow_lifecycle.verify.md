Divergence is the lost cycle/integrity defense + silent `false` on missing rows. Investigator severity (low) upheld.

- **D2 (close outcomes pass-through vs re-normalize) вЂ” CONFIRMED, low.** PY `close_workflow` does `records_json(workflow_outcomes(...))` (`lifecycle.py:157-159`) where `workflow_outcomes` parses `latest.outcomes` via `parse_outcomes_record` (`outcomes.py:130-138`) and `records_json` re-serializes canonical `{status,role,task_id,outcome}`. RUST `workflow_outcomes_json` (`lifecycle.rs:227-239`) clones `latest.outcomes` verbatim. Idempotent under current writers (iteration `outcomes` are themselves produced canonically by `project_iteration_outcomes`в†’`records_json` at iteration close, PY `attempt_coordinator.py:213-217` / RUST `iteration/mod.rs:298-308`). - **D3 (already-outstanding `is_error`) вЂ” CONFIRMED, medium.** PY `delegate_workflow.py:67-81` `is_error=True`; RUST `model_tools/workflow.rs:67-77` `ToolResult::ok`. - **D4 (dropped metadata keys) вЂ” CONFIRMED, low.** PY metadata has `submission_kind, workflow_task_id, workflow_id, task_id, attempt_id, initial_iteration_id, initial_attempt_id` (`delegate_workflow.py:115-123`). Harmless unless a consumer reads them; low upheld.

- **D5 (continuation start-failure has NO compensation) вЂ” CONFIRMED, high. This is the headline catch and it holds on both legs.**
  - *Primary leak (confirmed):* RUST `handle_iteration_closed` deferred branch (`lifecycle.rs:164-172`) does `create_iteration_with_coordinator(...).await?` then `coordinator.create_and_start_first_attempt().await.map(|_| ())` with no rollback. On attempt-start failure, `start_attempt` (`iteration/mod.rs:123-144`) calls `close_attempt_after_startup_failure` which closes only the **attempt** (FAILED/StartupFailed, `:277-295`); the **new iteration stays OPEN, its coordinator stays REGISTERED, the workflow stays OPEN**, and `Err` propagates. PY routes through `_start_deferred_iteration` (`lifecycle.py:207-231`) which on failure sets new iteration CANCELLED, deregisters its coordinator, and `close_workflow(succeeded=False)`.
  - *Decisive proof it is a gap, not a simplification:* Rust DOES compensate the **initial** path (`starter.rs:116-161` + test `compensation_rolls_back`) but provides no equivalent on the **continuation** path. Python compensates both.
  - *Secondary leak (confirmed, lower probability):* PY deregisters the **old** iteration in a `finally` (`lifecycle.py:146-147`) so it always runs. RUST runs `self.iteration_coordinators.deregister(&iteration.id)` only after `result` is bound (`lifecycle.rs:183`); the `?` on `create_iteration_with_coordinator` (`:166-168`) early-returns before that line, so a create-iteration failure also leaks the **old** coordinator. Lower probability (predecessor is SUCCEEDED+deferred, so create rarely fails) вЂ” keep as secondary.
  - Effect: a single planner-launch failure on a deferred continuation strands the workflow permanently OPEN (`check_workflow_status` reports "running" forever) and leaks process-local coordinator state with nothing surfacing the error to the parent. High severity upheld. Suggested fix: mirror Python вЂ” wrap the deferred branch so the old-iteration deregister is unconditional, and on `create_and_start_first_attempt` failure cancel the new iteration, deregister its coordinator, and `close_workflow(..., false)`.

- **Extra finding: cancel parity + planner/root role outcome divergence вЂ” CONFIRMED (NOT a false alarm), low.** Re-derived from `_runtime.py`, which the investigation correctly anchored.
