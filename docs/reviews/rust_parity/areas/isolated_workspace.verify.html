Disparity adjudication

  • D1 (medium/missing) Daemon exit drain / per-agent dispatch quiesce (exit_pending/begin_exit_drain/acquire_dispatch_slot) absent in Rust → REFUTED (overstated). Those symbols belong to the DELETED Python _control_plane (workspace_handle_lifecycle.py:216-253, dispatcher.py:94 acquire_dispatch_slot), an in-process daemon-thread quiesce model. The LIVE in-tree Python host lifecycle (isolated_workspace_lifecycle.py) has NO begin_exit_drain/exit_pending either — its exit does _cancel_by_agent then daemon RPC, which Rust mirrors exactly (eos-sandbox-host/isolated_workspace.rs:84-87). The daemon-side quiesce is re-expressed in Rust as the active_command_sessions gate + force_cancel drain loop (isolated.rs:407-434, busy-wait to grace_s deadline). This is the porting model's deliberate re-architecture, not a missing dynamic. Severity → none.
  • D2 (medium/divergent) exit drains unconditionally / RequireNoInflightBackgroundTasks prehook not wired on isolated tools → REFUTED (false alarm — primary catch). meta.rs:65-67 wires Hook::RequireNoInflightBackgroundTasks on BOTH EnterIsolatedWorkspace AND ExitIsolatedWorkspace, exactly matching in-tree enter/exit_isolated_workspace/definition.py:34. The hook denies on local>0 (hooks.rs:512-521) and on daemon-count>0 / unavailable (hooks.rs:546-557). So exit does NOT drain unconditionally — it is refused at the tool layer when background work is in flight, then (if it passes) cancels/drains at the host layer. The investigator's "RequireNoInflightBackgroundTasks not found on enter/exit" is simply wrong (it is in meta.rs). The BlockInIsolatedMode on exec_command they cite (execution.rs:293) is a TEST FIXTURE; production wires BlockInIsolatedMode only on AskAdvisor (meta.rs:85), matching in-tree Python which wires it only on ask_advisor.py. Severity → none.
  • D3 (low/missing) Sampler loop (isolated_workspace.sampled cadence) not implemented → CONFIRMED (adjusted to low/none-runtime). Python pipeline.py:246-294,339-364 starts _sampler_loop in initialize() emitting isolated_workspace.sampled every sample_interval_s (default 0.5s). Rust carries the cap (caps.rs:61,115-119) but grep sampler|sampled|sample_interval in eos-isolated/src + eos-daemon/src finds only the cap field — no emitter loop. Genuinely absent. Impact: a pure observability/audit-cadence lane (sample lane), not a correctness/isolation invariant; the cap is preserved so it can be wired later. Severity low.
  • D4 (low/divergent) TTL sweep cadence fixed 500ms vs max(0.5, min(ttl_s/2, 30.0)) → CONFIRMED. py pipeline.py:264 interval = max(0.5, min(ttl_s/2.0, 30.0)). rs server.rs:131-143 drives crate::isolated::ttl_sweep on a fixed Duration::from_millis(500) tick (not read in session.rs). The stale CONDITION matches (session.rs:486-497 now - last_activity > ttl_s AND not active), only the tick cadence differs. With the default 1800s TTL, the only observable difference is the daemon waking every 0.5s instead of every 30s — a CPU-cost detail, not an eviction-semantics divergence. Severity low.
  • D5 (low/divergent) TTL 'active' guard uses command-session presence vs active_calls counter → CONFIRMED (adjusted). py pipeline.py:295-303,366-396 skips stale when h.active_calls == 0, where active_calls is a per-handle foreground in-flight counter. rs session.rs:486-497 skips agents in active_agents derived from active_command_sessions (isolated.rs:559-569); WorkspaceHandle (session.rs:57-95) has no active_calls field; record_tool_call only bumps last_activity. Different "active" definition. In the Rust daemon model the unit of foreground work IS a command session (registered on start, unregistered on finish — command.rs:706,1172,1438), so command-session presence is the faithful analogue of active_calls > 0. The guard's intent (don't evict a workspace with live foreground work) is preserved; the bookkeeping is re-expressed. Severity low.
  • E8 (low/missing) Orphan reaper does not /proc-scan for ns-holder processes lacking a persisted row → CONFIRMED. Python orphan_reaper.py:257-310,313-339 _reap_orphan_holder_processes + _iter_namespace_holder_processes walks /proc/<pid>, matches cmdline _NS_HOLDER_MARKER="sandbox.isolated_workspace.scripts.ns_holder", and SIGTERM/SIGKILLs stale holder trees whose pid/ppid is not in the live set — i.e. it reaps holders even with NO persisted row. Rust session.rs:626-644 reap_startup_orphans only reaps persisted-row holders (reap_persisted_holder:687-703) plus named veth/cgroup/scratch sweeps (:759-819); there is NO /proc scan for marker-matched orphan holders. Adjusted nuance: in Rust the holder is an in-process eosd ns-holder child tracked in a holder_children() registry (isolated.rs:77-80,187) rather than an unshare-detached ns_holder.py, so the dominant cross-restart leak the Python scan targeted is less likely; but a daemon hard-crash mid-enter (handle never persisted) can still strand a holder the Rust startup path will not collect by /proc scan. Real gap, low severity.

Overall verdict

High-fidelity port. All 7 core isolation invariants (agent-id keyed lifecycle, own net namespace, persistent upperdir, capture-but-never-publish, full teardown-on-exit, enter-rejects/exit-cancels background work, plugin/LSP block) are independently confirmed as correctly implemented in Rust with byte-for-byte constants and the strict > capacity gate. The investigator's two MEDIUM disparities (D1, D2) are REFUTED: they describe the DELETED _control_plane drain machinery, not the live in-tree contract, and D2 is an outright false alarm — RequireNoInflightBackgroundTasks IS wired on both isolated tools in meta.rs:65-67. The three genuine gaps are all LOW and observability/cleanup-only: the isolated_workspace.sampled sampler loop (D3) is unimplemented (cap preserved), the TTL sweep cadence is a fixed 500ms (D4) and its "active" guard uses command-session presence instead of an active_calls counter (D5, faithful re-expression), and the startup orphan reaper lacks a /proc ns-holder scan (E8, partly mitigated by in-process holder tracking). None of these weaken the isolation, no-publish, or teardown guarantees. Net: the dangerous "FALSE MATCH" class is absent here; the dangerous direction was actually the opposite — the investigator over-reported two MEDIUM disparities that the Rust already satisfies.
