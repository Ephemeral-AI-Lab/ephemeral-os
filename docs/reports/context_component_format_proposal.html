Context Component Format Proposal

Comparing the current markdown rendering vs. proposed XML-component rendering for the goal_iteration_frame recipe output. Attribute change: sequence_no → iteration_no.

1. Side-by-side: same data, two renderings

Current (markdown headings)

# Goal ## Goal Make the workspace pass the 2023.4.0 PR description. ## Iteration 1 accepted plan Run a workspace preflight probe and continue with the follow-up goal. ## Iteration 1 summary (empty) # Current Iteration Continue the first-three-messages capture by running one more preflight in iteration 2.

Proposed (XML components)

<goal> Make the workspace pass the 2023.4.0 PR description. </goal> <iteration iteration_no="1" status="prior"> <accepted_plan> Run a workspace preflight probe and continue with the follow-up goal. </accepted_plan> <summary> (empty) </summary> </iteration> <current_iteration iteration_no="2"> Continue the first-three-messages capture by running one more preflight in iteration 2. </current_iteration>

3. Attribute schema

TagAttributesNotes
<goal>—Top-level goal text. One per packet.
<iteration>iteration_no, statusstatus ∈ {prior, current}. iteration_no replaces former sequence_no wording.
<accepted_plan>—Child of <iteration>. Replaces ## Iteration N accepted plan.
<summary>—Child of <iteration>. Replaces ## Iteration N summary.
<current_iteration>iteration_noInline body, no child tags. Authoritative scope for the planner.

5. Where the change lives

Almost entirely in backend/src/task_center/context_engine/renderer.py (_render_block + _render_group). Recipes already produce structured ContextBlocks with the necessary metadata; the renderer just stops stringifying it into headings and starts wrapping in tags.

Rename: per your call, the iteration ordinal attribute is iteration_no, not sequence_no. The underlying Iteration.sequence_no field on the state model can stay as-is; only the rendered attribute name changes.
