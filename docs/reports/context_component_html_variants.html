XML → HTML: three ways to spell the same component

The XML you saw before renders in browsers but isn't valid HTML5. Below are three variants that all encode the same iteration component, ranked by HTML compliance.

Variant A — XML-as-rendered-by-browser (works, but not valid HTML)

What the previous proposal showed. Browsers render it; an HTML5 validator would flag every tag.

<iteration iteration_no="1" status="prior">
  <accepted_plan>
    Run a workspace preflight probe and continue with the follow-up goal.
  </accepted_plan>
  <summary>(empty)</summary>
</iteration>

Validator verdict: Fail. Custom element names without hyphens, attribute name iteration_no with underscore.

Variant B — HTML5 custom elements (valid HTML5)

Custom element names must contain a hyphen per the HTML5 spec. Attributes should be lowercase with hyphens.

<task-iteration iteration-no="1" status="prior">
  <accepted-plan>
    Run a workspace preflight probe and continue with the follow-up goal.
  </accepted-plan>
  <iteration-summary>(empty)</iteration-summary>
</task-iteration>

Validator verdict: Pass. All tag names are valid custom-element names; all attribute names are lowercase.

Variant C — Semantic HTML with classes & data-* (most portable)

Use the standard HTML element vocabulary (<section>, <article>, <header>) and put the type info in class + data-*. This is the most portable form — works in every parser, every renderer, every framework.

<article class="iteration" data-iteration-no="1" data-status="prior">
  <header>Iteration 1 · prior</header>
  <section class="accepted-plan">
    <h4>Accepted Plan</h4>
    <p>Run a workspace preflight probe and continue with the follow-up goal.</p>
  </section>
  <section class="summary">
    <h4>Summary</h4>
    <p>(empty)</p>
  </section>
</article>

Recommendation for the renderer

The renderer's output is consumed by Claude, not by a browser or HTML validator. "Is it valid HTML?" is the wrong question; the right question is "do the structural tags help the LLM segment the context?"

For the prompt: Variant A (plain XML tags) is fine and the cleanest. Claude is explicitly trained on this pattern. The transcript is not served to a browser; it lives in message.jsonl and gets fed to a model API.

For the debug viewer / transcript inspector (if you want one): Variant B or C in a separate HTML-rendering layer that post-processes the raw prompt. The wire format stays XML-clean; a viewer transforms it to valid HTML on read.

Concrete split:

  • Wire format (what Claude sees): Variant A — tight, semantic, low token cost.
  • Debug viewer (what humans see in a browser): Variant B or C — produced by a one-pass transform from A.
