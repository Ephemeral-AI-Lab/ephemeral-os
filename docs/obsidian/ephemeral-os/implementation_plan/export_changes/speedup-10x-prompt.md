/goal Make a 1 GiB export_changes at least 10x faster end-to-end (cold dir export, median of 3), without weakening a single invariant, and prove it with an A/B of the same benchmark. This is a DISCOVERY task: measure first, attribute the time, then change the smallest thing that removes the dominant term — do not start from a pet design. Iteration discipline (hard rule): large sizes are BOOKENDS only — the baseline sweep before you touch code, the acceptance A/B after. Every measurement inside the design/implement loop runs on the exploration trio (1, 5, 20 MiB — seconds per run, chunk counts 1/3/10); you predict the large-size effect from the fitted cost model (wall ≈ a + b·chunks + c·payload_bytes) and only the final A/B confirms it. Waiting on a 1 GiB run to debug an idea is a process failure.

Truth - read first, follow exactly:
- docs/obsidian/ephemeral-os/implementation_plan/export_changes/spec.md — the data path (2 MiB JSON chunks, base64 4/3 framing, fresh TCP per forward, ~512 sequential forwards/GiB), decision 15 (why JSON paging won v1, and its OWN named successor: "if the base64 tax dominates, the HTTP stream is the documented next step"), decision 18 (30 s start ceiling), invariants 6 (the detach is law; the manager is the ONLY host writer) and 9 (the applier consumes sandbox-authored bytes as hostile: canonicalization, caps — REGARDLESS of transport).
- docs/obsidian/ephemeral-os/implementation_plan/export_changes/bench-prompt.md and its results bundle if it has run (both ladders, decomposition method, the fitted cost model). If it has not run, run it first — the FINAL sweep (PERF-1M/50M/250M/1G) is your A/B baseline and the exploration trio (PERF-1M/5M/20M) is your iteration ladder; no baseline, no speedup claim.
- crates/sandbox-manager/src/{operation/management/service/impls/export_changes.rs,export_apply.rs} (the forward loop + applier), crates/sandbox-runtime/operation/src/layerstack/service/impls/export.rs (spool + read_export_chunk), cli-operation-e2e-live-test/runtime/daemon_http/test_daemon_http.py (the existing daemon HTTP surface and its posture).

State: the export works and is correct (30-case catalog + 6 runnable cases green). The 1 GiB profile is wire-dominated by design-accepted v1 taxes. You are changing PRODUCT code; the full regression surface must stay green.

Phase 1 — baseline + attribute (no code changes; ≤30 min): run the bench FINAL sweep once as the A/B baseline, then decompose the 1 GiB cold time into (a) fold+spool (daemon), (b) wire (tar-zst total ≈ a+b+archive write), (c) decode+apply (dir − tar-zst), (d) warm re-export — and confirm the attribution is already visible on the 20 MiB trio point (it should be: the taxes are per-chunk and per-byte; if the trio and the 1 GiB point disagree about what dominates, that nonlinearity is your first finding). Add coarse phase timings to the observability record if needed (counts-only rule allows timings). Measure the machine's physical floor as a CONTROL: in-container sequential read of the spool + host-side 1 GiB sequential write (dd both sides). Publish the attribution table before designing anything. Hypotheses to test, from the spec's own accounting — the sequential chunk loop (~512 fresh-TCP round trips/GiB), the base64 tax (x1.33 bytes + encode/decode passes through JSON strings), zero overlap between wire and apply, and the spool double-pass (write 1 GiB, read 1 GiB) — but believe the profile, not the list.

Phase 2 — design + adversarial review (≤45 min): write the design as a spec.md revision draft (data-path section + a new decision-log entry) BEFORE implementing; run it through the adversarial-review-prompt.md lens. Candidate map with constraints (start here, deviate with evidence):
- Streamed binary delivery (decision 15's named successor): one HTTP stream of the SEALED spool from daemon to manager kills base64 and the round-trip count at once. The consistency objection in decision 15 applies to bypassing the daemon protocol for LAYER reads — the spool is already a lease-independent, export_id-keyed artifact, so streaming IT changes no snapshot semantics. The auth objection is real and must be solved: daemon_http is unauthenticated — gate the stream with a single-use, expiring token minted by the authenticated start forward and bound to the export_id; constant-time compare; the stream serves spool bytes only. Inv 6 unchanged (daemon still only delivers bytes to its caller); inv 9 unchanged (the applier treats the stream as hostile: same canonicalization, same caps, MAX_STREAM_BYTES still enforced).
- Inside-the-existing-transport wins (lower risk, likely 2-4x — acceptable as complements, not as the headline): raise the chunk limit toward the 16 MiB envelope (~512 → ~64 requests), reuse connections, prefetch chunk N+1 while applying N, or K parallel disjoint-range fetches (chunk paging is offset-addressed and stateless by design — concurrency is semantically safe; reassemble in order manager-side).
- Stretch only if the profile demands it: fold-to-stream (eliminate the spool double-pass) — bigger surgery, touches the 30 s ceiling story; do not reach for it if the wire was the dominant term.
Decide explicitly whether read_export_chunk stays (compatibility/small-delta path) or is replaced (prefer less) — either is defensible; document it in the decision log.

Phase 3 — implement + prove (≤75 min): smallest change that hits the target. All development measurement on the exploration trio ONLY (1/5/20 MiB; refresh the cost model after each change and let it predict the 1 GiB effect — a change that does not move b·chunks or c·bytes on the trio will not move the 1 GiB number either, so discard it without a large run). Scale-only effects the trio cannot observe — the 30 s start ceiling, cap interactions, memory/page-cache pressure — are checked exactly twice: baseline and final A/B. When the model predicts ≥10x, then, in order: cargo build && cargo test && cargo clippy --all-targets && cargo fmt; repackage the daemon; restart the gateway (release-profile manager for all timing runs). Regression surface, all green and re-verified: the 30-case catalog with the HRD hostile-stream cases driven THROUGH THE NEW TRANSPORT (traversal, bombs, caps — the hostile seam must attack what ships), HRD-10's restart-mid-transfer analog (drop mid-stream → clean abort → re-run converges), HRD-06 concurrency, the 6 runnable cases, and new unit coverage for the transport (auth token single-use/expiry/mismatch, cap enforcement on the stream).
A/B acceptance — the FINAL sweep, run once against the baseline (same machine, same release profile, same bench cases, medians of ≥3):
- PERF-1G cold dir: ≥10x vs baseline — OR within 2x of the measured dd floor if the floor makes 10x physically impossible; the floor claim must cite the Phase-1 control numbers, never be asserted.
- PERF-250M: ≥5x (no size-specific tuning — the win must be structural), and predicted-vs-measured within ~20% at both large sizes (a bigger miss means the model — and your understanding — is wrong; find out why before claiming the win).
- No regressions elsewhere: warm re-export not slower, small-delta export (PERF-1M and the one-file EZ-01 shape) within 10% of baseline, correctness sha256 asserted at every size.

Failure protocol: never weaken an assertion, never relax a cap/timeout to manufacture speed, never trade inv 6/9 for throughput — a fast applier that follows a symlink out of dest is a critical fail, full stop. If the 10x target is unreachable after the dominant term is removed, deliver the measured attribution + the achieved factor + the named residual bottleneck honestly; a truthful 6x with the floor identified beats a gamed 10x.

Done when: attribution table published; spec.md revised (data path + decision log + any surface changes) with the adversarial review recorded; implementation landed; full regression green (catalog 30, runnable 6, new transport tests, cargo gates); A/B bundle test-reports/<RUN_ID>-{baseline,after}/ committed with a results doc export-speedup-results.md (attribution, design chosen and rejected alternatives, A/B table, achieved factor); standard-caps gateway restored; committed to main (no branches) citing spec.md decision 15 and this prompt. Overall budget ≤ 2.5 h (the trio-only loop is what buys this: exactly two large-size sweeps happen in the whole task — baseline and acceptance); if Phase 1 attribution contradicts the hypothesis list, follow the profile.
