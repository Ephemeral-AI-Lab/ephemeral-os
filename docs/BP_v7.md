# Ephemeral OS – 商业计划书

**商业报告书 · 2026 年 4 月 · CONFIDENTIAL**

## 1. 项目定位：Multi-Agent 原生的 Coding Agent

Ephemeral OS 是一款以 **Multi-Agent 原生架构**为核心的 AI Coding Agent 系统。

我们选择从压力最大、可验证性最强的场景——软件工程——切入，围绕 **瞬态智能体集群、Overlay 共享沙盒、天眼系统、Tree-of-DAG 调度 Harness** 构建了一套完整的基础设施，期望为 AI 编码范式带来实质性的推进：

- 用户输入高层自然语言目标，系统自动将其拆解为可并行的子任务；
- 前端、后端、数据库、测试等专精智能体在同一个实时共享沙盒中协同工作；
- 任务完成后，所有智能体瞬态释放，不留冗余状态；
- 长任务、跨仓库、跨天级的复杂工程交付，有机会第一次成为可规模化完成的事情。

一句话：**Cursor、Claude Code、Devin 代表了"AI 程序员"这条路；我们希望走出"AI 工程协作系统"这一步。**

## 2. 行业现状与值得解决的问题

### 2.1 长任务仍是行业共同的瓶颈

当前主流 AI 编程工具——Claude Code、Codex、OpenCode、Cursor、Devin 等——都在长任务上遇到了相似的挑战：**一条指令往往难以完成一个真正的长任务**。

- 在 SWE-bench / SWE-Evo 等长任务基准上，SOTA 单智能体得分目前大约在 25% 左右。
- SWE-Evo 最难档的任务涉及 **5000 万行代码、约 2700 个 P2P 测试**，已经接近中大型企业系统的整体改造量级，目前还没有哪一个 single-agent 架构能完整跑通。
- 这一瓶颈的根源，很大程度不是模型能力不足，而是**单智能体在长链路下的上下文管理成本**——上下文越长越难维护，错误与不一致累积得越快，回滚成本也越高。

我们认为，这是一个真正值得从架构层面回答的问题。

### 2.2 一个值得关注的观察：CooperBench 的 50 → 25

在 Claude Team、Codex、OpenCode 等产品已经支持多个 subagent 协作编辑同一个 workspace 的前提下，一个值得关注的观察是：

> **在 CooperBench 上，single-agent 约能取得 50 分；而同一批任务换成基于 A2A / P2P 通信的"multi-agent"方案时，得分会回落到 25 分左右。**

这并不是对任何一款产品的否定——恰恰相反，它说明"从 single-agent 加装 multi-agent 能力"这条路径本身存在一些尚未解决的工程问题。下一章我们会展开讨论。

### 2.3 其它结构性痛点

| 多 Agent 并发写冲突 | 依赖锁或串行化，吞吐受限     | **Overlay + OCC**：独立 overlay 层 + 乐观并发控制 |
| 安全合规门槛        | 隔离与审计链条不完整       | 独立沙盒实例 + 私有化部署                          |

## 3. 核心理论：共享 Workspace 已是共识，真正的差异在三件事

### 3.1 现状客观描述

需要先做一个客观说明：Claude Team、Codex、OpenCode 都已经支持多个 agent 在同一个 workspace 上协作编辑——这一点行业已达成共识。Ephemeral OS 与它们的差异并不在"是否共享 workspace"，而在共享之后，如何让协同真正变得高效。

我们观察到三个仍值得继续工程化的问题：

#### ① Agent 间以 P2P / A2A 消息为主，存在信息传递成本

- 现有方案中，subagent 之间主要通过 **agent-to-agent / peer-to-peer 消息**沟通。
- 即便共享了 workspace，接收方通常不会主动观察变化，而是依赖显式通知。
- 每一次消息传递既是一次同步点，也带来额外的理解成本。CooperBench 中 50 → 25 的回落，很大程度反映的就是这一类协同开销。

#### ② 任务切割粒度仍偏粗

- 现有架构的任务拆解多由一个 single-agent 完成，拆解粒度自然受限于单体的上下文与认知边界。
- 拆出的子任务往往仍以"完成模块 X"、"重构服务 Y"为单位，单个子任务运行时间较长，上下文消耗也较大。
- 粒度偏粗，会让子任务之间真实的依赖关系难以被精确刻画，可并行的部分也就难以被充分识别。

#### ③ 实际可并发的 Agent 数量受到限制

- 受①②的影响，现有产品中**实际可稳定运行的 agent 数量普遍在 5 个以内**。
- 一旦多个 agent 改到相邻区域，主要依赖 git merge 来解决冲突，行级冲突容易触发，并发会逐步回退到近似串行。
- 在缺乏 OCC、语义合并与裁决 Agent 的情况下，并发度向上扩展会遇到瓶颈。

### 3.2 Ephemeral OS 的系统性回应

| 行业普遍现象                   | Ephemeral OS 的补强                                                                                                                                   |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| ① P2P / A2A 消息带来的同步与传递成本 | **天眼系统**：让 Agent 不需要"被告知"，而能直接"看见"。动作、产出、依赖关系实时呈现给调度层与其它 Agent，显著减少显式通信。                                                                           |
| ② 任务切割粒度偏粗               | **Tree-of-DAG 调度 + Multi-Agent 原生拆解**：调度器持续观察 workspace 状态，把任务递归拆解到接近"原子级 patch"的粒度，依赖关系因此能被精确刻画。                                                  |
| ③ 并发能力受限                 | **Overlay + OCC + 语义合并**：每个 Agent 在自己的 overlay 上自由读写、不加锁；提交时由 OCC 校验，冲突区域由语义合并层与裁决 Agent 协同处理。**实际稳定运行 10 个以上 Agent 并发协作**，相较行业普遍的 ≤5 个，提供了一个新的上限。 |

> 行业已经走到了共享 workspace 这一步，我们希望在它之上再补齐天眼、细粒度调度与 OCC 并发控制三块基础设施，让共享真正转化为高效协同。

## 4. 产品与技术优势

### 4.1 瞬态智能体引擎

按需诞生、专注执行、使命完成即释放；集体智慧在系统层持续沉淀。

### 4.2 Overlay 沙盒：下一代 Git Worktree + OCC 合并

这是 Ephemeral OS 比较深的一条护城河。它不是一个特性，而是一整层基础设施。

#### 4.2.1 Git Worktree 的局限

业界最朴素的"多 agent 并行写代码"方案，是为每个 agent 开一个 git worktree。但在 multi-agent 场景下，worktree 有几个限制：

1. **粒度较粗**：worktree 是整个工作目录的物理拷贝，开销较大、启动较慢，不太适合支撑高并发瞬态 Agent。
2. **状态互不可见**：worktree 之间彼此不可见，中间产物需要等 commit + merge 才能被感知。
3. **冲突检测偏文本层**：git merge 主要看行级 diff，对代码语义的理解有限。
4. **缺少运行时观测**：worktree 是静态文件视图，天眼系统较难介入。

Git worktree 的设计面向的是人类开发者，不是为高并发 AI Agent 的协作场景量身设计的。

#### 4.2.2 我们的 Overlay

Overlay 是我们自研的一层 **"瞬态智能体专用的可叠加文件系统 + 并发控制运行时"**。可以理解为**下一代 Git Worktree**：共享只读基底 + 每个 Agent 一层轻量可写 overlay + OCC 智能合并 + 天眼实时观测。由四层咬合构成：

**① 文件系统层：分层、瞬态、Copy-on-Write**

- 共享只读 base layer（仓库快照），所有 Agent 共用，零拷贝、零启动开销。
- 每个 Agent 派生独立可写 overlay 层，只记录差异（CoW），秒级创建、秒级回收。
- 文件读取按 overlay → base 顺序穿透，Agent 看到"私有 + 共享"叠加后的完整视图。

**② 并发控制层：OCC 乐观并发控制**

- 多个 Agent 在自己的 overlay 上自由读写、不加锁，最大化并行度。
- 提交时进行 OCC 校验：读过的文件版本号与当前 base 是否一致。
- 一致 → fast-forward 合并；不一致 → 触发语义级冲突解决。
- "先做后校验"比传统加锁更贴合 multi-agent 协作的实际场景。

**③ 语义合并层：超越行级 diff**

- 在 AST、符号表、语义引用层进行合并判断。
- 改同一函数不同语句 → 自动合并；改同一行不同字符 → 轻量裁决 Agent 协助决议；语义级冲突 → 上交上层 Agent 重新规划。

**④ 可观测层：与天眼系统咬合**

- 读写、版本变化、OCC 冲突、合并决策都被天眼实时捕获。
- Overlay 不只是文件系统，它同时也是 multi-agent 系统的"事件总线"。

#### 4.2.3 为什么构成护城河

Overlay 同时涉及 **分布式文件系统、OCC 并发控制、语义级代码理解、实时可观测性** 四件事。单独任何一件都不算新，但把四件事咬合在同一个运行时、面向 multi-agent 场景端到端优化，需要较长的系统级工程积累。

### 4.3 Tree-of-DAG 调度架构：Agent-Team + Subagent Swarm 的结合

这是 Ephemeral OS 第二条比较独特的架构选择。

#### 4.3.1 为什么不是普通 DAG？

传统 agent orchestration 使用扁平 DAG：任务拆成一组有依赖关系的子任务，按拓扑顺序推进。问题有二：

1. **粒度刚性**：DAG 一旦确定就不易调整，遇到复杂任务容易出现"某一个节点依然很重"的情况。
2. **失败处理粗**：一个节点失败，要么重跑、要么整条链路回退，缺少更细的恢复机制。

#### 4.3.2 Tree-of-DAG：每个节点本身可以是一个 DAG

Ephemeral OS 采用 **Tree-of-DAG** 结构：

- **顶层是一个 agent-team DAG**：由多个角色互补的 agent（前端、后端、测试、审阅等）构成，描绘任务的主干依赖关系。
- **每个节点内部可以再展开为一个子 DAG**：当某个节点本身太重时，承接该节点的 agent 会派生出 **subagent swarm**，把工作进一步拆解为更细的子 DAG 并原地执行。
- **递归可嵌套**：子 DAG 的节点如果依然过重，还可以继续展开下一层。
- 这使得"agent-team（横向多专家协作）"与"subagent swarm（纵向递归拆解）"在同一套数据结构里被统一表达。

#### 4.3.3 Replan Harness：让长任务可以被"修"而不只是"重跑"

Tree-of-DAG 的另一个重要特性是与 **Replan Harness** 的深度结合：

- 任何节点执行失败或得到的结果不达预期时，系统不会立刻重跑整条链路，而是触发该节点的 **局部 replan**。
- Replan 发生在**最小可能的子树**上：只重新规划失败点所在的那部分子 DAG，其它已经成功的分支保持不变。
- Replan 结果被合并回父 DAG，整个 Tree-of-DAG 保持一致。
- 配合 Overlay 的 CoW 回滚能力，失败节点的中间状态可以干净丢弃而不影响其它 Agent。

#### 4.3.4 这套架构带来的直接收益

- **粒度可以足够细**：通过递归展开，每个实际执行的叶子节点都是 agent 上下文能舒适承载的原子级任务。
- **并发可以足够高**：同一层 DAG 里无依赖的节点可同时展开为 swarm，实际稳定并发 10+ Agent。
- **失败可以局部恢复**：长任务不再"一错全错"，replan 的代价只与失败子树大小成正比。
- **架构可以统一表达**：行业中常见的 agent-team 与 subagent 两种范式，在我们这里是同一件事的不同层级。

### 4.4 Harness 调度层

围绕 Tree-of-DAG，提供任务拆解、依赖管理、波次调度、失败重试、断点续跑、预算控制一体化——这是 single-agent 路线相对薄弱的一块。

### 4.5 模型中立 + 弹性部署

兼容 Claude / GPT / Qwen / DeepSeek / Kimi / Minimax 等主流模型；云端 SaaS + 本地私有化双模式。

**核心优势一句话**：**Multi-agent 原生架构 + Tree-of-DAG 调度 + Overlay 并发 + Replan Harness 的组合，让我们在长任务和高并发任务上，能跑到现有 single-agent 派生方案难以抵达的区间。**

## 5. 长期愿景：从 Coding Agent 走向通用 General Agent

### 5.1 所有通用 Agent 的底座都是 Coding Agent

无论龙虾、Manus，还是任何一个冲击"通用 AI agent"的产品，底层做的都是同一件事：

> 把用户的自然语言目标，翻译成一系列可执行的代码/命令/工具调用，在一个沙盒里运行、观察、修正、迭代。

调用 API、操作浏览器、清洗数据、跨系统编排、自动化办公——本质都可以归结为写代码。**通用 agent 的底座 = coding agent + 工具适配层 + 沙盒**。coding agent 底座越强，通用 agent 的天花板就越高。

### 5.2 龙虾（OpenClaw）：路径已被验证，pi-mono 底座仍有提升空间

龙虾代表了目前通用 agent 的一个较高水平，证明了模型中立 + 工具适配 + 沙盒这条路线是走得通的。

同时我们观察到一个值得讨论的技术选择：**龙虾的底层是 pi-mono——一个单体 coding agent**。这意味着：

1. **单体上下文天花板**：规划、决策、执行、反思都挤在同一个 agent 的上下文里，任务越长，上下文管理成本越高。
2. **Multi-agent 协同偏弱**：派生的 subagent 仍是独立 single-agent 副本，缺少共享沙盒与实时可见性，更多依赖主 agent 串行转述。

这是一个从第一天就埋下的架构选择，短期内并不容易改造。

### 5.3 Ephemeral OS 的演进机会

| <br />       | 龙虾（pi-mono）  | Ephemeral OS（multi-agent 原生）          |
| ------------ | ------------ | ------------------------------------- |
| 底层 coding 内核 | Single agent | **Multi-agent 原生**                    |
| 上下文          | 单体，长任务下管理成本高 | 每个瞬态 Agent 上下文独立、保持干净                 |
| 并行度          | 以派生为主        | **原生并发，10+ Agent 稳定协作**               |
| 长任务稳定性       | 受单体边界制约      | **通过 Tree-of-DAG + Replan 获得结构化恢复能力** |
| 通用任务覆盖       | 较高           | **对等，且可横向扩展**                         |

我们希望的，不是取代龙虾，而是**用一套更适合多 agent 协同的底座，把通用 agent 这条路延伸到更远的地方**。

### 5.4 三步走：从 Coding Agent 到通用 Agent OS

**Step 1（2026）**：以编码场景为锚，把 multi-agent 原生底座打磨到生产级。SWE-Evo 目标从 \~25% 提升到 50%+。

**Step 2（2026 Q4 - 2027）**：抽象 **Universal Workspace**——文件、浏览器、终端、API、第三方工具统一为可 overlay、可观测、可回滚的资源。任务协议从"写代码"泛化到"完成任意数字任务"。

**Step 3（2028+）**：插件 / 技能市场 + 垂直行业方案（金融 / 医疗 / 制造 / 政务），朝通用智能操作系统演进。
