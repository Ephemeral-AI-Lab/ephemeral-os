EphemeralOS Sandbox Workspace Architecture

Current code-backed guide to backend/src/sandbox, focused on LayerStack, overlay, OCC, workspace modes, daemon/background handling, provider adapters, and ephemeral-workspace plugins.

Evidence rule This page is based on the current checkout. Behavioral claims include code references such as backend/src/sandbox/layer_stack/stack.py:105-129. It does not rely on the older docs/sandbox-architecture.html behavior where normal snapshots were described as rendered lowerdirs.

1. Overview §

Why need it?Many agents and plugins need to inspect or mutate one repository without corrupting shared state.
What it doesSeparates durable workspace truth, temporary execution state, conflict control, and runtime dispatch.
Works withLayerStack stores versions, overlay exposes a filesystem, OCC publishes changes, daemon routes calls.
Big pictureThe system makes every operation see a clear snapshot and only publishes through one controlled path.
Tool / Pluginread, write, edit, shell, LSP
→
DaemonOP_TABLE, in-flight registry
→
Workspace pipelineephemeral or isolated
→
LayerStackmanifest + immutable layers
Overlaymount layer paths + private upper/work
→
OCCvalidate, stage, commit queue
→
Publishernew immutable layer + manifest CAS

The important split is lifetime. main_workspace is the persistent repository identity. ephemeral_workspace creates private overlay state per tool/plugin operation. isolated_workspace creates one private overlay namespace at enter_isolated_workspace and discards it at exit_isolated_workspace.

2. LayerStack §

Why need it?Overlay users need a stable snapshot while other calls keep publishing new changes.
What it doesStores workspace versions as immutable layer directories plus one active manifest.
Works withOCC publishes layers, overlay mounts layer paths, plugins lease projections.
Big pictureLayerStack is the durable truth; overlays are temporary views over that truth.

2.1 Storage §

Why need it?The runtime needs one place to find manifests, layers, staging, and workspace binding.
What it doesStores manifest.json, workspace.json, layers/, staging/, and metadata.
Works withDaemon bootstrap builds the base; workspace binding prevents path ambiguity.
Big pictureThe repository is copied once into a base layer, then every change is an added layer.

The default layer stack root is /tmp/eos-sandbox-runtime/layer-stack backend/src/sandbox/daemon/paths.py:5-16. Actual calls may pass another layer_stack_root. 2.2 Projection §

Why need it?Tools need the latest workspace view without copying the full repository for each call.
What it doesReturns a lease plus ordered layer_paths for the active manifest.
Works withOverlay mounts those paths; MergedView reads them directly for fast read APIs.
Big pictureA projection is not a rendered tree by default; it is a pinned list of existing layers.

Current LayerStack.prepare_workspace_snapshot() reads the manifest, acquires a lease, and returns layer_paths copied from the manifest. It sets layer_stack.materialize_s to 0.0 and does not build a transient lowerdir for normal operations backend/src/sandbox/layer_stack/stack.py:105-129. The result object contains lease_id, manifest_version, root_hash, manifest, timings, and optional layer_paths backend/src/sandbox/layer_stack/stack.py:43-62.

L000004-...

Projection handle

MergedView still exists for direct logical reads and materialization. It reads newest-first, honors whiteouts and opaque directories, and can list directories or iterate paths backend/src/sandbox/layer_stack/view.py:60-193. Its materialize() method applies layers oldest-first into a destination backend/src/sandbox/layer_stack/view.py:195-219. That rendered tree is used by flush/squash/special paths, not by normal per-tool overlay lowerdirs.

2.3 Publish §

Why need it?Concurrent writers need an atomic way to append changes without losing another commit.
What it doesStages layer changes, moves the layer into layers/, then atomically writes a new manifest.
Works withOCC validates changes before calling the publisher; leases prevent live layers from being removed.
Big pictureEvery successful mutation becomes one new immutable layer and one manifest version bump.

The publisher performs a CAS check before writing, writes changes into a staging layer, moves it into layers/, then re-checks the manifest before writing the new active manifest backend/src/sandbox/layer_stack/publisher.py:49-132. Delete and opaque directory semantics use overlay-style marker files backend/src/sandbox/layer_stack/changes.py:123-142. Leases pin layers by refcount and expose pinned layers to squash/GC backend/src/sandbox/layer_stack/lease.py:38-86.

3. Overlay §

Why need it?Shells and plugins need a normal filesystem, not a layer-by-layer API.
What it doesMounts layer paths as read-only lowerdirs plus a private writable upperdir/workdir.
Works withLayerStack supplies lower layers; OCC consumes captured upperdir changes.
Big pictureOverlay turns a versioned layer log into a temporary working tree for one operation or one isolated session.
workspace_root mountpoint
what the tool sees inside its mount namespace
upperdir
private writes for this operation/session
workdir
overlayfs scratch required by the kernel
layer_paths
shared lowerdirs from the active LayerStack manifest

The overlay lifecycle primitive allocates a per-operation run dir, leases a snapshot, requires layer_paths, allocates upper and work, and returns an OverlayHandle backend/src/sandbox/overlay/lifecycle.py:20-79. Upper/work directories live under /eos-mount-scratch/eos-sandbox-runtime; there is intentionally no fallback because overlayfs requires a suitable writable filesystem backend/src/sandbox/overlay/writable_dirs.py:13-52.

The kernel mount helper calls Linux fsopen, fsconfig, fsmount, and move_mount. It adds each layer with lowerdir+ in manifest order, where the first path is the highest-priority lower backend/src/sandbox/overlay/kernel_mount.py:45-67. Inputs are validated with O_DIRECTORY | O_NOFOLLOW and passed as /proc/self/fd/... paths to pin the checked directories backend/src/sandbox/overlay/kernel_mount.py:139-198.

Does overlay replace the whole filesystem? No. A fresh ephemeral operation runs in a new mount namespace and overmounts only the bound workspace_root. Other inherited filesystem paths remain available subject to denylist policy. Isolated workspace does the same idea once inside a persistent namespace at /testbed. The code mount target is workspace_root, not / backend/src/sandbox/overlay/kernel_mount.py:45-67.

Fresh ephemeral calls write a JSON payload and launch unshare -Urm python -m sandbox.overlay.namespace_entrypoint backend/src/sandbox/overlay/namespace_runner.py:66-128. The entrypoint validates/mounts the overlay, runs the tool from workspace_root, unmounts, and writes timings backend/src/sandbox/overlay/namespace_entrypoint.py:85-175. For isolated workspaces, the runner passes mount_overlay=false and executes inside the already-mounted namespace backend/src/sandbox/overlay/namespace_runner.py:131-197.

4. OCC §

Why need it?Multiple operations can edit overlapping files from different snapshots.
What it doesRoutes, validates, stages, serializes, and publishes changes with stale-base rejection.
Works withLayerStack provides live manifests; overlay capture converts filesystem changes into typed OCC mutations.
Big pictureOCC is the gate that decides whether temporary workspace changes become durable truth.

OCC accepts typed changes tagged as api_write, api_edit, or overlay_capture backend/src/sandbox/occ/changeset.py:15-125. The preparer groups changes by normalized path, drops .git, routes gitignored paths to DIRECT, routes tracked paths to GATED, and attaches base hashes when needed backend/src/sandbox/occ/changeset_preparation.py:41-158.

Changeapi_write, api_edit, overlay_capture
→

→
Queueserialize publish, batch disjoint non-atomic paths
→
Transactionrevalidate against live manifest
→
Layernew immutable manifest version

The stager uses different profiles for direct and gated changes. Direct changes skip hash checks and support symlinks; gated changes check hashes and reject unsupported symlinks backend/src/sandbox/occ/path_staging.py:84-115. The commit transaction reads the live active manifest inside the layer-stack transaction, validates each path group, stages accepted deltas, and calls publish_layer once backend/src/sandbox/occ/commit_transaction.py:65-141. The commit queue runs a background worker, drains/batches work, retries bounded CAS conflicts, and splits atomic or overlay-capture changes away from unrelated batches backend/src/sandbox/occ/commit_queue.py:55-239.

Overlay changes are converted without rereading large files into memory: write changes thread content_path and final_hash into WriteChange, deletes become DeleteChange, symlinks become SymlinkChange, and opaque markers become OpaqueDirChange backend/src/sandbox/occ/overlay_change_conversion.py:19-72.

5. Workspace Modes §

Why need it?Not every task needs the same workspace lifetime or privacy model.
What it doesDefines persistent, per-operation, and explicit private-session workspace behavior.
Works withDaemon dispatch chooses isolated if open, otherwise ephemeral; main provides binding and base.
Big pictureSame LayerStack truth, different temporary writable lifetimes.

main_workspace

Persistent identity: the bound repo plus LayerStack snapshots, written only through OCC.

ephemeral_workspace

Default per-call mode: each operation gets a fresh overlay or a direct file fast path.

isolated_workspace

Opt-in private mode: enter once, run many tools, exit and discard the upperdir.

5.1 main_workspace §

Why need it?The sandbox needs one persistent repository identity to anchor every projection.
What it doesThin package anchor around layer-stack and OCC based workspace truth.
Works withHost bootstrap uploads runtime, ensures git, then ensures the workspace base.
Big pictureMain workspace is the source identity; the active LayerStack manifest is the mutable truth.

The main_workspace package intentionally remains a thin facade: its README says implementation remains in sandbox/layer_stack and sandbox/occ so the three workspace concepts can have sibling package anchors backend/src/sandbox/main_workspace/README.md:1-6. The host lifecycle calls ensure_workspace_base after uploading the runtime and ensuring git backend/src/sandbox/host/bootstrap.py:300-312. Runtime binding fails closed if the binding or manifest is missing/empty backend/src/sandbox/main_workspace/workspace_binding.py:12-35.

5.2 ephemeral_workspace §

Why need it?Most tool calls should not keep private workspace state after the operation ends.
What it doesCreates a fresh overlay for shell/search/plugin operations, or direct OCC fast paths for simple file APIs.
Works withLayerStack leases lower layers, overlay captures upperdir, OCC publishes accepted changes.
Big pictureEvery call gets a clean snapshot and releases its scratch space when done.

Daemon routing uses an active isolated pipeline only if the caller already has an isolated handle; otherwise it returns get_sandbox_overlay(..., start=False) backend/src/sandbox/daemon/tool_call_router.py:27-35. The start=False path is important: it gives the daemon a cached EphemeralPipeline facade without starting a persistent workspace mount backend/src/sandbox/ephemeral_workspace/overlay_registry.py:33-70.

5.3 isolated_workspace §

Why need it?Some low-confidence or experimental tasks need a private scratch workspace across many tool calls.
What it doesOpens one persistent overlay namespace on enter, runs tools inside it, discards upperdir on exit.
Works withEnter leases LayerStack paths; runtime wires namespace/network/cgroup; exit releases lease and removes scratch.
Big pictureIsolated workspace gives privacy and continuity without publishing changes to shared truth.

The tools module registers both lifecycle tools with the normal sandbox tool set backend/src/tools/sandbox/_lib/registry.py:17-31. enter_isolated_workspace calls host lifecycle enter with the current caller, layer stack root, background manager, and sandbox id backend/src/tools/isolated_workspace/enter_isolated_workspace/definition.py:24-47. exit_isolated_workspace calls host lifecycle exit and returns discarded upperdir bytes and phase timings backend/src/tools/isolated_workspace/exit_isolated_workspace/definition.py:24-46.

9. Space Model §

Why need it?The design must avoid copying a large repository for every tool call or background job.
What it doesShares immutable lower layers and allocates only private upper/work scratch per operation/session.
Works withLayerStack leases pin shared layers; overlay upperdirs hold only changed bytes; OCC publishes compact deltas.
Big pictureConcurrent work scales with changed data, not with repository size.
Shared lowerdirsone LayerStack layer set
→
Operation A upperonly A changed bytes
Operation B upperonly B changed bytes
Operation N upperonly N changed bytes
Lower layers are referenced by path and lease; each concurrent operation adds only its own upper and work directories.

For normal ephemeral operations and isolated enter, additional lowerdir storage is O(1) repository bytes per operation: code returns existing layer_paths and a lease, not a rendered copy of the tree backend/src/sandbox/layer_stack/stack.py:105-129. The mount still carries O(manifest depth) path references and the shell path can pre-squash deep manifests before mounting backend/src/sandbox/ephemeral_workspace/pipeline.py:221-230, but it does not allocate O(repository bytes) lowerdir space.

For n concurrent operations, writable scratch is O(n * per-operation changed bytes) plus small metadata. Each operation/session has its own upper and work directories backend/src/sandbox/overlay/writable_dirs.py:46-52. Capturing changes walks only the upperdir and emits writes/deletes/symlinks/opaque markers backend/src/sandbox/overlay/capture.py:19-93. OCC then publishes accepted changes as one new immutable layer rather than copying the whole workspace backend/src/sandbox/occ/commit_transaction.py:113-141.
