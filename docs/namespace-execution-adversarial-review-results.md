# Findings (merged, severity-ordered, deduped)

### L0 — must-fix before implementation

**F0a. [L0] The observer record's `request_id` is the EXTERNAL ORIGIN id, not the execution id; the spec's "unify identity" silently collides three meanings.**
Evidence: `namespace_execution.rs:39,175` (`request_id: begin.request_id`); `exec_command.rs:84` (`request_id: origin_request_id.clone()`); a test pins `completed[0].request_id == Some("req-external") != Some("cmd_1")` (Reviewer 3, exec_command.rs:528/565). The wire `request_id` is the *command* id (`process.rs:302 = spec.id = command_session_id.0`), a different value again.
Cut: unify only `CommandSessionId` ≡ `NamespaceExecutionId` ≡ wire `request_id`. Leave the observer field as the external origin id and **rename it `origin_request_id`** so the unification cannot be misread.
Saving: prevents an L0 regression (the cited test breaks under a naive read); collapses 3 wire/registry id meanings to 1 + keeps 1 distinct origin id.
Risk/limit: none — the test already pins the contract.

**F0b. [L0] `RunnerChild` must yield a completion *event*, not an exposed reapable `Child`, or the Bridge seam is fork-only.**
Evidence: spec dispatch step 7 calls `child.wait()` on the returned `RunnerChild` (md:249,266); a persistent runner-server backend (the seam's stated reason, md:621-624) has **no per-exec daemon-side pid** — completion is a protocol frame. Live fork uses `child.wait()` + result-fd read (pty.rs:233-243); server would not.
Cut: `RunnerChild` exposes `fn wait_completion(&self) -> io::Result<RunResult>` (fork impl = `wait()` + read result-fd; future server impl = block on Exited frame). The watcher calls `wait_completion`, never sees a pid. PTY master + pgid attach as separate fields for the `Pty` backing.
Saving: 0 fields, but makes the "swappable backend" claim structurally true instead of baking the fork shape into the engine watcher — the prompt's "bakes in the wrong architecture" hazard.
Risk/limit: fork impl still owns pgid for `cancel()` and the result-fd read; both attach cleanly to `RunnerChild`.

### L1 — large cuts

**F1. [L1] Collapse the watcher poll loop into a blocking `wait_completion()`.**
Evidence: `completion.rs:75-84` spins `take_exit()` every 5 ms; `process.rs:261 try_wait()`. Watcher has nothing to do until exit.
Cut: watcher does one blocking `child.wait()` (→ `wait_completion`), reads result-fd, finalizes, resolves the promise. Delete the `COMPLETION_POLL` watcher loop; delete `take_exit`'s `try_wait`/`exit_taken` guard + `PtyProcessExit::Running` (the watcher is sole owner).
Saving: −1 poll loop, −1 enum-variant path, −1 mutex (`exit_taken`), ~5 ms mean + up to 5 ms tail latency.
Risk/limit: **cancel-safety verified** — `cancel` is `killpg(pgid)` from the *write_command_stdin/request* thread (pty.rs:246-250,485), child runs `process_group(0)` (pty.rs:345), bash inherits pgid (no override in shell_exec). A blocking `wait()` on the watcher cannot block a `killpg` on another thread, and the kill unblocks `wait()`. Add this invariant to spec "Finalization / Terminal Semantics."

**F2. [L1] Collapse the yield-settle loop and the completed-record loop into the same condvar.**
Evidence: `completion.rs:194-211` (loop C, polls `is_exited()`); `completion.rs:214-241` (loop D, polls the `completed` map) reached from `helpers.rs:63` — loop D exists *only* because completion and finalization run on two threads with no handoff.
Cut: watcher runs `finalize` inline before `promise.resolve()`, so "resolved" ⟹ "completed record exists." Delete `wait_for_completed_record` (loop D) entirely; the Completed branch reads the resolved promise/registry. Loop C's `is_exited()` poll becomes `promise.wait_timeout(deadline)`.
Saving: −2 poll loops, **−1 whole thread** (the shared finalizer #4), −1 mpsc thread-hop. Daemon-side poll loops 3 → 0. `completion.rs` (242 LOC) deleted.
Risk/limit: the quiet-period growth detection is a separate concern (F3).

**F3. [L2→reported under L1 group] Bound the quiet-period growth check to the promise wait — stop the 5 ms `stat` loop.**
Evidence: `completion.rs:199` calls `process.transcript_len()` → `std::fs::metadata().len()` every 5 ms; QUIET_MS is itself 50 ms.
Cut: drive off `promise.wait_timeout(remaining)` and re-check transcript length only on each timeout slice (~50 ms), not 5 ms.
Saving: ~10× fewer `stat` syscalls during settle; observable quiet-period behavior unchanged.
Risk/limit: the kernel gives no readiness signal for "file stopped growing," so this stays a coarse poll, not fully event-driven — hence L2, not a 4th loop deletion.

**F4. [L1] Delete the start-ack pipe + its IPC round-trip; the request-fd read is the barrier.**
Evidence: start-ack is a 3rd pipe (pty.rs:328,391), a write at `allow_start` (pty.rs:299, *before* the request write at :303), and a daemon blocking `read_exact` (runner.rs:23,184) that runs *before* the request read (runner.rs:24). The daemon drops `request_read`/`result_write` at pty.rs:347-349 between fork and any payload write; the runner blocks in `read_payload_from_fd` (runner.rs:24,202) until the daemon writes the request. The request-fd read already provides ordering.
Cut: delete the start-ack pipe, `--start-ack-fd`, `wait_for_start_ack`/`_reader` (runner.rs:23,140-147,175-189), `RunnerCliConfig.start_ack_fd`, and the `PendingPtyProcess`/`allow_start`/`RequestPayload` two-phase split (pty.rs:40-49,291-316). Spawn writes the request immediately after the fork + `drop`s + `try_clone`.
Saving: −1 pipe (2 fds + 2 `fcntl`), −1 IPC round-trip, −1 daemon helper pair, ~40 LOC of two-phase dance.
Risk/limit: confirmed load-bearing-free — the daemon holds the only `request_write`; the runner cannot proceed early because its real work follows the blocking request read. A future server backend keeps any barrier *behind the launcher seam*, not in the protocol envelope.

**F5. [L1] Delete the result-fd reader thread; read the result-fd inline in the watcher.**
Evidence: pty.rs:359,443-451 spawns a thread that only `read_to_end`s the result pipe into an mpsc; `take_runner_result` does `recv_timeout` (process.rs:284). With F1's blocking watcher, the result is available exactly when the child exits.
Cut: after `child.wait()`, the watcher reads the result-fd directly (child closed its write end → prompt EOF). Delete `spawn_runner_result_reader`, the `runner_result_done` mpsc, `take_runner_result`'s recv.
Saving: **−1 thread per execution** (threads 4 → 2: PTY-reader + watcher), −1 mpsc, removes the 500 ms grace as a distinct hop.
Risk/limit: keep the PTY-output reader thread — it must drain the master concurrently or the child blocks on a full PTY buffer (pty.rs:398-441). Irreducible.

**F6. [L1] One id space, allocated once by the engine.**
Evidence: two independent `AtomicU64` counters — `process_store.rs:24,47-50` (`cmd_N`) and `namespace_execution.rs:19,144-147` (`namespace_execution_N`) — both allocated per command (exec_command.rs:64,76). Plus a third format `isolated-{mode}-{id}` on the mount path (setns_runner.rs:141).
Cut: engine allocates the one id; `CommandSessionId(id.0)` wraps it; mount path also takes its `request_id` from the engine id. Delete `CommandProcessStore::allocate_command_session_id` + its `next_id`; delete the `isolated-…` builder.
Saving: id spaces 3 → 1 (+ the distinct origin id, F0a); −1 atomic; removes `cmd_N` vs `namespace_execution_N` join skew.
Risk/limit: public id string changes `cmd_*` → `namespace_execution_*`; the spec accepts this via `CommandSessionId` as the public face. The workspace mount unit-test assertion (setns_runner.rs:306) moves with the builder.

**F7. [L1] Lifecycle enums 3 → 1; delete write-only `CommandLifecycleState`, fold `FinalizationState` into the promise.**
Evidence: `rg lifecycle_state` proves **zero reads** of `active.lifecycle_state` — all of process_store.rs:284, finalize.rs:201/206, quiesce.rs:197/199, coordinator.rs:66, write_command_stdin.rs:33, exec_command.rs:341 are assignments/decls. `FinalizationState` (process_store.rs:314) mirrors the promise's `None`/`Ok`/`Err`; its readers (`wait_for_completed_record`, `begin_terminal_completion`, `status_lookup.rs:30`) all live in the deleted finalizer/loop.
Cut: delete `CommandLifecycleState` (4 variants) + the field + 6 write sites; delete `FinalizationState` + `CommandCompletionPromise.exited` — the `CompletionPromise<Result<T>>` carries unresolved/Ok/Err. `status_lookup`'s `Failed` reads the promise error.
Saving: lifecycle enums 3 → 1; −2 enums, −1 promise type, ~25+ LOC; "done?" representations 3 → 1.
Risk/limit: the remount coordinator branches on the `RemountCancellationToken` and pgid, **not** on `lifecycle_state` (verified) — so deleting these three command-view variants is safe. `NamespaceExecutionLifecycle` stays as the public Phase 4.6 axis.

**F8. [L1] Per-execution state stores 3 → 2; the command "index" is a view over the registry, not a second map.**
Evidence: `NamespaceExecutionStore` is *already* a pure projection (namespace_execution.rs:35-48 holds no process/PTY/transcript). The spec's "thin per-session `CommandExecution` index" (md:297,335) would be a 2nd `HashMap` keyed by the *same* id the registry uses.
Cut: the registry owns `CommandExecution` (= `InteractiveExecution<CommandTerminalResult>` + cursor/ownership) keyed by `namespace_execution_id`; `write_command_stdin`/`read_command_lines` look up there. No parallel command map.
Saving: state stores 3 → 2; removes one `HashMap` + `Mutex` + insert/lookup/remove. The two survivors are non-overlapping (registry = handles/transcript; observer = serializable timing/status).
Risk/limit: registry stores an operation-typed payload; keep generic via the `Output`-typed entry. No engine→command type leak.

**F9. [L1] Lock acquisitions per exec ~7 → ~3 by deleting the finalizer re-lock dance.**
Evidence: completion takes `CommandCompletionState.inner` (completion.rs:113), then `active()` (finalize.rs:126), `update_active` (finalize.rs:205), `complete_active` (2 locks, process_store.rs:127,142), `complete_namespace_execution` (obs lock) — the begin→active→update→complete walk.
Cut: watcher finalizes inline: registry `complete(id)` (1 lock, live→completed atomically), promise `resolve` (1 condvar), observer `on_terminal` (1 obs lock). Delete `CommandCompletionState`, `begin_terminal_completion`/`mark_active_completion`/`ActiveCompletionRecord`.
Saving: ~7 → ~3 locks; removes the `CommandCompletionState` mutex and the `ActiveCompletionRecord` DTO (7-field copy).
Risk/limit: the obs-store lock stays separate (Phase 4.6 Observer boundary) — 3 is the floor, not 1.

**F10. [L1] One result/outcome type on the op finalize path; delete `ShellOutcome`/`ShellStatus`; both families finalize from `RunResult`.**
Evidence: the runner emits exactly `RunResult { exit_code, payload{success,status} }` for every mode (protocol.rs:37, shell_exec.rs:58-64, runner.rs:43-64). The spec wraps it as `ShellOutcome` for shell (md:209) but hands mount the raw `Value` (md:217) — two shapes for one wire type. `CommandRunnerResult::from_value` already parses `{exit_code,status}` from it (pty.rs:158).
Cut: `ShellOperation::finalize(self, outcome: RunnerOutcome)` and `MountOperation::finalize(self, outcome: RunnerOutcome)` where `RunnerOutcome` is a thin newtype over `RunResult` with `status()`/`exit_code()` accessors. Delete `ShellOutcome`, `ShellStatus`, `parse_report`'s bare-`Value` param, and the `ShellOutcome::from(run_result)` mapping.
Saving: result types on command path 4 → 2; −1 struct, −1 status enum, −1 mapping builder, −1 divergent op input, **−1 distinct finalize signature** (both families identical at completion).
Risk/limit: mount diagnostics — see F11.

**F11. [L1] Give mount `finalize` the `exit_code` (via `RunnerOutcome`) so `MountOverlayOp` keeps its failure message.**
Evidence: live mount failure surfaces `output.status` + `output.stderr` (setns_runner.rs:160-167); `RunResult`/`parse_report(payload)` sees neither.
Cut: `RunnerOutcome` carries `exit_code` (already in `RunResult`); the daemon `MountOverlay` arm puts failure text into `payload` (it owns the `anyhow` context at runner.rs:53-59, ~3-line change). The 2-field `RunResult` then absorbs both families' diagnostics.
Saving: turns a future breaking change to the mount seam into one envelope-internal change now; unifies on one outcome shape (with F10).
Risk/limit: the ~3-line daemon change is inside the "unchanged runner" boundary — flag it in the spec.

**F12. [L1] Two request builders → one; engine authors the request from `(NamespaceTarget, op-args, id)`; drop the dead `env` axis.**
Evidence: two builders (`build_namespace_runner_request` process.rs:291; `ns_runner_request` setns_runner.rs:134). The shell builder always sends `args:{command,cwd}`; the runner reads an `env` key (shell_exec/request.rs:116) the command builder **never sets**; `cwd` is always `"."` (exec_command.rs:200 passes `None`).
Cut: engine builds the request; ops contribute only `args` + `timeout`. Do **not** add `ShellOperation::env()`/`ExecCommand.env` (no live producer — defer to `ShellOp<O>`); drop `cwd` from the command request until a caller sets a non-default. Per-exec authored fields 8 → 2.
Saving: −1 builder, −1 speculative trait method, smaller serialize; wire format unchanged (Phase-4.6-safe).
Risk/limit: keep `timeout_seconds` (read by wait.rs:16); the 6 target fields stay on the wire (the runner reads each).

**F13. [L1] Status: 5 reps/4 conversions → 2 reps/1 conversion; one terminal enum parsed once.**
Evidence: str (shell_exec.rs:68) → `CommandCompletionStatus` String + cancel override (pty.rs:104,119) → `CommandStatus` (finalize.rs:246) → `as_str` (contract.rs:41) AND → `NamespaceExecutionTerminalStatus` (finalize.rs:257) → `as_str` (namespace_execution.rs:78). Spec adds `ShellStatus` (a 6th).
Cut: parse the runner status string once into `NamespaceExecutionTerminalStatus` (the observable enum, variants `Ok/Error/TimedOut/Cancelled`) in `RunnerOutcome::status()`; the cancel override moves to that single parse (cancel is known engine-side from `cancel()`). The command response renders via one `as_str`. "Running" is the promise-unresolved branch (no terminal status), which the yield code already distinguishes via `is_finished()`. Delete `CommandStatus` as a separate enum, `terminal_status`, `namespace_terminal_status`, `ShellStatus`.
Saving: reps 5–6 → 2, conversions 4 → 1; ~40 LOC + 2 enums.
Risk/limit: wire JSON strings stay byte-identical (`ok|error|timed_out|cancelled|running`) — they already match between the two `as_str`s, so the merge is byte-safe.

### L2 — meaningful cuts

**F14. [L2] Delete the `Execution<T>` trait — no polymorphic call site.**
Evidence: both families use concrete types (md:79,275,294,335); `wait_for_command_yield` holds a concrete handle. No `Box<dyn Execution>` anywhere.
Cut: `id/is_finished/wait/wait_timeout` become inherent methods on `NamespaceExecution<T>`; `InteractiveExecution<T>` forwards via its `execution()` accessor.
Saving: −1 public trait, −2 impls.
Risk/limit: re-add (6 lines) only behind a real second consumer.

**F15. [L2] Delete `InteractiveShellOperation` marker — one impl, one entry, hard-codes `Pty`.**
Evidence: only `ExecCommand` impls it; `run_shell_interactive` is the only consumer and already hard-codes `Backing::Pty` (md:261).
Cut: `run_shell<S: ShellOperation>` passes `Pty` itself; the method name carries "interactive."
Saving: −1 trait + its impl.
Risk/limit: reassess at the first `Backing::Pipe` shell producer (deferred `ShellOp<O>`).

**F16. [L2] Delete the `Backing` enum — implicit in which spawn method the engine calls.**
Evidence: `run_shell` always `Pty`, `run_mount` always `Pipe`; the two live spawn shapes are different functions (pty.rs:318 vs setns_runner.rs:194), not a runtime branch.
Cut: launcher exposes `spawn_pty(req) -> (RunnerChild, PtyMaster)` and `spawn_piped(req) -> RunnerChild`. Delete `Backing` + the threaded parameter; `RunnerChild` carries no `Option<PtyMaster>` (the master returns only from `spawn_pty`).
Saving: −1 enum, −1 param, −1 disguised-discriminant `Option` field on `RunnerChild`.
Risk/limit: two return shapes — the natural cost of two disjoint families.

**F17. [L2] Make `NsRunnerInvocation` `pub(crate)`, concrete on the engine; seam tests at the `RunnerChild` boundary.**
Evidence: single production impl; the test-fake + future-server arguments are real but don't need a *public* `Arc<dyn>` on the engine ctor (md:231).
Cut: engine field is a concrete `NsRunnerLauncher`; `pub(crate)` trait kept only as the swap seam (with `spawn_pty`/`spawn_piped` from F16). Tests inject fake `RunnerChild`s.
Saving: removes a public trait + a vtable hop from the engine's public surface; preserves the future-server story crate-internally.
Risk/limit: this is the one legitimate seam — not deleted, just denied public/`dyn` status while it has one impl (the prompt allows the server to return *behind* it).

**F18. [L2] Delete the `MountOperation` trait + 2 impls; mount becomes one engine method + 2 closures.**
Evidence: exactly two mount members, one call site each (setns_runner.rs:43,61); neither participates in the observability ledger (begin is `exec_command`-only). Reviewer 6 confirms they're real (3 daemon arms today) so the engine is justified — but the *trait* is ceremony for n=2 fixed strategies with no central dispatch.
Cut: `run_mount<O>(&self, mode_flag, target, id, parse: impl FnOnce(RunnerOutcome) -> Result<O>) -> Result<NamespaceExecution<O>>`. Call sites pass `("--mount-overlay", …, |_| Ok(()))` and `("--remount-overlay", …, |o| Ok(RemountOverlayResult::from_payload(o.payload())))`. `mount_args` folds into `request.args`.
Saving: −1 trait, −2 impls, the proposed +90 LOC `ops.rs`; `MountOverlayOp::Output=()` and `RemountOverlayOp::Output=RemountOverlayResult` diverge naturally via the closure return.
Risk/limit: promote to a trait only if a 3rd mount-like op or ledger participation appears. `signal_net_ready` stays out (no ns-runner re-exec).
**Conflict note:** this *overrides* Reviewer 6's "keep `MountOperation`." See Conflicts Resolved.

**F19. [L2] Drop the engine `NsRunnerMode` enum — pass the flag literal; reuse the daemon's `NsRunnerOperation`.**
Evidence: the daemon already owns `NsRunnerOperation{Run,MountOverlay,RemountOverlay}` mapped to flags (runner.rs:91-95,122). A spec `NsRunnerMode` is a parallel mode space across the IPC boundary.
Cut: the launcher takes the `&'static str` flag (`["--mount-overlay"]` etc.); shell passes none. The CLI string is the single source of truth. (Pairs with F18's closure form.)
Saving: −1 public enum; no cross-crate coupling to the daemon's private enum.
Risk/limit: rename the daemon's `dispatch_runner_mode` param to match (it's `pub(crate)`, free) — naming only.

**F20. [L2] Delete `ActiveCommandProcess.cancellation` + `CancellationState`.**
Evidence: write-only — `.cancellation` reads (quiesce.rs:116/130/141/157/183, coordinator.rs:62) are all on `CommandRemountQuiesce.cancellation` (the `RemountCancellationToken`); the two writes (write_command_stdin.rs:34, coordinator.rs:67) are never read. Kill semantics come from `cancel_process()`/`runtime.kill`.
Cut: delete the enum + field + 2 writes; the spec's `mark_cancellation_requested()` (md:348) is dropped.
Saving: −1 enum (+`Instant`), −1 field.
Risk/limit: none — `cancel()` does the real work.

**F21. [L2] Delete `ActiveCommandProcess.remount_switch_state`; the coordinator's `CommandRemountQuiesce.switch_state` is the only live copy.**
Evidence: 3 writes (coordinator.rs:69, quiesce.rs:149/194), **zero reads** of the per-command mirror; the read copy is `CommandRemountQuiesce.switch_state` (quiesce.rs:106,166).
Cut: delete the field; the spec's "move into coordinator state" is a **deletion** — no new coordinator-side per-command map (the single `CommandRemountQuiesce.switch_state` suffices).
Saving: −1 field, −3 writes.
Risk/limit: none — write-only.

**F22. [L2] `remount_cancellation` IS read — keep one token on the coordinator keyed by id-set; drop the per-command mirror.**
Evidence: read via `same_token` (quiesce.rs:145-147,187-190) to gate writes/cancel-vs-resume — but always compared to `CommandRemountQuiesce.cancellation`, which the coordinator holds.
Cut: the coordinator owns the `RemountCancellationToken` + affected id-set; drop the per-command `remount_cancellation` field and the `same_token` stamp-back dance.
Saving: −1 field, −4 read + −3 write sites.
Risk/limit: preserve "a stale resume doesn't cancel a command a newer quiesce owns" — the id-set + token-on-coordinator gives the same guarantee.

**F23. [L2] Delete the `CommandFinalizedMetadata`/`CommandPublishFinalization`/`CommandPublishStatus` family — publish is always `None`.**
Evidence: `apply_workspace_completion_policy` returns `Ok(None)` both arms (finalize.rs:113,117); no construction with non-None publish exists; top-level `CompletedCommandRecord.finalized` has **zero reads**.
Cut: delete all 3 types (incl. `Box<PublishReject>`, `LayerStackRevision`, `Vec<PathBuf>`), the `finalized` field on `CompletedCommandRecord` and `FinalizationState::Failed`, and the `finalized`/`finalized_override` params in finalize.rs. Publish returns as the op's `finalize` policy when it actually lands.
Saving: −3 types, −2 fields, simplifies 4 finalize signatures.
Risk/limit: none today — Forbidden-list compliant (deletion, not a shim).

**F24. [L2] Delete dead result fields: `stdout`, `signal`, `timed_out`; don't carry them into the new shapes.**
Evidence: `CommandTerminalResult.stdout` written once (finalize.rs:241), zero reads (yields use the transcript window); `final_stdout()` (process.rs:257) re-reads the whole transcript only to fill it. `CommandProcessExit.signal` written (process.rs:250), zero field reads. `ShellOutcome.timed_out` is `status == "timed_out"` (derivable).
Cut: drop all three; `CommandTerminalResult` = `{status, exit_code, command_total_time_seconds}`.

# Conflicts Resolved

1. **`Execution<T>` trait — Reviewer 4 (delete) vs Reviewer 6 (needs polymorphism for the swap seam).**
   Decision: **delete the trait** (F14); keep `NsRunnerInvocation` as the swap seam (F0b/F17). Rationale: the swap seam is the *launcher* (`NsRunnerInvocation`/`RunnerChild`), not the *handle*. No call site is generic over `NamespaceExecution` vs `InteractiveExecution` — both families hold concrete types (verified: md:79,275,294,335). Reviewer 6's seam concern is satisfied by `RunnerChild::wait_completion()` (F0b), which is independent of the handle trait. The polymorphism that matters is at the spawn boundary, and it stays.

2. **`MountOperation` — Reviewer 4 (delete the trait, use closures) vs Reviewer 6 (keep it; it's the justifying second producer).**
   Decision: **delete the trait, keep the closures** (F18); the engine method `run_mount` IS the second producer that justifies the engine. Rationale: Reviewer 6 is right that mount/remount are *real* (3 daemon arms, runner.rs:42-64) — but "real second producer" justifies the *engine*, not a *trait*. Two fixed ops with one call site each and no shared dispatch table are the textbook case where a trait + 2 impls loses to a method + 2 closures (the spec's own Strategy-vs-enum note, md:122, resolved for n=2 fixed). Reviewer 6's substantive requirement — divergent `Output` (`()` vs `RemountOverlayResult`) and failure diagnostics — is fully met by the closure's return type + F11's `RunnerOutcome.exit_code`. If a 3rd mount op or ledger participation appears, promote then.

3. **`ShellOutcome` — Reviewer 6/Reviewer 4 (delete, pass `RunResult`) vs spec (`ShellOutcome` struct) vs Reviewer 5 (if typed, name `ShellResult`).**
   Decision: **delete `ShellOutcome` and `ShellStatus`; finalize takes `RunnerOutcome` (newtype over `RunResult`)** (F10/F13). Rationale: one wire type (`RunResult`) → one finalize input; the newtype's `status()`/`exit_code()` accessors give shell ops typed access without re-parsing, and give mount ops the `exit_code` they need (F11). This also unifies the two finalize signatures (Reviewer 6 F8) and makes the deferred `ShellOp<O>` additive.

4. **Reviewer 1 ("`coordinator.rs`/`quiesce.rs` branch on lifecycle_state, keep 3 remount variants") vs Reviewers 2/3 ("`CommandLifecycleState` is 100% write-only, delete it").**
   Decision: **Reviewers 2/3 win — delete `CommandLifecycleState` entirely** (F7). Rationale: `rg lifecycle_state` proves every command-view occurrence is an assignment; the remount code branches on the `RemountCancellationToken` and pgid (verified at quiesce.rs:145-157, coordinator.rs:62), never on `active.lifecycle_state`. Reviewer 1's "keep `{Running, QuiescedForRemount, Cancelled}`" is based on a misread; those three variants have zero readers.

5. **Naming rename scope — Reviewer 5 (rename handle → `ExecutionHandle`, store → `NamespaceExecutionLedger`) vs the public-API freeze cost.**
   Decision: **adopt the rule and the renames within the new crate's surface** (F30), since the engine handle is *brand-new* (no back-compat cost — name it `ExecutionHandle<T>` from day one) and the registry is new (`ExecutionRegistry`). For the *existing* public `NamespaceExecutionStore`/`NamespaceExecutionId`: keep `NamespaceExecutionId` (the spec deliberately shares it, md:158) and **prefer** renaming the store to `NamespaceExecutionLedger` but treat it as L3/optional if the crate's public surface is frozen. The load-bearing half — never naming the new handle `NamespaceExecution<T>` and never naming the registry `*ExecutionStore` — is mandatory; the observer rename is desirable. This is the minimal way to honor "one stem per concept" without a churny rename of a frozen public type.

6. **Reviewer 1 (envelope ser/deser stays 2/2, fields irreducible) vs Reviewer 2/3 (drop `cwd`/`env`, drop `timeout` from target).**
   Decision: both hold, no conflict — **wire fields stay** (the runner reads all 6 target fields per F12's note), **authored fields drop to 2** (F12), **`env`/`cwd` are not added** (no producer), **`timeout` leaves `NamespaceTarget` for the op** (F28). Serialize/deserialize stays 2/2 (the envelope is still JSON; no server to amortize against).

  1. validate / resolve-or-create session / admission / remount guard
  2. id = engine.allocate_id()                                  (one id space)
  3. ledger.begin(id, workspace_session_id, "exec_command")     (observer: Starting)

# Recommended Design (one)

A single workspace-agnostic engine over the `ns-runner` re-exec, with **two public traits** (`ShellOperation`, `ExecutionObserver`), **two engine methods** (`run_shell` generic over the shell strategy; `run_mount` taking a mode flag + parse closure), and **one `pub(crate)` launcher seam** (`NsRunnerLauncher::spawn_pty`/`spawn_piped` returning a `RunnerChild` whose `wait_completion()` is the single completion event — fork today, server later). Every execution is one `ExecutionHandle<T>` carrying one `CompletionPromise<T>`; the interactive subtype adds a `PtyMaster` by composition (no trait union, no `Deref`). One `ExecutionRegistry` (live+completed by `NamespaceExecutionId`) is the only per-execution store the engine owns; the command service holds no second map. One `NamespaceExecutionLedger` (today's `NamespaceExecutionStore`, renamed) is a pure `ExecutionObserver` projection — the unchanged Phase 4.6 surface. Completion is a blocking `wait_completion()` on one watcher thread that finalizes inline and resolves the condvar: zero daemon poll loops, zero start-ack handshake, two threads and two pipes per exec. Operations finalize from one `RunnerOutcome(RunResult)`; status is one enum (`NamespaceExecutionTerminalStatus`) parsed once; identity is one id space (`namespace_execution_N`) with the external origin id kept distinct as `origin_request_id`. `CommandProcessStore` and its satellites (`completion`, `finalize`, `launch`, `status_lookup`) are deleted; the command "thin view" is `{transcript cursor, ownership}` on the registry-stored `CommandExecution`, with all write-only state (`CommandLifecycleState`, `CancellationState`, `remount_switch_state`, `CommandFinalizedMetadata` family, `stdout`/`signal`/`timed_out`) deleted, not migrated. Remount coordination owns its own `RemountCancellationToken` + id-set and queries the registry.

### Findings

**1. [L1] Collapse watcher poll-loop B into a blocking `child.wait()` — remove poll loop B + the per-process `try_wait` self-throttle.**
Evidence: `completion.rs:75-84` spins `take_exit()` every 5 ms; `take_exit` → `process.rs:261` `try_wait()`. The watcher has nothing else to do until the child exits.
Cut: watcher does one blocking `child.wait()` (the spec's hop 7), then reads result-fd, finalizes, resolves the condvar promise. Delete `COMPLETION_POLL`-driven loop in the watcher; delete `take_exit`'s `try_wait`/`exit_taken`/`PtyProcessExit::Running` machinery (the watcher is the sole owner now, so the "two pollers can't both finalize" guard at `process.rs:71,229` is dead).
Saving: −1 poll loop, −1 enum variant path (`PtyProcessExit::Running`), −1 mutex (`exit_taken`), removes ~5 ms mean + up to 5 ms tail latency on completion. This is the spec's core promise; confirmed achievable.
Risk/limit: **Cancel-correctness — verified safe.** The daemon-side `ns-runner` child runs `process_group(0)` (`pty.rs:345`); cancel is `killpg(child_pgid)` via `PtyProcess::terminate` (`pty.rs:246-250,486`) issued from the `write_command_stdin` request thread, never from the watcher. A blocking `child.wait()` on thread #3 cannot block a `killpg` on another thread, and the kill makes `wait()` return. So `InteractiveExecution::cancel` stays responsive with a blocking watcher. The spec's hop list (lines 256-272) is correct; make the cancel-vs-blocking-wait invariant explicit in "Finalization / Terminal Semantics" (around line 386).

**2. [L1] Collapse poll loops C and D into the same condvar — the yield path and the completed-record handoff both become condvar waits.**
Evidence: `completion.rs:194-211` (loop C) polls `is_exited()`; `completion.rs:214-241` (loop D, called from `helpers.rs:62`) polls `process_store.completed()` *after* C already saw `Completed`, because the finalizer (thread #4) populates the completed map asynchronously (hops 24-25). That second loop exists **only** because completion and finalization are split across two threads with no handoff.
Cut: with the watcher running `finalize` inline before `promise.resolve()` (spec hops 7), "promise resolved" ⟹ "completed record exists." `wait_for_command_yield`'s Completed branch reads the resolved promise/registry entry directly (spec line 374-375) — **delete `wait_for_completed_record` entirely** and delete loop D. Loop C's `is_exited()` poll becomes `promise.wait_timeout(quiet_or_yield_deadline)`.
Saving: −2 poll loops (C+D), **−1 whole thread (the shared finalizer thread #4)**, removes the watcher→finalizer mpsc hop (hop 24). `completion.rs` (242 LOC) deletion is real; `helpers.rs:62` simplifies. Net daemon-side poll loops: **3 → 0**.
Risk/limit: The quiet-period yield still needs to detect transcript growth (see finding 3); the *completion* edge is now event-driven, but the *quiet-period* edge is a separate concern handled there.

**3. [L2] The quiet-period yield still polls `transcript_len()` (a `stat` per 5 ms) — bound it to the promise wait, don't keep a standalone 5 ms `stat` loop.**
Evidence: `completion.rs:199` calls `process.transcript_len()` → `process.rs:220,316` → `std::fs::metadata(path).len()` every `COMPLETION_POLL`. Even after findings 1-2, a naive port keeps this `stat` loop alive for the quiet-period UX.
Cut: drive the loop off `promise.wait_timeout(remaining)` with a coarse re-check of transcript length only when the timeout slice elapses (the QUIET_MS/yield_time_ms cadence, not 5 ms). The quiet-period semantics ("output stopped growing for QUIET_MS") tolerate a 50 ms granularity since QUIET_MS is itself 50 ms — there is no reason to `stat` at 5 ms.
Saving: ~10× fewer `stat` syscalls during an active command's settle window; keeps the externally observable quiet-period/yield behavior. No new concept.
Risk/limit: must keep `output_len()` as the cursor source; do not regress the "running vs settled" decision. This is the single remaining poll in the lens after findings 1-2, and it is reducible to event-coarse, not fully event-driven (the kernel gives no readiness signal for "transcript file stopped growing"), so I down-rank it to L2 rather than claim a 4th loop deletion.

**4. [L1] The start-ack pipe + its IPC round-trip is removable — ordering can be guaranteed by fd-readiness, not a handshake byte.**
Evidence: start-ack exists as a 3rd pipe (`pty.rs:328,391-396`), a write (`pty.rs:299`), and a daemon-side blocking `read_exact` (`runner.rs:23,184-189`). Its only job: make the runner block until the daemon has finished spawning so the daemon can `drop(request_read)`/`drop(result_write)` and clone the master *before* the child proceeds. But the request payload is itself written *after* spawn (`pty.rs:303` in `allow_start`), and the runner already blocks on `read_payload_from_fd` (`runner.rs:24,202-208`) until that write arrives. The request-fd read is the natural barrier; the start-ack byte is redundant with it.
Cut: delete the start-ack pipe, `--start-ack-fd`, `wait_for_start_ack`/`wait_for_start_ack_reader` (`runner.rs:23,140-147,175-189`), `PendingPtyProcess`/`allow_start`/`RequestPayload` two-phase split (`pty.rs:40-49,291-316`). Spawn writes the request immediately after fork; the runner's existing blocking request read provides the ordering barrier. The daemon's `drop(request_read)`/`drop(result_write)` and `try_clone` happen between fork and the request write, which is already the case (`pty.rs:347-356` run before any payload write would).
Saving: **−1 pipe (2 fds + 2 `fcntl`), −1 IPC round-trip (hop 11/13), −1 daemon helper fn pair, the entire `PendingPtyProcess`/`allow_start` two-phase dance (~40 LOC)**. The `--start-ack-fd` CLI arg, its parse arm (`runner.rs:140-147`), and the `RunnerCliConfig.start_ack_fd` field all go.
Risk/limit: must confirm the daemon completes `drop(request_read)` before the runner could ever see EOF-vs-data on the request pipe. Since the daemon holds `request_write` and only it writes, the runner's `read_to_string` blocks until the daemon writes — there is no race where the runner proceeds early. The current code already relies on request-fd as a barrier (the runner does real work only after `read_payload_from_fd` returns). I am confident this is load-bearing-free; this is the single biggest avoidable IPC round-trip in the lens. If the engine wants to keep a barrier for a future server backing, keep it *behind the `NsRunnerInvocation` seam*, not in the protocol envelope.

**5. [L2] The result-fd reader thread (#2) is removable — read the result-fd inline in the watcher after `child.wait()`.**
Evidence: `pty.rs:359,443-451` spawns a thread that only does `read_to_end` on the result pipe and parks the bytes in an mpsc; `take_exit` later does `recv_timeout` (`process.rs:243,284`). With finding 1's blocking watcher, the result is available exactly when the child exits.
Cut: after the watcher's `child.wait()` returns, the watcher reads the result-fd directly (bounded; the child has closed its write end, so `read_to_end` returns promptly). Delete `spawn_runner_result_reader`, the `runner_result_done` mpsc field (`pty.rs:36,443`), and `take_runner_result`'s recv (`process.rs:284-288`).
Saving: **−1 thread per execution (thread #2)**, −1 mpsc, removes the 500 ms `recv_timeout` grace as a separate hop. Threads per execution: **4 → 2** (output reader + watcher), combined with finding 2's finalizer deletion.
Risk/limit: the result-fd `read_to_end` must not deadlock if the runner exits without writing (error paths). Guard with the fact that `child.wait()` already returned (writer is dead → EOF), so the read terminates. Keep the output-reader thread #1: it must drain the PTY concurrently with execution to avoid the child blocking on a full PTY buffer, so it is genuinely irreducible.

**6. [L2] Per-exec request envelope carries fields the runner does not need per-call on the mount path; and on the shell path the `env` axis is dead.**

Risk/limit: keep `timeout_seconds` (read by `wait.rs:16`). This is internal; no observability impact.

**7. [L2] One id space, allocated once — `CommandSessionId` is a thin newtype face of `NamespaceExecutionId`, not a parallel id with its own counter.**

Saving: **id spaces 2 → 1**, −1 atomic counter, removes the dual-allocate at `exec_command.rs:64,76`. Eliminates the `cmd_N` vs `namespace_execution_N` skew that today makes the two ledgers non-joinable by string.

**8. [L1] Lifecycle enums 4 → 1 on the engine path; `FinalizationState` is pure redundancy with the promise.**

**9. [L2] Lock acquisitions per exec drop from ~7 to ~3 by deleting the finalizer's re-lock dance.**

### 1. [L1] `Execution<T>` trait has no polymorphic call site — delete it; use the two concrete handle types directly

**Evidence.** The spec introduces `pub trait Execution<T>` with `id/is_finished/wait/wait_timeout` and impls it for *both* `NamespaceExecution<T>` and `InteractiveExecution<T>` (docs/namespace-execution.md:170-177). I searched for any consumer that is generic over "either handle." There is none, and the spec's own pseudocode proves it: the command path always holds the concrete `InteractiveExecution<CommandTerminalResult>` (wrapped as `CommandExecution`, lines 294, 335) and the mount path always holds the concrete `NamespaceExecution<M::Output>` and immediately calls `.wait()` (lines 275, 300-301). The two families are explicitly disjoint — "no single trait unions them" (line 79). `wait_for_command_yield`, the one hot consumer, today takes a concrete `&CommandProcess` + `&CommandCompletionPromise` (helpers.rs:29-49) and after the refactor watches one concrete `CommandExecution`; it is never handed a bare `NamespaceExecution`.

**Cut.** Delete the `Execution<T>` trait. Make `id/is_finished/wait/wait_timeout` plain inherent methods on `NamespaceExecution<T>`; `InteractiveExecution<T>` keeps its inherent `execution()` accessor (line 180) and forwards the four methods (4 one-line delegations) or callers reach through `.execution()`.

**Saving.** Removes 1 public trait + 2 trait impls; collapses the public-concept count by one trait. No dynamic dispatch existed to lose (the engine returns concrete types). Removes the temptation of a `Box<dyn Execution<T>>` registry value.

**Risk/limit.** If a future caller genuinely needs "any execution," re-add the trait then — it is a 6-line addition behind a real second consumer. Today it is indirection with a single shape per call site.

Either (a) the method `run_shell_interactive<S: ShellOperation>` simply passes `Backing::Pty` itself (the name already says "interactive"), or (b) add `const BACKING: Backing` to `ShellOperation` if/when a pipe-shell appears. ### 3. [L1] `Backing { Pty, Pipe }` is implied by which method spawned — drop it from the public seam; branch internally

**Evidence.** `enum Backing { Pty, Pipe }` is passed into `NsRunnerInvocation::spawn(request, backing)` (docs/namespace-execution.md:247-251). But `run_shell_interactive` always passes `Pty` (line 261) and `run_mount` always passes `Pipe` (line 275). The caller of `spawn` is the engine, and the engine already knows which family it is in (it is in `run_shell_interactive` vs `run_mount`). Live code confirms the two spawn shapes are genuinely different functions today, not a runtime branch on an enum: `spawn_current_exe_ns_runner` opens a PTY pair + start-ack (pty.rs:318-375) while `run_child` uses `Stdio::null()` pipes + no start-ack (setns_runner.rs:194-232).

**Cut.** Replace the single `spawn(request, backing)` with two methods on `NsRunnerInvocation` — `spawn_pty(request) -> RunnerChild` and `spawn_piped(request) -> RunnerChild` — or keep one trait method per family. Delete the `Backing` enum. The "binary internal" decision becomes "which function the engine called," which is strictly less state.

**Saving.** Removes 1 enum (2 variants) + 1 parameter threaded through the seam. The spec itself frames `Backing` as "internal and not serialized" (line 437); making it *implicit in the call* removes it entirely rather than demoting it.

**Risk/limit.** Two trait methods vs one method + enum is the same surface count if the trait stays; but combined with finding #4 (collapsing the trait to one impl) the enum becomes pure overhead. If you keep `NsRunnerInvocation` as a trait, two methods is still cleaner than one-method-plus-discriminant because the PTY child carries a `PtyMaster` the piped child does not — an enum param forces an `Option<PtyMaster>` in the unified `RunnerChild`, which is finding #6.

### 4. [L1] `NsRunnerInvocation` trait is premature with a single impl — make the engine hold a concrete launcher; gate the future server behind a cfg/feature, not a `dyn`

**Evidence.** Spec: `pub trait NsRunnerInvocation: Send + Sync` with `real = fork ns-runner; fake in tests` (docs/namespace-execution.md:247-250, 233). The justification is explicitly future-tense: "If a persistent runner server returns later, only that seam changes" (lines 51-55, 622-624). The substrate facts confirm the server "is already reverted" and "may only return BEHIND the invocation seam." So today there is exactly one production impl. The test-fake argument is real but does not require a *public* trait object on the engine — the engine's watcher consumes a `RunnerChild` (a blocking `child.wait()` + result-fd read, lines 266-270); a test can inject a fake child far more cheaply than a fake of the whole fork-and-PTY dance.

**Cut.** Two viable collapses, pick one:
- (a) Keep a trait but make it crate-private (`pub(crate)`) and seam the *test* fake at the `RunnerChild` boundary, not the spawn boundary. Engine field becomes a concrete `NsRunnerLauncher`, tests construct the engine with a pre-made fake `RunnerChild` queue.
- (b) Keep the trait but stop calling it a public Bridge "pattern"; it is a test seam. Either way, drop `Arc<dyn NsRunnerInvocation>` from the *public* engine constructor surface.

**Saving.** Removes one *public* trait + the `Arc<dyn ...>` indirection (one heap indirection + vtable on the hot spawn path) from the engine's public API; the future-server story is preserved by the same seam being crate-internal. The Bridge "swap fork↔server" is a real future need but costs nothing to defer to `pub(crate)` until the second impl exists.

**Risk/limit.** This is the one finding where the spec's stated goal (decouple operations from *how* the runner launches) is legitimate. I am not deleting the seam — I am denying it *public* status and a `dyn` hop while it has one impl. If you accept finding #3, the trait has two methods (`spawn_pty`/`spawn_piped`); keep them but `pub(crate)`.

### 5. [L2] `ShellOutcome` duplicates data already in `RunResult.payload` — derive it in `finalize`, don't introduce a struct

**Evidence (FACT).** The runner's `RunResult` is `{ exit_code: i32, payload: Value }` where payload is `{success, status}` (protocol.rs:37-41; shell_exec.rs:58-64). The spec's `ShellOutcome { exit_code: i64, status: ShellStatus, timed_out: bool }` (docs/namespace-execution.md:209) is "built from the runner's `RunResult.payload` plus exit code" (line 221). But `timed_out` is *already* encoded in `status` — the runner sets `status = "timed_out"` precisely when it timed out (shell_exec.rs:55, 68-69), and today's command path recovers the timeout purely from the status string (`"timed_out" => CommandStatus::TimedOut`, finalize.rs:249-254) with no separate boolean. So `ShellOutcome.timed_out` is a redundant axis.

**Cut.** Either (a) pass `RunResult` (or `&RunResult`) straight into `ShellOperation::finalize` and let the op map it — there is exactly one shell op (`ExecCommand`) and its `finalize` already owns the `status.as_str()` match (finalize.rs:246-264) — or (b) if you keep a typed `ShellOutcome`, drop `timed_out` (it is `status == TimedOut`). I recommend (a): one fewer type on the command result path.

**Saving.** Removes 1 struct (`ShellOutcome`) + likely 1 enum (`ShellStatus`, see finding #9) + 1 redundant field. Collapses the result-type count on the command path from {RunResult, ShellOutcome, CommandTerminalResult} toward {RunResult, CommandTerminalResult}.

**Risk/limit.** Passing `RunResult` into `finalize` slightly widens what the op sees (it gets `exit_code: i32` + raw payload). That is acceptable: `finalize` is op-specific policy and already must parse mount payloads (`parse_report(serde_json::Value)`, line 217) — so the mount family already takes raw `Value`. Symmetry argues the shell family should too.

### 6. [L2] `RunnerChild` will carry an `Option<PtyMaster>` (Pipe has none) — split the child type by family instead of optionalizing

**Evidence (FACT + ADVICE).** Live: the PTY child owns a `writer`/reader/`runner_result_done`/`child` (pty.rs:32-38, 361-368) and a PTY master; the piped child owns only pipes + stderr (setns_runner.rs:194-232) and has *no* PTY. If `NsRunnerInvocation::spawn` returns a single `RunnerChild` (docs/namespace-execution.md:249), that struct must hold an `Option<PtyMaster>` (None for mount) — a nullable field that is structurally always-Some for one family and always-None for the other. The spec already routes the PTY only into `InteractiveExecution` (line 168) and never into `NamespaceExecution`, so the master never belongs on the mount path.

**Cut.** Have `spawn_pty` return `(RunnerChild, PtyMaster)` and `spawn_piped` return `RunnerChild` (no master). The engine moves the master into `InteractiveExecution` for shell and discards nothing for mount. `RunnerChild` becomes "the reaped process + result-fd reader" only.

**Saving.** Removes 1 optional field that is a disguised family discriminant; keeps `RunnerChild` single-responsibility (reap + read result). Pairs with finding #3.

**Risk/limit.** Two return shapes instead of one; this is the natural cost of the two families the spec already declares disjoint (line 79). It is less weight than an `Option` that every reader must `expect()`.

### 7. [L2] `MountOperation::operation_name` + the `Output`/`parse_report` generics buy nothing for two fixed, single-call-site ops — express mount as one engine method taking `(NsRunnerMode, args, parse_fn)`

**Evidence (FACT).** The mount family has exactly two members, each with one call site: `mount_overlay` (setns_runner.rs:43-59) and `remount_overlay` (setns_runner.rs:61-89). Their outputs differ: mount returns `()` (it only checks success, setns_runner.rs:157-167), remount returns `RemountOverlayResult::from_payload(&result.payload)` (setns_runner.rs:177-183). Neither needs an `operation_name()` for the engine — the spec puts `operation_name` on `MountOperation` (line 214) but the Observability Contract says **`begin` stays in the operation layer** and the engine "only drives running/terminal by id" (lines 428-432). The *workspace* crate is where these ops live (lines 570-572), and the workspace mount/remount call sites are *not* in the operation layer that owns `begin`. So who calls `operation_name()` for a mount? Nobody on the observable path — the mount path has no `NamespaceExecutionStore.begin` today and the spec doesn't add one for it (the ledger is fed by `exec_command` only; the observer's `on_running/on_terminal` are by id, lines 423-424).

**Cut.** Drop the `MountOperation` trait entirely. Add one engine method:
`run_mount<O>(&self, mode: NsRunnerMode, target, id, parse: impl FnOnce(RunResult) -> Result<O>) -> Result<NamespaceExecution<O>>`.
The two workspace call sites pass `(MountOverlay, target, id, |_| Ok(()))` and `(RemountOverlay, target, id, |r| Ok(RemountOverlayResult::from_payload(&r.payload)))`. `mount_args` (the `{}` vs `{probe_path,...}` JSON, lines 216) folds into the `NamespaceTarget`/request the caller already builds — it is just `request.args`.

**Saving.** Removes 1 trait (`MountOperation`) + its 2 impls (`MountOverlayOp`, `RemountOverlayOp`, the proposed `+90` `ops.rs`, lines 501, 572). The mount path becomes two closures at two call sites — exactly the "one call site each" minimal form. Net concept delta on the mount side: from {MountOperation + 2 impls} to {0 traits, 2 closures}.

**Risk/limit.** If a *third* mount-like op appears, or if mounts later need the `begin` ledger entry with an `operation_name`, promote the closure form to a trait then. With two fixed ops and no ledger participation, the trait + two impls is pure ceremony. This is the spec's own Strategy-vs-enum tradeoff (line 122) resolved correctly for n=2 fixed strategies that never go through a central dispatch.

### 8. [L2] `ShellOperation`'s `cwd`/`env` accessors are dead for the only impl — drop them from the trait

**Saving.** Removes 2 trait methods + 2 always-constant fields from `ExecCommand` (the spec's `cwd`/`env` fields at line 282/290). Shrinks the new `exec.rs` below its `~40` estimate (line 497).

**Risk/limit.** Forward-compat only; defaulted methods cost nothing and keep the door open. The point is they must not be *required* methods that every op stubs with a constant.

### 9. [L3] `ShellStatus` is a fourth status enum mirroring three existing ones — reuse `NamespaceExecutionTerminalStatus`, don't add a new enum

**Risk/limit.** `CommandStatus` also has `Running` (a non-terminal value the others lack); the engine's outcome is always terminal, so `NamespaceExecutionTerminalStatus` is the right reuse. No behavior change — the runner string is the source of truth in both designs.

### 10. [L2] `FinalizeCx<'_>` is introduced but its contents are unspecified and the only `finalize` needs only the workspace handle — pass that concretely, don't add a context type

**Evidence (FACT + ADVICE).** Spec: `finalize(self: Box<Self>, outcome: ShellOutcome, cx: FinalizeCx<'_>)` (docs/namespace-execution.md:205, 288, 550). The only `finalize` body the spec sketches does "destroy one-shot session; apply_workspace_completion_policy" (lines 289-291). **Cut.** Replace `FinalizeCx<'_>` with the one borrow the policy uses — but note that borrow is a *workspace* type, and the engine is workspace-agnostic (lines 138-142). **Saving.** Removes the `FinalizeCx` type entirely and keeps the engine watcher free of any workspace borrow (defends the decoupling goal). One fewer type; preserves "one-shot workspace destroyed" behavior (test plan line 644) at the command layer where the workspace service already lives.

**Risk/limit.** Moves the destroy trigger from "inside the watcher via cx" to "command-layer terminal observer." Behaviorally identical (terminal ⟹ child gone, lines 392-395). If a future op truly needs engine-side context, add a typed cx then; for the single command op it is avoidable.

## Naming Coherence Reviewer

**1. [L0] `NamespaceExecutionStore` is reused for two different stores; the spec's own engine registry collides with the live observability store.**
Evidence (FACT): The live observability store is `operation/src/namespace_execution.rs:17` `pub struct NamespaceExecutionStore` — the projection ledger (active/pending/recent maps). The spec keeps that name for the observer (`docs/namespace-execution.md:126` `ExecutionObserver` → `NamespaceExecutionStore`; line 432 `operation::namespace_execution::NamespaceExecutionStore` implements `ExecutionObserver`). But the spec *also* introduces a second per-execution store — the engine's live+completed registry — and names it `ExecutionRegistry` in prose (lines 129, 229) yet `registry.rs` in the file plan (line 492). So a reader sees two stores of executions (`NamespaceExecutionStore` = ledger, `ExecutionRegistry` = engine source-of-truth) whose names share no stem, while the disposition table (lines 403–411) explicitly says the registry takes over the role `CommandProcessStore` played. Three "store of executions" names for two concepts.

Saving: 3 store-names → 2; removes the "which store?" ambiguity at every call site that touches both (engine ctor at spec line 229 holds `registry`, then immediately an `observer: Arc<dyn ExecutionObserver>` that *is* the other store). Concepts: −1.

**2. [L1] `NamespaceExecution<T>` (engine handle) collides head-on with the `NamespaceExecution*` observability family.**
Evidence (FACT): The operation crate already owns a saturated `NamespaceExecution*` namespace: `NamespaceExecutionId`, `NamespaceExecutionStore`, `NamespaceExecutionRecord`, `NamespaceExecutionLifecycle`, `NamespaceExecutionTerminalStatus`, `RuntimeNamespaceExecutionSnapshot` (`namespace_execution.rs:14,17,35,51,69,104`). The spec adds `NamespaceExecution<T>` as the *generic completion handle* (spec lines 90, 96, 160, 176) — a different concept (an in-flight handle, not a tracking record). A reader seeing `NamespaceExecution` cannot tell whether it is "the observability record family" or "the engine's promise handle." `NamespaceExecutionId` is deliberately shared (spec line 158 moves it down), which is correct and *increases* the confusion for the non-shared `NamespaceExecution<T>`.
Cut (ADVICE): Rename the engine handle to drop "Namespace": the genus handle is `Execution<T>` is the trait (spec line 170) — make the base struct `ExecutionHandle<T>` (or just `Execution<T>` as a struct is taken by the trait, so `ExecutionHandle<T>`). Then `InteractiveExecution<T>` stays (it is unambiguous), and `CommandExecution` stays. The engine crate's prefix becomes `Execution*` (`ExecutionHandle`, `ExecutionRegistry`, `ExecutionObserver`, `Execution<T>` trait), and `Namespace*` stays the observability/runner-protocol prefix (`NamespaceExecutionLedger`, `NamespaceRunnerRequest`, `NamespaceTarget`).
Saving: removes the worst overload in the spec — `NamespaceExecution` would otherwise name both a tracking record family AND a live handle. Concepts disambiguated: the reader's "is this the ledger or the handle?" question disappears. 0 round-trips but high reader-tax reduction.
Risk/limit: `ExecutionHandle<T>` vs trait `Execution<T>` is still close; acceptable because struct-implements-trait is idiomatic (cf. `Future`/`impl Future`). The single rule (below) keeps it legible.

**3. [L1] `NsRunnerMode` (spec) vs `NsRunnerOperation` (live daemon enum) — the spec renames an existing enum without saying so.**
Evidence (FACT): Live daemon enum is `NsRunnerOperation { Run, MountOverlay, RemountOverlay }` (`sandbox-daemon/src/runner.rs:91-95`), selected by `dispatch_runner_mode(mode: NsRunnerOperation, …)` (line 37) — note the *function* says "mode" but the *type* says "Operation." The spec's `MountOperation::mode(&self) -> NsRunnerMode` (lines 215, 251) invents `NsRunnerMode` for the same three-variant concept. So the spec adds a third label ("Mode") on top of the live "Operation" type and "mode" parameter, for one concept. Worse, the spec *also* uses `operation` heavily for the strategy traits (`ShellOperation`, `MountOperation`, `operation_name`), so `NsRunnerOperation` (the runner subcommand selector) and `MountOperation` (the strategy) would be two different "operation" concepts one field apart.
Cut (ADVICE): Pick one. Since the spec's strategy traits own "operation" (`ShellOperation`/`MountOperation`/`operation_name`), rename the runner selector to `NsRunnerMode` *and actually rename the live enum* `NsRunnerOperation` → `NsRunnerMode`, fixing `dispatch_runner_mode`'s parameter to match its name. Do not leave two names live.
Saving: collapses 3 labels (type `NsRunnerOperation`, param `mode`, spec `NsRunnerMode`) → 1 (`NsRunnerMode` everywhere). Removes the `NsRunnerOperation` vs `MountOperation` near-collision. Concepts: the runner-mode selector and the strategy "operation" become visibly distinct (Mode vs Operation).
Risk/limit: `NsRunnerOperation` is `enum`-private to the daemon (`sandbox-daemon/src/runner.rs:90`, no `pub`), so this rename is internal and free.

### The single naming RULE the scheme follows

**Prefix encodes layer; suffix encodes role; one stem per concept.**
- **`Namespace*` (full word)** = cross-crate *domain* nouns that survive in observable or protocol surfaces: `NamespaceRunnerRequest`, `NamespaceTarget`, `NamespaceExecutionId`, `NamespaceExecutionLedger`.
- **`Ns*` (abbrev)** = runner/protocol-adjacent internal selectors: `NsFds`, `NsRunnerMode`, `NsRunnerInvocation`.
- **`Execution*` (no `Namespace`)** = the engine's *live mechanism* — handle/trait/registry/observer: `Execution<T>` (trait), `ExecutionHandle<T>`, `InteractiveExecution<T>`, `ExecutionRegistry`, `ExecutionObserver`.
- **Suffix law:** `*Outcome` = a finished result; `*Verdict` = a poll/yield decision; `*Result` = a typed result DTO; `*Ledger`/`*Registry` = a store (Ledger = projection/observability, Registry = source-of-truth); never two stems (`Store`/`Registry`/`Ledger`) for the same store.
- **Op/handle/API triad is intentional and stays:** `exec_command` (API) → `ExecCommand` (strategy) → `CommandExecution` (handle); the `Execution = live handle` rule makes the third leg self-evident.

### Findings

**1. [L1] `MountOperation::parse_report(serde_json::Value)` is leaky: it drops `exit_code` and `stderr`, which the live mount-failure path actually reads.**
Evidence (FACT):
- Live mount failure surfaces both: `setns_runner.rs:160-167` builds `SetupFailed { step: format!("... status {}: {}", output.status, String::from_utf8_lossy(&output.stderr)) }`. `run_child` returns a full `Output { status, stdout, stderr }` (`setns_runner.rs:227-231`), and `stderr` is piped explicitly (`process.rs:215` → `setns_runner.rs:215 .stderr(Stdio::piped())`).
- The spec's report-only seam is `fn parse_report(report: serde_json::Value) -> Result<Self::Output, _>` (`namespace-execution.md:217`) and `run_mount` "is identical except … `parse_report` over `RunResult.payload`" (`md:274`). `RunResult` is `{ exit_code: i32, payload: Value }` only — **no stderr** (`protocol.rs:37-41`).
- For `MountOverlay`, the daemon synthesizes `ok_result()` = constant `{success:true,status:ok}` on success and `?`-propagates an `anyhow` context string on failure (`runner.rs:53-59`). So on mount failure today the *only* signal is the child's exit status + its stderr text — exactly the two things `parse_report(payload)` cannot see.

Cut: give `MountOperation::finalize` the **same** input the shell side gets — a single outcome struct carrying `{ exit_code, payload }` (i.e. the whole `RunResult`), not just `payload`. Then `RemountOverlayOp` reads `payload`, and `MountOverlayOp` reads `exit_code` for its `SetupFailed`. Do NOT add a stderr field to `RunResult`: instead make the daemon `MountOverlay` arm put the failure text into `payload` (it already owns the `anyhow` context at `runner.rs:53-59`), so the existing 2-field `RunResult` absorbs both families.
Saving: removes a future breaking change to `MountOperation::parse_report` (which would otherwise need an `exit_code`/error param added the first time someone debugs a mount failure); unifies shell+mount finalize on **one** outcome shape (RunResult) instead of two (`ShellOutcome` for shell, raw `Value` for mount). Concretely: 1 trait-method signature change now vs. 1 breaking change later, and −1 divergent input type on the op path.
Risk/limit: requires the daemon `MountOverlay` arm to serialize its failure into `payload`; that is a ~3-line change in `runner.rs`, inside the "unchanged runner" boundary the spec drew — but it is the minimum to keep mount diagnostics. Flag it explicitly in the spec rather than discovering it in phase 4.

**2. [L1] `ShellOutcome` and `parse_report`'s `Value` are the SAME information (`RunResult`) split into two shapes — collapse to one `RunnerOutcome` both families finalize from.**
Evidence (FACT): The runner emits exactly one thing for every mode: `RunResult { exit_code, payload }` (`protocol.rs:37`, `shell_exec.rs:58-64`, `setns.rs:343`, `runner.rs:43-64`). The spec then re-wraps that as `ShellOutcome { exit_code, status, timed_out }` (`md:209`) for shell, but hands mount the raw `Value` (`md:217`). Two finalize input shapes for one wire type.
Cut: delete `ShellOutcome` and the bare-`Value` `parse_report` param; give both `ShellOperation::finalize` and `MountOperation::finalize` the wire type `RunResult` (or a thin newtype `RunnerOutcome(RunResult)` with `status()`/`timed_out()` accessors so shell ops don't re-parse). One outcome type, two finalizers.
Saving: −1 result type (`ShellOutcome`) on the command path, against the prompt target "minimize result/outcome types"; the spec's families then differ only in `mode()`/`mount_args()` vs `command()`/`cwd()`/`env()`, which is the genuine variation. `timed_out` is derivable from `payload.status == "timed_out"` (`shell_exec.rs:68-75`), so it is not even a needed field — see finding 6.
Risk/limit: `RunResult` lives in `namespace-process`; the engine crate already depends on it (`md:138-139`), so no new dependency. Clean.

**3. [L2] `ExecCommand::Output = CommandTerminalResult` keeps a dead `stdout: String` field that the spec's own "no stdout capture" decision proves is unused.**

Minimal shape: `RunnerChild` exposes `fn wait_completion(&self) -> io::Result<RunResult>` (fork impl = `child.wait()` then read result-fd; server impl = block on the Exited frame). Risk/limit: confirm no current mount path needs the static-fn property; live code shows `from_payload` is already a free function over `&Value` (`result.rs:19`), trivially expressible as `finalize(self, outcome)`.

### Next-op cost check (sketch: `workspace_probe`, a read-only shell op that parses stdout)

ADVICE, grounded in the traits above. A `workspace_probe` op that runs a command and parses its stdout would need, end to end:

2. `impl ShellOperation` with `operation_name/command/cwd/env/timeout_seconds` (5 trivial getters) + `finalize` parsing output — ~25 LOC.

For any stdout-parsing shell op it does NOT hold — it costs a protocol change. This is acceptable *if the spec stops claiming the engine makes the next op universally cheap* and instead scopes the claim to "next op with no new output channel." Minimal spec edit: in the LOC table footnote (`md:531-533`) and the `ShellOp<O>` future bullet (`md:617`), state plainly that stdout-parsing ops require the one-time `RunResult.payload` capture extension, so the cheapness claim is honest.

If finding 2 is taken (finalize gets the whole `RunResult`), the stdout-capture extension is purely additive (`payload` gains a `stdout` key) and needs **no** signature change to `finalize` — making the eventual `ShellOp<O>` truly ~30-80 LOC. That is the payoff of unifying the outcome type now.
