architecture

One substrate, multiple user experiences

The future roadmap should avoid parallel runtimes. Each surface should ride the same engine, sandbox, stores, and event stream.

Engine loop: the direct single agent and the TaskCenter agents should both use the same run lifecycle and StreamEvent model. Product surfaces consume events; they do not own execution policy.
roadmap

Product bets after the foundation

Filter by planning horizon. The first two are built foundations; the rest turn runtime primitives into product value.

DoneFoundation

TaskCenter coding collaboration

Complex work becomes a planned graph with generator tasks, verification, evaluation, retries, and continuation.

  • Expose graph health in future UIs.
  • Use as escalation from direct agent.
  • Keep lifecycle as the durable source of truth.
DoneIntegrity

Sandbox operation correctness

Read, write, edit, shell, LSP, and future tools operate through a guarded sandbox boundary.

  • Show OCC and layer events to users.
  • Make conflicts explainable.
  • Keep provider portability behind adapters.
NextAgent

Single general agent

A fast everyday coding agent that uses the same tools and sandbox, then delegates to TaskCenter when decomposition is needed.

  • Persistent `GeneralAgentSession`.
  • Direct-to-handoff context bridge.
  • Mode controls: ask, edit, review, plan.
NextMobile

Mobile control plane

A phone-first cockpit for watching runs, approving actions, inspecting failures, and nudging long-running work.

  • Read-only mission and run feed first.
  • Live stream and approval actions second.
  • Sandbox timeline and push alerts third.
NextAPI

Agent control plane API

A stable HTTP and event-stream surface that lets CLI, web, mobile, and automation share one runtime contract.

  • Start, pause, resume, cancel runs.
  • Stream typed events.
  • Query stores and audit artifacts.
LaterReplay

Replay and eval studio

Turn prompts, messages, tool calls, sandbox events, and performance data into searchable run replays.

  • Prompt and message inspection.
  • Failure triage from first concrete issue.
  • Scenario dashboards for quality trends.
LaterPlugins

Plugin marketplace

Use the LSP catalog as the first plugin blueprint, then add test, browser, database, cloud, and docs catalogs.

  • One file per tool.
  • Shared runtime/session logic.
  • Sandbox-installed sidecars where needed.
LaterMemory

Project memory graph

Promote durable summaries, decisions, APIs, failing tests, and constraints into retrievable project context.

  • Summaries as typed evidence.
  • Design decisions as reusable context.
  • Staleness and provenance visible.
LaterGovernance

Policy, spend, and permissions

Make model/tool budgets, approval gates, forbidden paths, and team policies first-class runtime controls.

  • Per-agent and per-project budgets.
  • Approval queue for risky tools.
  • Audit-friendly policy history.

readiness

Where the platform is strong now

The next product work should preserve the hard-won runtime boundaries instead of building another agent stack.

Execution substrate

Engine loop, terminal tools, sandbox operations, and plugins are already coherent.

Collaboration model

TaskCenter has the graph, retry, evaluator, and nested-mission model.

Product API

The CLI and runner paths exist, but a stable server/control-plane API is still missing.

Mobile UX

There is a strong concept prototype; production needs API, auth, streaming, and actions.

strategic invariant

Do not split the runtime.

The roadmap gets powerful when every interface uses the same engine, sandbox, TaskCenter stores, context packets, and audit artifacts. General agent, mobile, replay, and plugins should be product lenses over one execution substrate.
