Tools Architecture Wiki

This wiki maps every immediate tool folder to the engine dispatch loop, sandbox workspace routing, TaskCenter terminal state, and background execution behavior. It is a documentation artifact only; the runtime code was not refactored as part of this pass.

Scope And Ownership

Tool surface

backend/src/tools owns the agent-facing tool definitions, factories, terminal submissions, helper launches, background management tools, and sandbox wrappers.

Engine bridge

engine.query.loop, engine.tool_call.streaming, and engine.tool_call.dispatch decide when tool calls execute, defer, run in background, or terminate the agent loop.

Sandbox bridge

Sandbox wrappers call typed sandbox.api.tool requests, then daemon routing chooses layer-stack/OCC, per-call ephemeral overlay, or explicit isolated workspace execution.

TaskCenter bridge

Submission and terminal tools are the persistent handoff channel between agents and TaskCenter state: planner rows, generator DAG tasks, delegated goals, evaluator outcomes, and helper outputs.

End-To-End Tool Workflow

1
Agent registry is built.

engine.agent.factory asks the tools facade and factory for allowed tool instances. Tool metadata comes from BaseTool: intent, background policy, task type, terminal flag, hooks, and context requirements.

2
Provider request exposes schemas.

The query request converts registered tools to API schemas. Optional non-terminal background tools receive an LLM-facing background input; terminal tools remain exclusive.

3
Streaming and final-message dispatch are coordinated.

Mid-stream execution is allowed only when deferral rules permit it. If terminal tools are present, the query loop defers all tool calls until the full assistant message can be validated.

4
One tool lifecycle runs.

execute_tool_once parses input, runs pre-hooks, executes the body, validates output, runs post-hooks, and stamps successful terminal tools with does_terminate=True.

5
Result ownership branches.

Terminal results stop the agent loop. Background launches create bg_N records. Sandbox tools cross into daemon workspace routing. Submission tools mutate TaskCenter state or return helper metadata.

Critical Routes To Keep Straight

Terminal tools

A successful terminal call becomes does_terminate=True, wins the query-loop stop condition, and must not be batched with sibling tool calls.

Background tools

run_subagent is always background. shell is optional background. Completion is delivered by check_background_task_result or wait_background_tasks in current code.

Ephemeral workspace

Without an isolated handle, shell/search and non-fast-path operations use a per-call overlay. Write-capable captures publish through OCC when they should affect the shared workspace.

Isolated workspace

enter_isolated_workspace is explicit, persistent for the session, and scratch-only. Writes are audited as changed paths but discarded on exit instead of published through OCC.
