6. Daemon and Background Tasks §

  • The daemon is the resident in-sandbox control plane: it owns RPC dispatch, cached LayerStack/OCC services, workspace routing, in-flight invocation tracking, and isolated-workspace lifecycle calls.
  • Host code calls one public transport; the host client chooses Docker TCP when available or the AF_UNIX thin-client path through provider exec.
  • Each request carries an op, invocation_id, and JSON object args; the dispatcher validates it, registers it for cancellation/heartbeat, dispatches the handler, attaches runtime timings, and deregisters it.
  • Engine background tasks are host-managed, but sandbox-bound background work is correlated to daemon invocations so cancellation, heartbeat, TTL reaping, and isolated-workspace gating agree.

6.1 Control Plane Shape §

The daemon is not the storage layer and it is not a provider adapter. It is the warm process inside the sandbox that receives one JSON envelope at a time, reuses expensive runtime state, and routes each operation to the right workspace implementation. Host-side code keeps the provider boundary in [[Provider]]; daemon-side code keeps workspace truth in [[LayerStack]] and [[OCC]].

6.2 Request Lifecycle §

A daemon request has two identities. The host transport always puts a wire-level invocation_id on the envelope; public sandbox requests also carry caller fields so daemon-side code can recover agent_id for routing, background counts, plugin gating, and audit correlation backend/src/sandbox/api/tool/_daemon_requests.py:8-14, backend/src/sandbox/api/transport.py:24-40.

  1. The public API builds a typed payload: file APIs add path/content/edit fields; shell adds command, cwd, timeout, description, and background=true when launched through the background path backend/src/sandbox/api/tool/shell.py:49-63.
  2. DaemonSandboxTransport attaches the daemon protocol version and calls call_daemon_api(); the host client injects layer_stack_root, chooses a transport path, and serializes {op, invocation_id, args} backend/src/sandbox/api/transport.py:44-77, backend/src/sandbox/host/daemon_client.py:97-183.
  3. Docker sandboxes can use a cached host TCP endpoint; otherwise the host executes the bundled thin client inside the sandbox, which connects to the AF_UNIX socket, sends one newline-terminated JSON envelope, and returns stdout JSON backend/src/sandbox/host/daemon_client.py:436-457, backend/src/sandbox/daemon/scripts/thin_client.py:13-55.
  4. On connection failure or selected empty-response cases, the host runs the launch script, probes api.runtime.ready, then retries the original envelope with bounded connect backoff backend/src/sandbox/host/daemon_client.py:281-394, backend/src/sandbox/host/daemon_client.py:397-433.
  5. The daemon server enforces request size and read-timeout bounds, validates JSON object shape, checks TCP auth when configured, and delegates to dispatch_envelope_async() backend/src/sandbox/daemon/rpc/server.py:54-62, backend/src/sandbox/daemon/rpc/server.py:82-152.
  6. The dispatcher validates op and args, supplies a missing invocation_id only as a fallback, derives agent_id, registers the current asyncio task in the in-flight registry, runs the handler from OP_TABLE, serializes the result, attaches runtime timings, and deregisters in finally backend/src/sandbox/daemon/rpc/dispatcher.py:55-113, backend/src/sandbox/daemon/rpc/dispatcher.py:116-176.
  7. The host decodes the daemon response. Non-null daemon error values raise transport-level failures unless the response is a typed guarded-result error, so expected policy conflicts remain normal tool results backend/src/sandbox/host/daemon_client.py:128-149.
Invariant The daemon may reuse caches, but it must not hide publish semantics. Direct file fast paths still go through OCC; shell and search operations still run through the workspace pipeline; isolated writes remain private scratch until explicit exit.

6.3 Operation Surface §

The operation table is registered at import time. Duplicate registration of the same handler is a no-op, while a different handler for an existing op raises immediately, so runtime extension remains explicit instead of silently overriding commands backend/src/sandbox/daemon/rpc/dispatcher.py:37-53. 6.4 Workspace Routing and State §

The key daemon router is dispatch_workspace_tool_call(). It creates a ToolCallRequest with invocation id, agent id, verb, intent, args, and background flag. If the agent has an open isolated handle, the call runs inside [[isolated_workspace]]. Otherwise single-file read_file, write_file, and edit_file use direct LayerStack/OCC fast paths when the path can bind cleanly. Everything else goes through the [[ephemeral_workspace]] overlay pipeline backend/src/sandbox/daemon/workspace_tool_dispatch.py:27-71.

6.5 Background Task Lifecycle §

Background execution is split across two control planes. The engine owns user-visible task aliases, result delivery, progress snapshots, and task status. The daemon owns the sandbox invocation that may still be running inside the sandbox. The bridge between them is the sandbox invocation id passed in execution metadata and then into the daemon envelope.

  1. The tool execution layer sees background=true for an optional background tool such as shell, validates the tool input before spawning, and allocates a short alias such as bg_1 backend/src/engine/background/dispatch.py:59-106.
  2. If the tool requires sandbox context, dispatch creates a daemon sandbox_invocation_id and carries it in ExecutionMetadata; the shell tool copies it into ShellRequest.invocation_id and marks background from the background task id backend/src/engine/background/dispatch.py:107-141, backend/src/tools/sandbox/shell/shell.py:169-180.
  3. BackgroundTaskSupervisor.launch() stores task id, tool input, asyncio task, agent id, sandbox id, sandbox invocation id, and uses_sandbox; sandbox-bound tasks start the heartbeat loop backend/src/engine/background/task_supervisor.py:67-155, backend/src/engine/background/task_supervisor.py:195-203.
  4. When the request reaches the daemon, the dispatcher registers the same invocation id with agent_id, op name, and background=true backend/src/sandbox/daemon/rpc/dispatcher.py:69-83.
  5. The task runs through normal daemon workspace routing: background shell is still an ordinary sandbox tool call that may publish through OCC unless it is inside isolated workspace.
  6. Result tools read the engine manager state. check_background_task_result returns current or terminal output; wait_background_tasks blocks up to a bounded timeout and marks newly completed tasks delivered backend/src/tools/background/check_background_task_result/check_background_task_result.py:78-128, backend/src/tools/background/wait_background_tasks/wait_background_tasks.py:47-123.

6.6 Cancellation and Heartbeats §

Cancellation is intentionally two-stage for sandbox-bound background tasks. The supervisor first asks the daemon to cancel the sandbox invocation, then it updates the local task state and cancels the host asyncio task when the task type permits it backend/src/engine/background/task_supervisor.py:281-311, backend/src/engine/background/task_supervisor.py:390-404. The daemon control handler cancels the registered task and waits briefly for cleanup, reporting whether the task was found and whether cleanup completed backend/src/sandbox/daemon/builtin_operations.py:56-72.

  1. Manager heartbeat groups running sandbox invocation ids by sandbox id every EOS_BACKGROUND_HEARTBEAT_INTERVAL_S seconds.
  2. It calls sandbox_api.heartbeat() for each sandbox, which maps to api.v1.heartbeat through daemon_invocations.heartbeat() backend/src/engine/background/task_supervisor.py:417-454, backend/src/sandbox/api/daemon_invocations.py:34-48.
  3. The daemon registry refreshes last_seen for invocation ids it still tracks and returns a touched count backend/src/sandbox/daemon/rpc/in_flight.py:87-96.
  4. If heartbeats stop, the daemon TTL reaper cancels stale background tasks, marks them ttl_reaped, and increments ttl_reaped_total without deleting the registry entry until normal deregistration backend/src/sandbox/daemon/rpc/in_flight.py:111-149.

The registry only counts entries that are both background and not done for an agent backend/src/sandbox/daemon/rpc/in_flight.py:98-109. That count is the daemon-side half of the isolated-workspace gate described in [[Isolated Background Gate]].

6.7 Isolated Background Gate §

Isolated vs ephemeral background behavior Ephemeral background shell tasks are normal sandbox-bound daemon invocations with heartbeats and TTL cleanup. Isolated enter is blocked while those tasks are still running for the same agent. Isolated exit cancels sandbox-bound background work before tearing down the namespace.

The host-side isolated lifecycle checks both local manager state and daemon-visible in-flight state before entering. A nonzero count returns ephemeral_jobs_in_flight, because an active background task may still write through the default ephemeral/OCC path and should not race an isolated namespace transition backend/src/sandbox/host/isolated_workspace_lifecycle.py:41-77, backend/src/sandbox/host/isolated_workspace_lifecycle.py:244-275. On exit, the host lifecycle cancels per-agent sandbox-bound background tasks, then calls daemon isolated exit and includes the eviction count in the returned phases backend/src/sandbox/host/isolated_workspace_lifecycle.py:98-136, backend/src/sandbox/host/isolated_workspace_lifecycle.py:253-264.
