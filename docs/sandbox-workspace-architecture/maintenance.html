11. Wiki Maintenance §

  • This page is the support layer for the sandbox workspace wiki.
  • It records page conventions, evidence rules, graph edge vocabulary, refresh triggers, validation commands, and change history.
  • The source tree remains the authority. Wiki pages should explain the current code path and name the files that prove each claim.
Page Conventions Evidence Policy Graph Vocabulary Refresh Checklist Change Log

11.1 Page Conventions §

Brief

Start with a short user-facing verdict: what this node owns, why it exists, and when to read it.

Workflow

Use sequences for state transitions and diagrams for cross-module boundaries. Every step should name the next owner.

Technical Dive

Record entry points, adjacent files, imports, call paths, state ownership, config/artifact surfaces, tests, and failure modes.

Maintenance Memory

End pages with stale triggers and related links so future edits update the smallest correct node.

11.2 Evidence Policy §

Source codeUse path:line references for concrete files and symbols. Do not claim a workflow step unless the current checkout exposes the path.
TestsName focused tests for behavior and live suites for runtime guarantees. Prefer narrow tests when a page explains one module boundary.
ArtifactsUse live artifact locations when behavior is proved by runs: .sweevo_runs/scenario_logs, performance reports, daemon JSONL audit files, and generated reports.
Open questionsIf the code does not prove a claim, keep it out of the page or label it as a refresh question.

11.3 Graph Vocabulary §

imports
Module A depends on module B.
exports
Module A exposes a symbol or registered operation.
calls
Function A invokes function B in the request path.
registers
A module adds a handler, tool, plugin op, or route.
owns-state
A module owns a cache, lock, lease, manager map, file, process, namespace, or lifecycle.
reads-config
A module consumes an env/config key or provider option.
emits-artifact
A code path writes a log, JSONL event, performance report, or scenario artifact.
tested-by
A behavior is covered by a focused test, live scenario, or report.
