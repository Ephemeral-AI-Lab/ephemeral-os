## Progress Tracker

| Phase | Status | Goal | Exit Gate |
|---|---:|---|---|

| 1. Aggressive Cleanup | Not started | Remove dynamic-plugin behavior, stale tests, and fake LSP setup paths that conflict with the new model. | No live/e2e path depends on user-uploaded LSP manifests, random setup commands, legacy `plugin.lsp.*`, or `sandbox.plugin.lsp.*`. |

## Goal

Replace the current dynamic plugin platform with a smaller, fixed plugin family
that starts with one first-party provider: `pyright_lsp`.

The new model keeps the product category as `plugin`, but removes dynamic plugin
registration as the source of operation names and behavior. Initial public ops
are static:

```text
sandbox.plugin.pyright_lsp.query_symbols
sandbox.plugin.pyright_lsp.definition
sandbox.plugin.pyright_lsp.references
sandbox.plugin.pyright_lsp.diagnostics
```

`sandbox.plugin.node_lsp.*` is intentionally reserved for a later provider. Do not use
generic `sandbox.plugin.lsp.*`; that namespace is retired by this migration. Also
remove the current dynamic-plugin spelling `plugin.lsp.*`; in this repo the old
LSP route is dynamic `plugin.<id>.<op>`, not a static `sandbox.plugin.*` op.

## Non-Goals

- No arbitrary end-user plugin package upload.
- No public `sandbox.plugin.ensure` path for arbitrary user manifests or staged
  packages.
- No user-provided setup commands.
- No manifest-defined operation names.
- No manifest-defined `PluginOperationIntent`.
- No write-capable LSP ops in the first migration.
- No `rename`, `code_action`, or edit application until the write policy is
  designed through normal OCC paths.

## Current Problems To Remove

The current plugin model makes too many runtime decisions from user-provided
manifest data:

- Dynamic plugin operation names are registered at runtime instead of being
  present in the static catalog.
- The gateway cannot know dynamic op mutability, so dynamic plugin ops are
  conservatively treated as mutating.
- `PluginOperationIntent` combines product policy and dispatch routing.
- The same framework supports read-only services, oneshot overlay writes, and
  self-managed OCC callbacks.
- The LSP e2e currently stages a fake package and fake pyright assets, but it is
  expressed through the same generic upload/setup/manifest path as arbitrary
  plugins.
- The current legacy LSP op spelling is `plugin.lsp.query_symbols`, generated by
  dynamic `plugin.<id>.<op>` dispatch after a catalog miss.

The replacement should keep the useful service runtime ideas, but remove dynamic
registration and random user behavior from the LSP path.

## Op Freshness Contract

Every `sandbox.plugin.pyright_lsp.*` op must first ensure that the service is current
enough for the operation.

```rust
enum LspFreshness {
    SnapshotOnly,
    AnalyzerReflected,
}
```

Initial map:

| Op | Freshness | Notes |
|---|---|---|
| `sandbox.plugin.pyright_lsp.query_symbols` | `AnalyzerReflected` | Start file-scoped if possible, but still prove pyright has observed the latest target file. |
| `sandbox.plugin.pyright_lsp.definition` | `AnalyzerReflected` | Requires pyright to reflect code updates. |
| `sandbox.plugin.pyright_lsp.references` | `AnalyzerReflected` | Requires index/type graph freshness. |
| `sandbox.plugin.pyright_lsp.diagnostics` | `AnalyzerReflected` | Must wait for current diagnostics or return a bounded stale/timeout status. |

There are two freshness levels:

```text
SnapshotOnly:
  The filesystem projection visible to pyright_lsp is the latest LayerStack
  snapshot, and any target file used by an LSP request has been opened or
  explicitly synced in pyright for that manifest. SnapshotOnly is reserved for a
  later optimization; the first migration treats every Pyright-backed op as
  AnalyzerReflected.

AnalyzerReflected:
  The projection is current, pyright has been notified of changed files, and
  pyright has reached a bounded ready/reflected state for the requested op.
```

## LayerStack Projection Requirements

Real pyright cannot run against the old fake AST fixture. It needs a stable
workspace path backed by the active LayerStack view.

The provider must support:

- materializing or mounting the active LayerStack snapshot into the pyright
  workspace view;
- tracking the manifest key visible to the service;
- detecting when the active LayerStack manifest key changes;
- updating the projection on refresh;
- notifying pyright about changed files, preferably using LSP file watcher
  events;
- falling back to restart/remount when pyright cannot reliably reflect the
  update;
- bounding refresh and analysis wait time.

The old generic refresh flow has useful concepts: snapshot acquisition, service
manifest key, health probes, and failed-service teardown. Reuse those concepts
only where they stay simpler than the old generic manifest router.

## Phase 0. Decision Lock

### Actions

- Freeze the public provider name as `pyright_lsp`.
- Freeze initial public op names:
  - `sandbox.plugin.pyright_lsp.query_symbols`
  - `sandbox.plugin.pyright_lsp.definition`
  - `sandbox.plugin.pyright_lsp.references`
  - `sandbox.plugin.pyright_lsp.diagnostics`
- Reserve `sandbox.plugin.node_lsp.*` for a future provider.
- Decide config location:
  - preferred: `daemon.plugin.enabled_plugins`;
  - alternate: new top-level `sandbox.enabled_plugins` if product wording needs
    to be sandbox-level.
- Decide provisioning mode:
  - pyright assets are trusted first-party assets;
  - no arbitrary user package upload;
  - if package upload remains mechanically useful, it must be a daemon-owned
    first-party provisioning step, not a public user upload API.

### Exit Gate

- Spec and implementation tickets use `sandbox.plugin.pyright_lsp.*`, never
  `sandbox.plugin.lsp.*` or legacy dynamic `plugin.lsp.*`.
- The first migration scope excludes write LSP ops.

## Phase 1. Aggressive Cleanup

### Remove Or Retire

- Dynamic LSP namespace:
  - remove all legacy `plugin.lsp.*` test usage, especially
    `plugin.lsp.query_symbols`;
  - remove all `sandbox.plugin.lsp.*` test usage;
  - remove docs that describe LSP as a dynamic `plugin_id = "lsp"` manifest.
- Public dynamic provisioning surface:
  - remove `sandbox.plugin.ensure` from the public catalog/API, or demote it to a
    daemon-internal first-party provisioning path;
  - remove user-supplied `PluginEnsureInput.manifest`;
  - remove user-supplied `PluginPackageInput.staged_package_root` and package
    upload roots from public request input;
  - remove public manifest setup command handling through `PluginManifest.setup`;
  - remove public manifest operation registration through
    `PluginManifest.operations`;
  - remove user-controlled `PluginOperationIntent`;
  - remove or quarantine `DYNAMIC_PLUGIN_POLICY`, gateway `plugin.*` forwarding,
    daemon unknown-op dynamic fallback, and `dispatch_registered_op`.
- User-uploaded LSP package path:
  - staged package upload for LSP;
  - `.package-sha256` LSP e2e fixture;
  - LSP-specific `setup.sh` fixture;
  - fake node shim fixture;
  - fake `langserver.index.js` fixture;
  - fake Python AST PPC service fixture.
- Generic-plugin tests that exist only to preserve dynamic platform behavior:
  - manifest-defined op dispatch tests;
  - random setup-command success/failure tests;
  - package reload tests for arbitrary plugin ids;
  - dynamic dispatch race tests for arbitrary plugin ops;
  - oneshot overlay write plugin tests;
  - self-managed OCC callback plugin tests;
  - intent-contract tests for manifest `PluginOperationIntent`.

### Keep Temporarily If Needed

- Process lifecycle helpers if reused by `pyright_lsp`.
- PPC/JSON framing if it remains simpler than writing a new JSON-RPC transport.
- Shared process supervision, teardown, and bounded response-size helpers.
- Health probe concepts.
- Snapshot/refresh lock concepts.

These helpers must not keep the public generic plugin API alive. Keeping a helper
is allowed only when the final provider is still statically registered and
dispatched through `BuiltinOp`/`OpRequest`.

### Exit Gate

- No live e2e depends on staged LSP package upload.
- No test calls `plugin.lsp.query_symbols` or
  `sandbox.plugin.lsp.query_symbols`.
- No LSP path uses a manifest-provided service command.
- No public API accepts a user manifest, user setup command, staged plugin
  package, or user-controlled `PluginOperationIntent`.
- No first-party plugin op reaches daemon execution through gateway `plugin.*`
  forwarding or daemon unknown-op dynamic fallback.
- Remaining old generic plugin tests are explicitly marked as temporary or
  deleted.

## Phase 2. Implement New Static Plugin Module

### Protocol And Typed Request Changes

- Add static catalog entries:
  - `sandbox.plugin.list`
  - `sandbox.plugin.health`
  - `sandbox.plugin.pyright_lsp.query_symbols`
  - `sandbox.plugin.pyright_lsp.definition`
  - `sandbox.plugin.pyright_lsp.references`
  - `sandbox.plugin.pyright_lsp.diagnostics`
- Add typed `OpRequest` variants for each.
- Remove LSP dependence on unknown-op dynamic fallback.
- Remove or quarantine the global dynamic `plugin.*` fallback and
  `DYNAMIC_PLUGIN_POLICY`; first-party plugin ops must resolve through static
  catalog entries.
- Mark initial `sandbox.plugin.pyright_lsp.*` ops as read-only in the catalog.

### Config

Add plugin enablement:

```yaml
daemon:
  plugin:
    ppc_root: /eos/plugin/ppc
    ppc_timeout_ms: 5000
    service_probe_timeout_ms: 5000
    max_response_bytes: 8388608
    enabled_plugins:
      - pyright_lsp
```

Validation rules:

- unknown plugin ids are invalid;
- duplicate plugin ids are invalid;
- `pyright_lsp` is disabled unless explicitly present or unless product decides
  it is enabled by default;
- disabling `pyright_lsp` makes `sandbox.plugin.pyright_lsp.*` return a typed
  `plugin_disabled` rejection.

### Runtime Model

Create a static plugin registry:

```rust
enum BuiltinPluginProvider {
    PyrightLsp,
}

struct PluginRegistry {
    enabled: BTreeSet<BuiltinPluginProvider>,
    pyright_lsp: Option<PyrightLspRuntime>,
}
```

The registry owns:

- enabled provider list;
- lifecycle state;
- cheap list state;
- active health probe routing.

### Daemon Adapter Layout

```text
crates/daemon/src/op_adapter/plugin/
  mod.rs
  admin.rs
  pyright_lsp.rs
```

`admin.rs` owns:

- `sandbox.plugin.list`;
- `sandbox.plugin.health`;
- compatibility shaping for `sandbox.plugin.status`, if kept as an internal or
  operator-only view. It must not keep public dynamic plugin provisioning alive.

`pyright_lsp.rs` owns only static LSP op adapters and response shaping.

### Exit Gate

- `sandbox.plugin.pyright_lsp.*` reaches daemon dispatch through `BuiltinOp` and
  `OpRequest`.
- `sandbox.plugin.list` reports configured provider state without probing.
- `sandbox.plugin.health` actively probes enabled services.
- `sandbox.plugin.ensure` is gone from the public surface, or explicitly
  internal-only with no user manifest/setup/upload fields.
- Dynamic plugin fallback is not used by any first-party plugin provider.

## Phase 3. Implement `pyright_lsp`

### Provider Setup

`pyright_lsp` must use real pyright. The first implementation should support
one of these trusted provisioning modes:

1. pyright and node are already present in the sandbox image; or
2. daemon installs first-party pyright assets into a daemon-owned runtime root;
   this is not exposed as user-provided setup.

Provider config should include enough paths to make this explicit:

```yaml
daemon:
  plugin:
    pyright_lsp:
      node_path: /eos/runtime/plugins/pyright_lsp/node/bin/node
      pyright_langserver_path: /eos/runtime/plugins/pyright_lsp/node/lib/node_modules/pyright/langserver.index.js
      workspace_root: /testbed
      analysis_timeout_ms: 10000
      refresh_timeout_ms: 5000
```

Exact paths can change during implementation, but the contract must distinguish
trusted provider assets from user-uploaded plugin packages.

Health output must include enough proof to prevent fake-asset substitutions:

- provider id and enabled/disabled state;
- process id and running state when started;
- configured `node_path` and `pyright_langserver_path`;
- successful LSP `initialize` result, including capabilities and server info when
  available;
- active manifest key and projection root;
- last init, refresh, analysis, and process-exit errors.

### Service Lifecycle

`PyrightLspRuntime` owns:

- process start;
- stdin/stdout JSON-RPC transport;
- request id multiplexing;
- cancellation/timeout handling;
- service restart;
- health state;
- active LayerStack manifest key;
- projection root;
- open document state, if needed.

### JSON-RPC/LSP Transport

Implement LSP framing rather than PPC operation framing:

```text
Content-Length: <bytes>\r\n
\r\n
<json-rpc payload>
```

Runtime must handle:

- requests;
- responses;
- notifications;
- server diagnostics notifications;
- shutdown/exit;
- unexpected process exit;
- bounded response sizes.

### Workspace Projection And Refresh

For every op:

```text
ensure_enabled(pyright_lsp)
ensure_service_running()
ensure_projection_current()
notify_changed_files_or_open_target_documents()
wait_for_analysis_reflection()
send_lsp_request()
shape_response()
```

Refresh implementation must account for pyright indexing:

- `SnapshotOnly` is not used by the initial public ops. If added later, it must
  still prove the target file content has been opened or synced in pyright for
  the manifest being reported.
- `AnalyzerReflected` must wait until pyright has observed changes and reached
  a bounded reflected state.
- If watcher notification is unreliable, restart the service or reopen affected
  files as a fallback.
- Diagnostics must not silently return stale diagnostics for a newer manifest.
- All four ops must include `manifest_key`, `freshness`, and a bounded stale or
  timeout status in success/error output.

### LSP Method Mapping

Initial public ops map to real LSP methods or notification-backed caches:

| Public op | LSP behavior |
|---|---|
| `sandbox.plugin.pyright_lsp.query_symbols` | File-scoped mode uses `textDocument/documentSymbol`; future workspace mode may use `workspace/symbol`. |
| `sandbox.plugin.pyright_lsp.definition` | Uses `textDocument/definition`. |
| `sandbox.plugin.pyright_lsp.references` | Uses `textDocument/references`. |
| `sandbox.plugin.pyright_lsp.diagnostics` | Opens or syncs the target file and returns current `textDocument/publishDiagnostics` cache entries for the reported manifest. |

### Initial Op Contracts

`sandbox.plugin.pyright_lsp.query_symbols`

```json
{
  "file_path": "pkg/module.py",
  "query": "name",
  "workspace": false
}
```

Output:

```json
{
  "success": true,
  "provider": "pyright_lsp",
  "manifest_key": "version:hash",
  "freshness": "analyzer_reflected",
  "stale": false,
  "symbols": []
}
```

`sandbox.plugin.pyright_lsp.definition`

```json
{
  "file_path": "pkg/module.py",
  "position": { "line": 10, "character": 4 }
}
```

`sandbox.plugin.pyright_lsp.references`

```json
{
  "file_path": "pkg/module.py",
  "position": { "line": 10, "character": 4 },
  "include_declaration": true
}
```

`sandbox.plugin.pyright_lsp.diagnostics`

```json
{
  "file_path": "pkg/module.py"
}
```

Diagnostics may also support workspace mode later, but the first e2e should
start file-scoped.

### Exit Gate

- `pyright_lsp` runs real pyright.
- Query symbols, definition, references, and diagnostics return typed responses.
- Code updates through LayerStack are visible to pyright.
- Every initial op proves it is not answering from stale state or returns a
  bounded stale/timeout status.
- Health reports provider state, real LSP initialization, and last
  init/refresh/analysis errors.

## Phase 4. Live E2E Proof

### E2E Tests To Add

`pyright_lsp_setup_and_health`

- enable `pyright_lsp` in test config;
- acquire sandbox;
- call `sandbox.plugin.list`;
- call `sandbox.plugin.health`;
- assert pyright provider is enabled, running, and healthy;
- assert health includes the configured node/langserver paths, process id,
  successful LSP initialize/capabilities, active manifest key, projection root,
  and no last init/analysis error.

`pyright_lsp_query_symbols`

- write Python file through `sandbox.file.write`;
- call `sandbox.plugin.pyright_lsp.query_symbols`;
- assert symbol result comes from real pyright-backed provider;
- assert response namespace is `pyright_lsp`, not old `lsp`.

`pyright_lsp_definition_and_references`

- write two Python files with import/reference structure;
- call `sandbox.plugin.pyright_lsp.definition`;
- call `sandbox.plugin.pyright_lsp.references`;
- assert locations map to expected files and ranges.

`pyright_lsp_navigation_refreshes_after_update`

- write two Python files with import/reference structure;
- call `sandbox.plugin.pyright_lsp.definition` or
  `sandbox.plugin.pyright_lsp.references`;
- change the target definition or reference through `sandbox.file.write`;
- call the same navigation op again;
- assert the new result reflects the update and the old location/result is not
  returned.

`pyright_lsp_diagnostics_refreshes_after_update`

- write a file with a type or syntax issue;
- call `sandbox.plugin.pyright_lsp.diagnostics`;
- fix the file through `sandbox.file.write`;
- call diagnostics again;
- assert diagnostics reflect the fix after analyzer refresh.

`pyright_lsp_rejects_when_disabled`

- run with `enabled_plugins: []`;
- call a `sandbox.plugin.pyright_lsp.*` op;
- assert typed `plugin_disabled` rejection.

### E2E Tests To Remove Or Rewrite

- `test_plugin_lsp_dispatch.rs` fake package setup must be deleted or rewritten.
- Any e2e that stages `/eos/scratch/uploads/plugins/lsp/...` must be removed.
- Generic dynamic plugin tests should not be used as proof for `pyright_lsp`.
- Old `plugin.lsp.query_symbols` and `sandbox.plugin.lsp.query_symbols`
  assertions must become `sandbox.plugin.pyright_lsp.query_symbols`.
- Generated e2e readmes/indexes must stop advertising `plugin.lsp.*` as active
  coverage.

### Exit Gate

- Live e2e proves real pyright process setup.
- Live e2e proves health/list.
- Live e2e proves all four initial ops.
- Live e2e proves refresh after code update for at least diagnostics and one
  navigation op.
- Live e2e proves no LSP behavior depends on staged plugin packages, fake node,
  fake pyright, fake AST services, or `plugin.lsp.*`.

## Migration Risks

| Risk | Mitigation |
|---|---|
| Pyright answers stale data after LayerStack updates. | Enforce freshness modes and add diagnostics/navigation refresh e2e. |
| Projection remount does not trigger pyright index refresh. | Send watched-file notifications; restart/reopen fallback on timeout. |
| Static ops accidentally keep dynamic fallback behavior. | Add tests proving `sandbox.plugin.pyright_lsp.*` resolves through `BuiltinOp`. |
| Cleanup misses the current legacy route spelling. | Search for and remove `plugin.lsp.*`, `plugin_id = "lsp"`, and `sandbox.plugin.lsp.*`. |
| Public `sandbox.plugin.ensure` keeps arbitrary setup alive. | Remove it from the public catalog/API or make it internal first-party provisioning only. |
| Cleanup removes useful process lifecycle code too early. | Keep reusable process/health helpers until `PyrightLspRuntime` is stable. |
| Disabled plugin behavior becomes unknown-op. | Return typed `plugin_disabled` for known static ops. |
| Future `node_lsp` gets blocked by pyright-specific naming. | Keep provider boundary as `plugin/<provider>` and reserve `sandbox.plugin.node_lsp.*`. |

## Final State

After the migration:

- `sandbox.plugin.pyright_lsp.*` is a static, typed daemon capability.
- `sandbox.plugin.node_lsp.*` remains available for a later static provider.
- Users cannot upload arbitrary plugin code.
- Users cannot provide plugin manifests, staged package roots, setup commands, or
  operation intents.
- Plugin operation intent is no longer user-controlled.
- Initial LSP ops are read-only.
- Real pyright runs as a monitored daemon service.
- LayerStack updates are reflected before analysis-sensitive LSP answers.
- Live e2e proves setup, health, operation behavior, and refresh.
