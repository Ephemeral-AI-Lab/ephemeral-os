1.Bird's-eye view: the four nested loops

EphemeralOS structures every piece of delegated work as a strictly nested four-level state machine. Each level has its own status, its own state-transition rules, and its own creation/closure handler. A user-facing request enters as a Goal, which spawns Iterations until terminal; each iteration spawns Attempts until terminal; each attempt spawns the harness task triple (one planner, a DAG of generators, one evaluator) until terminal.

Goal

The user-visible unit of delegated work. Created by a parent task. Closes once succeeded, failed, or cancelled.

Iteration

One attempt-budget's worth of retries. New iterations only start via a deferred goal from the prior iteration.

Attempt

One pass through plan→generate→evaluate. Multiple attempts allowed within an iteration up to attempt_budget.

Harness Task

The leaf agent runs: plannergeneratorevaluator. Each is one Task row driven by one launched agent.

Every level's closure delivers a report upward. Attempt closure delivers to its Iteration's IterationManager; iteration closure delivers to the goal's IterationClosureRouter; goal closure delivers to the parent task via GoalClosureReportRouter, which un-blocks the parent (lifting it out of waiting_goal).

Exactly one iteration of a Goal is OPEN at any moment. The next iteration only starts after the previous one closes (succeeded with a deferred-goal, or failed within budget).

Goal closure path

  1. The active iteration's IterationManager closes and emits an IterationClosureReport.
  2. IterationClosureRouter.route(report) pattern-matches the outcome:
    • SuccessDeferred ⇒ do not close goal; create a continuation iteration (see §3).
    • TerminalSuccess ⇒ GoalHandler.close_goal(succeeded=True).
    • AttemptPlanFailed ⇒ GoalHandler.close_goal(succeeded=False).
  3. On close, GoalClosureReportRouter finds the parent task (requested_by_task_id), and delivers the report via lifecycle_target.apply_goal_closure_report(report). The parent transitions out of WAITING_GOAL ⇒ RUNNING and can read the goal outcome.

Where new iterations come from

There are only two codepaths that create an iteration:

  1. Initial iteration — IterationFactory.create_initial(goal=…), invoked exactly once by GoalStarter.start() immediately after the Goal row is inserted. The iteration inherits the Goal's goal text and creation_reason=INITIAL.
  2. Deferred continuation — IterationFactory.create_from_deferred_goal(previous_iteration=…), invoked by IterationClosureRouter when the closing iteration emits SuccessDeferred. Pre-conditions:
    • parent Goal is still OPEN
    • previous iteration is SUCCEEDED
    • previous iteration's deferred_goal_for_next_iteration is non-null
    The new iteration's goal field is the previous iteration's deferred_goal_for_next_iteration. sequence_no = prev + 1. creation_reason = DEFERRED_GOAL_CONTINUATION.

Within a single iteration, attempts retry up to attempt_budget:

def handle_attempt_closed(self, attempt: Attempt) -> None:
    if attempt.status == AttemptStatus.PASSED:
        self._close_iteration_passed(attempt)   # may emit TerminalSuccess or SuccessDeferred
        return
    # FAILED
    if self._attempts_used < self.attempt_budget:
        self.create_next_attempt()              # new attempt, same iteration
    else:
        self._close_iteration_failed(attempt)   # emits AttemptPlanFailed

4.Nested goal creation — how a Task spawns a child Goal

A nested goal is a Goal whose requested_by_task_id points at a Task that is itself running inside some other Goal's harness. The depth of nesting is what keeps recursive delegation bounded — see also the nested_goal_depth_within_handoff_range predicate in _core/agent_routing.py.

                                           is open)                & report delivered)
Nesting is created when an executor task submits an "execution handoff". GoalStarter spins up a fresh Goal with the executor task as parent, then chains an initial Iteration, an Attempt, and the Attempt's planner Task.

Step-by-step

  1. A leaf executor task (running in some attempt's DAG, role executor_success_handoff) submits a terminal tool that signals "hand off a sub-goal". The submission carries the proposed goal text.
  2. GoalStarter.start(parent_task_id, goal=…)
    • validates the parent task is RUNNING and has no other open child goal
    • creates the new Goal row with requested_by_task_id = parent_task_id
    • creates the initial Iteration via IterationFactory.create_initial
    • creates the initial Attempt via IterationManager.create_unstarted_initial_attempt
    • marks parent task WAITING_GOAL — the parent cannot consume tools again until this child Goal closes
  3. The AttemptOrchestrator takes over and launches the child Attempt's planner.
  4. When the child Goal closes, its GoalClosureReport is delivered back to the parent task via GoalClosureReportRouter; the parent transitions WAITING_GOAL → RUNNING, receives the outcome as a message, and resumes.
A task has at most one open child Goal at any time. Tasks transition to WAITING_GOAL the moment their handoff is accepted, and back to RUNNING only when that child Goal closes.

Depth-bounded role routing

Nested executors don't inherit infinite handoff capability. The agent routing layer checks nested_goal_depth_within_handoff_range; once the cumulative depth exceeds the configured maximum, the resolver picks the executor_success_failure variant instead of executor_success_handoff, removing the handoff terminal from the agent's tool surface. This is how the system enforces a finite recursion budget across the whole goal tree.

5.Attempt harness — planner → DAG generators → evaluator

The AttemptOrchestrator is the heart of the harness. It is event-driven (not poll-driven): every external event (planner submission, generator submission, evaluator submission, exhaustion) calls into the orchestrator, which then re-reads state from stores and dispatches the next wave of work.

5.1 Plan submission → DAG persistence

When the planner submits a plan via apply_plan_submission:

def apply_plan_submission(self, submission: PlannerSubmission) -> None:
    ...
    generator_ids = self._persist_generator_tasks(submission.tasks)
    runtime.attempt_store.set_generator_task_ids(attempt.id, list(generator_ids))
    runtime.attempt_store.set_stage(attempt.id, AttemptStage.GENERATE)
    self._dispatcher.dispatch_ready_work()

The planner's tasks are passed through ordered_generator_tasks(...), which is a Kahn topological sort over local_ids. The sort guarantees:

  • all local_ids are unique
  • all deps reference existing local_ids
  • there are no cycles (otherwise raises)

Each ordered task is persisted with its real Task ID and a needs tuple holding the Task IDs of its dependencies. This is what the dispatcher uses later; the planner's symbolic local_ids are not visible at runtime.

5.2 Generator dispatch — wave-by-wave from persisted DAG

The dispatcher is purely store-driven:

def _dispatch_generating(self, attempt: Attempt) -> None:
    task_records = runtime.task_store.list_generator_tasks_for_attempt(attempt.id)
    ready_ids = ready_pending_generator_ids(task_records)
    if ready_ids:
        for task_id in ready_ids:
            self._launch_ready_generator(attempt=attempt, task_id=task_id)
        return
    state = summarize_generator_dag(task_records)
    if not state.all_quiescent:
        return
    if state.any_failed_or_blocked:
        self._close_attempt(AttemptStatus.FAILED, AttemptFailReason.GENERATOR_FAILED)
        return
    if state.all_done:
        self._spawn_evaluator(attempt)

Generator-DAG dispatch. Every PENDING node whose entire needs[] is DONE fires in one wave; the dispatcher recurses on every submission to fire the next wave. After all generators reach DONE, the evaluator is spawned exactly once.

Failure cascade

On a generator failure submission, block_failed_descendants(task_id) walks the DAG via BFS and marks every pending descendant BLOCKED. On the next dispatcher call, summarize_generator_dag sees the quiescent failed-or-blocked state and closes the attempt with GENERATOR_FAILED. Evaluator is never spawned in this case.

5.3 Evaluator stage and attempt closure

def _dispatch_evaluating(self, attempt: Attempt) -> None:
    evaluator_task = runtime.task_store.get_task(attempt.evaluator_task_id)
    status = TaskCenterTaskStatus(evaluator_task["status"])
    if status == TaskCenterTaskStatus.DONE:
        self._close_attempt(AttemptStatus.PASSED, None)
    elif status == TaskCenterTaskStatus.FAILED:
        self._close_attempt(AttemptStatus.FAILED, AttemptFailReason.EVALUATOR_FAILED)

_close_attempt(status, fail_reason) writes the terminal state, deregisters itself from AttemptOrchestratorRegistry, and triggers IterationManager.on_attempt_closed(attempt.id), which decides whether to retry within budget, close the iteration as failed, or close it as succeeded (with or without a deferred goal).

6.Context Engine — Scope, Recipe, Packet, Renderer

Each agent launch in the harness needs a tailored prompt. The Context Engine builds that prompt deterministically from four ingredients: a ContextScope, a Recipe, the resulting ContextPacket, and the renderer. The pipeline is a pure function: same scope + same stores yields byte-identical output.

The context engine and agent-launch composer are two halves of the same pipeline: the engine produces a ContextPacket; the composer turns the packet into renderer-output + task-guidance + skill, bundled as AgentEntryMessages.

6.1 ContextScope — the only input

@dataclass(frozen=True, slots=True)
class ContextScope:
    goal_id:      str | None = None
    iteration_id: str | None = None
    attempt_id:   str | None = None
    task_id:      str | None = None

    @classmethod
    def for_planner(cls,   *, goal_id, iteration_id, attempt_id)            -> ContextScope: ...
    @classmethod
    def for_generator(cls, *, goal_id, iteration_id, attempt_id, task_id)   -> ContextScope: ...
    @classmethod
    def for_evaluator(cls, *, goal_id, iteration_id, attempt_id)            -> ContextScope: ...
    @classmethod
    def for_entry_executor(cls, *, task_id)                                 -> ContextScope: ...

    def assert_fields(self, required: frozenset[str]) -> None: ...

Notice: the scope holds identity tokens only. It does not carry the goal text, nor the iteration goal, nor prior attempts; recipes resolve those from ContextEngineDeps stores at build time. This is what makes the pipeline re-buildable from any persistent state.

6.2 Recipe — pure function with declared scope

RecipeBuild = Callable[[ContextScope, ContextEngineDeps], ContextPacket]

@dataclass(frozen=True, slots=True)
class ContextRecipe:
    id: str
    required_scope_fields: frozenset[str]
    build: RecipeBuild

Recipes are registered at startup by a recursive scan over context_engine/recipes/*.py for module-level constants whose name ends with _RECIPE: 

def register_builtin_recipes() -> None:
    for module_info in pkgutil.iter_modules(__path__, prefix=f"{__name__}."):
        module = importlib.import_module(module_info.name)
        for attr_name in dir(module):
            if not attr_name.endswith("_RECIPE"):
                continue
            value = getattr(module, attr_name)
            if isinstance(value, ContextRecipe):
                RecipeRegistry.register(value)

6.3 ContextEngine — the deterministic build

@dataclass(frozen=True, slots=True)
class ContextEngine:
    deps: ContextEngineDeps

    def build(self, recipe_id: str, scope: ContextScope) -> ContextPacket:
        recipe = RecipeRegistry.get(recipe_id)
        scope.assert_fields(recipe.required_scope_fields)
        return recipe.build(scope, self.deps)

@dataclass(frozen=True, slots=True)
class ContextEngineDeps:
    goal_store:           GoalStoreProtocol
    iteration_store:      IterationStoreProtocol
    attempt_store:        AttemptStoreProtocol
    task_store:           TaskStoreProtocol
    context_packet_store: ContextPacketStoreProtocol | None = None
Recipes are pure functions of (scope, deps). They read stores; they never write. They never see globals. This makes a context build reproducible from persistent state at any time, which is what enables the auditor and replayer to reconstruct the exact prompt a past agent saw.

6.4 ContextPacket — the assembled, not-yet-rendered output

class ContextPriority(StrEnum):
    REQUIRED, HIGH, MEDIUM, LOW

class ContextBlock(BaseModel):
    kind:        str                 # logical kind, e.g. "goal_statement"
    priority:    ContextPriority     # compression signal, not display order
    text:        str                 # byte-exact body (renderer does not mutate)
    source_id:   str | None
    source_kind: str | None
    metadata:    dict[str, str]      # tag, attrs, group_id, group_tag, group_attrs…

class ContextPacket(BaseModel):
    id:              str
    target_role:     str             # "planner", "generator", "evaluator", "entry_executor"
    target_id:       str | None      # attempt_id or task_id
    canonical_refs:  ContextRefs     # {goal_id, iteration_id, attempt_id, task_id}
    blocks:          list[ContextBlock]
    metadata:        dict[str, str]  # e.g. token_budget
    source_ids:      list[str]

6.5 Renderer — XML envelope + token-budget compression

The renderer is the only stage that turns blocks into text:

def render_context(self, packet: ContextPacket) -> str:
    kept_blocks = self._compress(list(packet.blocks), budget=self._budget_from(packet))
    self._validate_no_structural_closers(kept_blocks)
    sections = self._render_blocks(kept_blocks)
    body = "\n\n".join(s for s in sections if s)
    return body + "\n" if body else ""
  • Compression order: REQUIRED is never dropped; LOW is dropped first, then MEDIUM, then HIGH. The packet's metadata["token_budget"] defines the cap.
  • Structural-closer guard: any block body that contains a tag-closer that would prematurely end an enclosing tag raises a ContextEngineError. This is how recipes get away with pre-rendered XML (see attempts.py) without losing safety.
  • Grouping: blocks sharing metadata["group_id"] are emitted under a single wrapper tag derived from metadata["group_tag"] + metadata["group_attrs"]. This is how iterations and attempts get an outer <iteration ...> / <attempt ...> envelope around their children.
  • No mutation: text is appended verbatim, no whitespace normalization.

6.6 What-in-context outline — deterministic table of contents

In parallel with rendering, render_what_in_context(packet) walks the same packet and produces a bulleted outline. Each top-level tag is looked up in tag_dictionary.TAG_DICTIONARY to get a human-friendly label ("user's request", "failed prior attempt", etc.). 

TAG_DICTIONARY: list[TagDescriptor] = [
    TagDescriptor(tag="goal",          attr_filter=None,                         label="user's request"),
    TagDescriptor(tag="iteration",     attr_filter={"status": "prior"},          label="previous iteration's work"),
    TagDescriptor(tag="iteration",     attr_filter={"status": "current"},        label="active iteration"),
    TagDescriptor(tag="attempt",       attr_filter={"status": "prior",
                                                    "verdict": "fail"},          label="failed prior attempt"),
    TagDescriptor(tag="assigned_task", attr_filter=None,                         label="your assigned task"),
    ...
]

The outline is then injected into row 3's “What's in context” section, giving the agent a structured TOC over the same XML it just received. The two representations are derived from the same blocks and can never disagree.

7.Recipe inventory — what each role sees

Recipe Role Required scope Contains
planner planner goal_id, iteration_id, attempt_id <goal>, prior <iteration> groups, current <iteration status="current"> with <iteration_goal>, all failed <attempt> blocks for this iteration.
planner_closes_goal planner goal_id Single <goal> block. Used for leaf-node planners (nested depth > 1) where defers is forbidden; no iteration scaffolding.
generator generator goal_id, attempt_id, task_id <plan_spec>, one <dependency> per upstream task (latest summary), <assigned_task task_id="..."> — anchored last.

7.1 Planner recipe — assembling iteration history

The planner's context is, in order:

  1. <goal> — REQUIRED, the user's root request.
  2. For each closed prior iteration K < current:
    • <iteration iteration_no="K" status="prior"> wrapping <accepted_plan> + <summary>
    • Priority HIGH for the immediate predecessor; MEDIUM for older.
  3. <iteration iteration_no="N" status="current"> ⇒ <iteration_goal>
    • For iteration #1, body collapses to (identical to <goal>) to avoid duplication.
    • For iteration N>1, body is the deferred-goal text that seeded this iteration.
  4. For each failed attempt earlier in this iteration: <attempt status="prior" verdict="fail"> with pre-rendered XML body containing <plan_spec>, optional <deferred_goal_for_next_iteration>, <status_summary>, per-task <task>, <evaluation_criteria>, and the evaluator's <passed_criteria> / <failed_criteria>.

This shape lets the planner see exactly what was tried, why it failed, and what the iteration is now responsible for — without ever leaking implementation detail of non-failed attempts (there are none until one fails).

7.2 Generator recipe — minimal & task-scoped

  1. <plan_spec> — HIGH, the planner's contract for this attempt.
  2. <dependency id="..."> blocks — MEDIUM, one per needs[] entry, body is the latest summary of that upstream task. Renders flat (no grouping).
  3. <assigned_task task_id="..."> — REQUIRED, anchored last so the agent ends on its concrete obligation.

Notice what is absent: no <goal>, no iteration history, no failed attempts, no deferred_goal_for_next_iteration. Generators are intentionally narrow — they get exactly the contract, the inputs they depend on, and the task they must do.

7.3 Evaluator recipe — superset of planner + current attempt

The evaluator gets everything the planner sees (so it can judge in context), plus the current attempt:

  • <attempt status="current"> with <plan_spec>, per-task <task> (with latest summaries), and <evaluation_criteria> — but no evaluator verdict children yet (that's what the evaluator is about to produce).
  • If any generator task ended FAILED/BLOCKED/missing, the evaluator's current-attempt body collapses to <evaluator_summary status="bypassed" reason="generator_failed"> with the failed task IDs.

7.4 Shared helpers

Three internal modules in context_engine/recipes/ are not recipes themselves but are used by planner + evaluator:

  • iterations.py — goal_iteration_blocks() for the goal + prior iterations + current iteration scaffold.
  • attempts.py — failed_attempt_blocks() and current_attempt_block() which hand-build XML for an attempt body. Bodies are pre-rendered XML guarded by the renderer's structural-closer check.
  • summaries.py — latest_summary_text(summaries), picks the most recent summary/outcome text from a task's history.

8.1 Predicate-based agent selection

RuleBasedAgentResolver loads the base AgentDefinition, then walks its declared variants. Each variant carries a when field naming a predicate in PredicateRegistry; the first predicate that matches the ResolverContext(scope, deps) wins. 

for variant in base.variants:
    predicate = PredicateRegistry.get(variant.when)
    if predicate(ctx):
        return self._select_variant_target(variant)

This is how the executor splits into executor_success_handoff (allowed to delegate further) vs. executor_success_failure (terminal): the predicate nested_goal_depth_within_handoff_range looks at the goal-ancestry chain via goal.ancestry + stores, and flips role variants once depth crosses the threshold. Same base name in the planner's plan; different recipe and different terminal-tool surface at runtime.

10.Key invariants

A Goal has exactly one OPEN Iteration at any time. The next iteration starts only after the previous one closes; iteration switches happen via deferred-goal continuation.
An OPEN Iteration has at most one RUNNING Attempt at a time. Attempts retry strictly sequentially within attempt_budget.
An Attempt's stage is monotonically increasing: PLAN → GENERATE → EVALUATE → CLOSED. There is no rewind.
A planner submission with kind="completes" must have deferred_goal_for_next_iteration is None; one with kind="defers" must have it set. The two are mutually exclusive.
The generator DAG is topologically valid before dispatch. ordered_generator_tasks rejects cycles, duplicate local_ids, and references to undefined local_ids.
A generator task launches only when every Task ID in its persisted needs[] is DONE. Failure of any task transitively BLOCKEDs its descendants via BFS, and the attempt closes GENERATOR_FAILED without spawning the evaluator.
A Goal's parent task is always in WAITING_GOAL while the child Goal is OPEN, and transitions back to RUNNING when the child Goal closes.
Recipes are pure functions of (scope, deps). They only read stores. This makes context build reproducible from persistent state at any time, which is what enables the auditor and replayer to reconstruct the exact prompt an agent saw at any past point.
The renderer never mutates block text. ContextBlock.text is byte-exact, so the auditor's view matches the agent's view exactly.
The What's-in-context outline (row 3) and the rendered <context> envelope (row 2) are derived from the same blocks, so they can never disagree about what the agent has access to.
